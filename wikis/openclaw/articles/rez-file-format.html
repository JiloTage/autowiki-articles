REZ File Format and Asset Packaging

最終更新: 2026-04-08
The REZ format is a proprietary archive container developed by Monolith Productions for their RIOT engine, used to package all game assets in Captain Claw into a single distributable file. OpenClaw's REZ reader parses this archive at runtime, enabling copyright-safe engine distribution without bundled assets.

Overview

REZ File Format and Asset Packaging refers to the binary archive container and associated runtime asset pipeline used by Captain Claw (1997) and other games built on Monolith Productions' RIOT engine. The .REZ extension denotes a self-contained archive that stores all game resources — sprites, audio, music, level data, and fonts — in a hierarchical directory structure embedded within a single binary file. This design mirrors the packaging conventions of the era, in which game data was consolidated into one large file rather than distributed as a flat folder of loose files.

For the OpenClaw project, the REZ format is the essential bridge between the open-source reimplemented engine and the original copyrighted content. OpenClaw ships without any game assets; players must supply CLAW.REZ from a legally owned copy of the game. The engine then parses this archive at startup, extracting assets into memory without writing anything to disk. This approach upholds game preservation ethics by keeping engine code and game data strictly separated.

The REZ format was reverse-engineered by community developers examining the binary structure of CLAW.REZ, cross-referencing known file signatures (PCX headers, WAV RIFF chunks, MIDI headers) to identify individual assets within the archive. The resulting specification drove OpenClaw's native parser implementation.

File Structure

A REZ archive is a flat binary file organised into four logical regions: the file header, the directory tree, individual file entry descriptors, and the raw data payloads. All multi-byte integer fields are stored in little-endian byte order, consistent with the x86 architecture of the original DOS/Windows target platform.

The file header occupies a fixed block at the start of the file. It begins with a magic byte sequence that identifies the file as a REZ archive, followed by a version field, a 32-bit offset pointing to the start of the directory tree, the total size of the directory region, and a timestamp recording when the archive was assembled by Monolith's toolchain.

The directory tree mirrors a virtual filesystem hierarchy. Top-level directories such as SOUNDS, SPRITES, LEVEL, and FONTS are represented as named directory nodes. Each node records a count of its children and relative offsets to child entries, allowing a recursive descent parser to reconstruct the virtual path for every asset.

File entry descriptors appear after the directory tree. Each descriptor binds a filename and virtual path string to a data offset and size within the payload region, along with a short type tag and a checksum. The type tag allows the loader to quickly classify an asset (image, audio, level, font) without inspecting the payload.

The raw data payloads are stored contiguously from the offset recorded in the header, with individual assets laid out sequentially in the order they appear in the directory. No padding alignment is enforced between entries, so a parser must use the stored offset and size values precisely.

Types of Assets Stored

CLAW.REZ bundles every resource required to run Captain Claw without any external files. The archive contains five primary asset categories:

  • PCX sprite sheets — The primary image format. PCX is a run-length-encoded 8-bit (256-colour) bitmap format originating from the ZSoft PC Paintbrush application. Each sprite sheet encodes one or more animation frames on a fixed-width canvas. Captain Claw uses a magenta colour (RGB 255, 0, 255) as the transparency key, which is stripped during loading to produce surfaces with per-pixel alpha.
  • WAV audio files — Sound effects and voice clips are stored as standard RIFF/WAV files with PCM-encoded 16-bit samples at 22 050 Hz or 44 100 Hz. These files carry their own RIFF header within the REZ payload, so extraction requires only copying the payload bytes verbatim.
  • MIDI music — Per-level background music tracks are stored as standard Type-0 or Type-1 MIDI files. Because MIDI carries only note event data rather than sampled audio, the files are compact (typically 20–80 KB each). Playback relies on the host system's MIDI synthesiser or a software renderer.
  • WWD level data — World/level definitions use a proprietary binary format called WWD (described in the following section). Each of the 14 levels in Captain Claw is represented by one .WWD file inside the LEVEL directory subtree.
  • Font files — Bitmap fonts used for HUD elements, menus, and dialogue are stored as PCX images paired with character-map metadata. The font PCX encodes each glyph in a fixed grid, and the metadata defines character widths and Unicode mappings.

Beyond these five types, the archive also contains XML-like configuration files (using Monolith's proprietary dialect) that define object properties, animation state machines, and gameplay tuning parameters. These are text data embedded as raw ASCII payloads within the REZ structure.

WWD Level Format

The WWD (World/Wap Data) format is the binary encoding used by the RIOT engine to describe a complete game level. Each .WWD file inside CLAW.REZ encodes a tilemap, an object placement list, and a scripting event table in a compact sequential binary layout.

A WWD file opens with a header that records the level name, the number of tile planes, the dimensions of the primary tile plane (in tiles), and a pointer table to the three major data blocks: the tile plane data, the object list, and the image set reference strings.

The tile plane block stores the map as a flat array of 16-bit tile indices, row-major, where each index references a slot in the level's image set. Captain Claw levels typically feature two to four tile planes stacked for parallax scrolling: a foreground collision plane, one or two decorative midground planes, and a background sky plane. Each plane header specifies its scroll factor relative to the camera, enabling the engine to derive parallax offsets directly from player position.

The object list block is a variable-length sequence of object records. Each record encodes: an object type identifier, pixel-precise x/y coordinates, initial state flags, and a string-indexed property bag for type-specific parameters (for example, the patrol path length for a guard enemy, or the destination level for a level-exit trigger). Type identifiers map to C++ entity class names in OpenClaw's entity registry.

The scripting event table pairs trigger conditions (player entering a region, object being destroyed) with action sequences encoded as a compact bytecode. This bytecode drives cutscene animations, dialogue boxes, and timed spawning events. OpenClaw reimplements the bytecode interpreter faithfully to ensure that all in-game sequences fire at the correct moments.

OpenClaw's REZ Reader Implementation

OpenClaw implements a dedicated REZ parser in C++ that processes the archive in two phases: an index phase at startup and an on-demand extraction phase during gameplay. The parser is structured to avoid loading the entire archive into memory; instead it memory-maps or seeks into the file based on the offsets recorded in the directory.

During the index phase, the parser reads the file header to locate the directory tree, then recursively descends the directory tree, building an in-memory map from virtual path strings (e.g., SOUNDS/LEVEL1/MUSIC.MID) to pairs of (file offset, file size). This map is stored as a std::unordered_map keyed on the lowercased virtual path string, enabling case-insensitive lookups that match the original engine's behaviour under Windows.

During the extraction phase, the asset manager calls into the REZ reader with a virtual path. The reader seeks to the recorded offset, reads exactly the recorded number of bytes into a heap-allocated buffer, and returns ownership of that buffer to the caller. The caller — typically the image loader or audio manager — then interprets the bytes according to the asset type.

For PCX sprites, the extracted buffer is decoded into an SDL_Surface by a custom PCX decoder. The transparency key colour is remapped to SDL_ALPHA_TRANSPARENT, producing a surface with a per-pixel alpha channel. The surface is then converted to an SDL_Texture via the SDL renderer, at which point the surface is freed. Textures remain cached in a texture atlas keyed by virtual path until the level unloads.

For WAV audio, the buffer is passed directly to SDL_mixer's Mix_LoadWAV_RW using an SDL_RWops wrapper around the in-memory buffer, avoiding any intermediate disk I/O. MIDI files are similarly piped through SDL_mixer's MIDI playback subsystem using in-memory SDL_RWops.

WWD level data is consumed by a dedicated WwdReader class that deserialises the binary layout into a rich C++ object graph: a LevelData struct containing a vector of TilePlane objects and a vector of ActorDef objects. This object graph drives the level loader, which instantiates Box2D bodies and OpenClaw entity objects accordingly.

Comparison to Contemporary Archive Formats

The REZ format belongs to a family of proprietary game archive containers that proliferated in the mid-1990s, when distributing hundreds of individual asset files on CD-ROM was impractical given filesystem overhead and access latency. Several contemporary formats pursued similar goals with varying design choices:

  • WAD files (id Software, 1993–) — Used by Doom and Quake, WAD (Where's All the Data) archives store a flat list of named lumps. There is no directory hierarchy; lump names are limited to eight characters. WAD files were split into IWAD (game-provided) and PWAD (user-provided patch) files, establishing an influential modding convention.
  • PAK files (Quake, 1996) — Quake's PAK format introduced a simple flat directory with null-terminated filenames up to 56 characters and a contiguous data region, trading WAD's lump flexibility for longer, more descriptive paths. PAK became a template for many engines of the late 1990s.
  • GRP files (Duke Nukem 3D, 1996) — The BUILD engine's GRP format is essentially a flat concatenation of named files with a minimal directory block. GRP is notable for its simplicity: the entire directory fits in the first few kilobytes, and extraction is trivial to implement.
  • RES/HOG files (Descent, 1994) — Parallax Software's HOG format similarly concatenated named files with a leading directory, targeting the same DOS platform constraints.

REZ distinguishes itself from these alternatives primarily through its hierarchical virtual directory tree, which more closely resembles a miniature FAT filesystem than the flat lump models of WAD or GRP. This hierarchy made Monolith's internal asset pipeline more tractable — artists could organise resources into logical subfolders — at the cost of a more complex parser. For OpenClaw's purposes, the hierarchical structure is a feature: virtual paths are self-documenting, making asset lookups straightforward without needing to maintain a separate manifest.

The decision to preserve the REZ-based asset pipeline in OpenClaw rather than converting assets to a modern format (such as ZIP or a custom SQLite store) was deliberate. Converting assets would require distributing a conversion tool, potentially transforming copyrighted content in ways that complicate the legal standing of the project. Loading CLAW.REZ verbatim ensures that OpenClaw is purely a clean-room reimplementation, as discussed in Game Preservation Ethics.
