REZファイル形式とアセットパッケージング

最終更新: 2026-04-08
REZ 形式は Monolith Productions が RIOT エンジン向けに開発した独自アーカイブコンテナで、Captain Claw のすべてのゲームアセットを単一の配布可能ファイルにパッケージするために使用されます。OpenClaw の REZ リーダーは実行時にこのアーカイブを解析し、バンドルされたアセットなしに著作権上安全なエンジン配布を実現します。

概要

REZファイル形式とアセットパッケージングとは、Captain Claw（1997）および Monolith Productions の RIOT エンジンを基盤とする他のゲームで使用されるバイナリアーカイブコンテナとそれに関連するランタイムアセットパイプラインを指します。.REZ 拡張子は、スプライト・音声・音楽・レベルデータ・フォントなどすべてのゲームリソースを単一のバイナリファイル内に埋め込まれた階層的ディレクトリ構造として格納する、自己完結型アーカイブを示します。この設計は当時のパッケージング慣例を反映しており、ゲームデータがフラットなファイルフォルダとして配布されるのではなく、1 つの大きなファイルに統合されていました。

OpenClaw プロジェクトにとって、REZ 形式はオープンソースの再実装エンジンとオリジナルの著作権コンテンツを結ぶ不可欠な橋渡しです。OpenClaw にはゲームアセットが同梱されておらず、プレイヤーは正規に購入したゲームの CLAW.REZ を用意する必要があります。エンジンはこのアーカイブを起動時に解析し、ディスクに何も書き込まずにアセットをメモリに展開します。このアプローチはエンジンコードとゲームデータを厳密に分離することで ゲーム保存の倫理を守ります。

REZ 形式は、アーカイブのバイナリ構造を調べたコミュニティ開発者たちによってリバースエンジニアリングされました。アーカイブ内の個々のアセットを特定するために既知のファイルシグネチャ（PCX ヘッダー・WAV RIFF チャンク・MIDI ヘッダー）が照合されました。この成果として得られた仕様が OpenClaw のネイティブパーサー実装を主導しました。

ファイル構造

REZ アーカイブはフラットなバイナリファイルで、4 つの論理領域に編成されています：ファイルヘッダー・ディレクトリツリー・個別のファイルエントリ記述子・生データペイロード。すべての多バイト整数フィールドは、オリジナルの DOS/Windows ターゲットプラットフォームの x86 アーキテクチャと一致するリトルエンディアンバイトオーダーで格納されています。

図1: REZアーカイブのバイナリ構造 — ヘッダ・ディレクトリツリー・ファイルエントリ・データペイロードの4領域

ファイルヘッダーはファイルの先頭に固定ブロックとして配置されます。REZ アーカイブとして識別するマジックバイト列で始まり、バージョンフィールド・ディレクトリツリーの開始位置を示す 32 ビットオフセット・ディレクトリ領域の合計サイズ・Monolith のツールチェーンによってアーカイブが組み立てられた日時のタイムスタンプが続きます。

ディレクトリツリーは仮想ファイルシステム階層を再現します。SOUNDS・SPRITES・LEVEL・FONTS などのトップレベルディレクトリは名前付きディレクトリノードとして表現されます。各ノードは子の数と子エントリへの相対オフセットを記録し、再帰的な降下パーサーがすべてのアセットの仮想パスを再構築できます。

ファイルエントリ記述子はディレクトリツリーの後に続きます。各記述子はファイル名と仮想パス文字列をペイロード領域内のデータオフセットおよびサイズに結びつけ、短いタイプタグとチェックサムも含みます。タイプタグにより、ローダーはペイロードを調べることなくアセットを（画像・音声・レベル・フォントに）素早く分類できます。

生データペイロードはヘッダーに記録されたオフセットから連続して格納され、個々のアセットはディレクトリに現れる順序で連続して配置されます。エントリ間にはパディングアライメントが強制されないため、パーサーは格納されたオフセットとサイズ値を正確に使用する必要があります。

格納されるアセットの種類

CLAW.REZ は外部ファイルなしに Captain Claw を実行するために必要なすべてのリソースをバンドルしています。アーカイブには 5 つの主要なアセットカテゴリが含まれています：

  • PCX スプライトシート — 主要な画像形式。PCX は ZSoft PC Paintbrush アプリケーションに起源を持つランレングスエンコードされた 8 ビット（256 色）のビットマップ形式です。各スプライトシートは固定幅のキャンバス上に 1 つ以上のアニメーションフレームをエンコードします。Captain Claw はマゼンタ色（RGB 255, 0, 255）を透過キーとして使用しており、ロード時に除去されてピクセルごとのアルファチャンネルを持つサーフェスが生成されます。
  • WAV 音声ファイル — 効果音とボイスクリップは 22 050 Hz または 44 100 Hz で PCM エンコードされた 16 ビットサンプルを持つ標準 RIFF/WAV ファイルとして格納されています。これらのファイルは REZ ペイロード内に独自の RIFF ヘッダーを持つため、展開にはペイロードバイトをそのままコピーするだけで済みます。
  • MIDI 音楽 — レベルごとの BGM トラックは標準的な Type-0 または Type-1 MIDI ファイルとして格納されています。MIDI はサンプリング音声ではなくノートイベントデータのみを含むため、ファイルはコンパクト（通常 1 つあたり 20〜80 KB）です。再生はホストシステムの MIDI シンセサイザーまたはソフトウェアレンダラーに依存します。
  • WWD レベルデータ — ワールド/レベル定義は WWD（次のセクションで説明）と呼ばれる独自のバイナリ形式を使用します。Captain Claw の 14 レベルそれぞれが LEVEL ディレクトリサブツリー内の 1 つの .WWD ファイルとして表現されています。
  • フォントファイル — HUD 要素・メニュー・ダイアログに使用されるビットマップフォントは、キャラクターマップのメタデータと対になった PCX 画像として格納されています。フォント PCX は各グリフを固定グリッドにエンコードし、メタデータは文字幅と Unicode マッピングを定義します。

これら 5 つのタイプに加え、アーカイブにはオブジェクトのプロパティ・アニメーションのステートマシン・ゲームプレイの調整パラメーターを定義する XML 風の設定ファイル（Monolith 独自の方言を使用）も含まれています。これらは REZ 構造内に生の ASCII ペイロードとして埋め込まれたテキストデータです。

WWD レベル形式

WWD（World/Wap Data）形式は、RIOT エンジンが完全なゲームレベルを記述するために使用するバイナリエンコーディングです。CLAW.REZ 内の各 .WWD ファイルは、タイルマップ・オブジェクト配置リスト・スクリプティングイベントテーブルをコンパクトな連続バイナリレイアウトにエンコードします。

WWD ファイルはレベル名・タイルプレーンの数・プライマリタイルプレーンの寸法（タイル単位）・および 3 つの主要データブロック（タイルプレーンデータ・オブジェクトリスト・イメージセット参照文字列）へのポインタテーブルを記録するヘッダーで始まります。

タイルプレーンブロックはマップを 16 ビットのタイルインデックスのフラット配列として行優先順で格納し、各インデックスはレベルのイメージセット内のスロットを参照します。Captain Claw のレベルは通常、パララックススクロール用に積み重ねられた 2〜4 つのタイルプレーンを持ちます：前景衝突プレーン・1〜2 つの装飾的なミッドグラウンドプレーン・背景スカイプレーンです。各プレーンヘッダーはカメラに対するスクロールファクターを指定し、エンジンがプレイヤー位置から直接パララックスオフセットを導出できるようにします。

オブジェクトリストブロックは可変長のオブジェクトレコードシーケンスです。各レコードには、オブジェクトタイプ識別子・ピクセル精度の x/y 座標・初期状態フラグ・タイプ固有パラメーター（例：警備員敵のパトロールパス長、またはレベル脱出トリガーの目的レベル）のための文字列インデックスプロパティバッグがエンコードされています。タイプ識別子は OpenClaw のエンティティレジストリ内の C++ エンティティクラス名にマッピングされます。

スクリプティングイベントテーブルはトリガー条件（プレイヤーがある領域に入る・オブジェクトが破壊される）とコンパクトなバイトコードとしてエンコードされたアクションシーケンスをペアリングします。このバイトコードはカットシーンアニメーション・ダイアログボックス・時限スポーンイベントを制御します。OpenClaw はすべてのゲーム内シーケンスが正しいタイミングで発動することを保証するためにバイトコードインタープリタを忠実に再実装しています。

OpenClaw の REZ リーダー実装

OpenClaw は C++ で専用の REZ パーサーを実装しており、起動時のインデックスフェーズとゲームプレイ中のオンデマンド展開フェーズの 2 段階でアーカイブを処理します。パーサーはアーカイブ全体をメモリにロードすることを避けるよう設計されており、ディレクトリに記録されたオフセットに基づいてファイルをメモリマップするかシーク操作を行います。

インデックスフェーズでは、パーサーはファイルヘッダーを読んでディレクトリツリーの位置を特定し、次にディレクトリツリーを再帰的に降下して、仮想パス文字列（例：SOUNDS/LEVEL1/MUSIC.MID）から（ファイルオフセット・ファイルサイズ）ペアへのインメモリマップを構築します。このマップは小文字化した仮想パス文字列をキーとした std::unordered_map として格納され、Windows 上のオリジナルエンジンの挙動に合わせた大文字小文字を無視するルックアップを可能にします。

展開フェーズでは、アセットマネージャーが仮想パスを指定して REZ リーダーを呼び出します。リーダーは記録されたオフセットにシークし、記録されたバイト数をヒープ割り当てバッファに読み込み、そのバッファの所有権を呼び出し元に返します。呼び出し元（通常は画像ローダーまたは音声マネージャー）はアセットタイプに従ってバイトを解釈します。

PCX スプライトの場合、展開されたバッファはカスタム PCX デコーダによって SDL_Surface にデコードされます。透過キーカラーは SDL_ALPHA_TRANSPARENT に再マッピングされ、ピクセルごとのアルファチャンネルを持つサーフェスが生成されます。サーフェスはその後 SDL レンダラーを通じて SDL_Texture に変換され、その時点でサーフェスは解放されます。テクスチャはレベルがアンロードされるまで仮想パスをキーとしたテクスチャアトラスにキャッシュされます。

WAV 音声の場合、バッファは中間ディスク I/O を避けるためにインメモリバッファをラップした SDL_RWops を使って SDL_Mixer の Mix_LoadWAV_RW に直接渡されます。MIDI ファイルも同様に、インメモリの SDL_RWops を使って SDL_Mixer の MIDI 再生サブシステムに送られます。

WWD レベルデータは専用の WwdReader クラスによって消費され、バイナリレイアウトをリッチな C++ オブジェクトグラフにデシリアライズします：TilePlane オブジェクトのベクトルと ActorDef オブジェクトのベクトルを含む LevelData 構造体です。このオブジェクトグラフはレベルローダーを駆動し、それに応じて Box2D ボディと OpenClaw エンティティオブジェクトをインスタンス化します。

同時代のアーカイブ形式との比較

REZ 形式は 1990 年代半ばに広まった独自ゲームアーカイブコンテナファミリーに属します。当時は CD-ROM 上で数百の個別アセットファイルを配布することがファイルシステムのオーバーヘッドとアクセス遅延の観点から非現実的でした。同時代のいくつかの形式は、異なる設計上の選択をしながら同様の目標を追求しました：

  • WAD ファイル（id Software、1993 年〜） — Doom と Quake で使用された WAD（Where's All the Data）アーカイブは名前付きランプのフラットリストを格納します。ディレクトリ階層はなく、ランプ名は 8 文字に制限されています。WAD ファイルは IWAD（ゲーム提供）と PWAD（ユーザー提供のパッチ）に分割され、影響力のあるモディング慣例を確立しました。
  • PAK ファイル（Quake、1996 年） — Quake の PAK 形式は最大 56 文字のヌル終端ファイル名を持つシンプルなフラットディレクトリと連続データ領域を導入し、WAD のランプ柔軟性を長くてより説明的なパスのために犠牲にしました。PAK は 1990 年代後半の多くのエンジンのテンプレートとなりました。
  • GRP ファイル（Duke Nukem 3D、1996 年） — BUILD エンジンの GRP 形式は本質的に最小限のディレクトリブロックを持つ名前付きファイルのフラット連結です。GRP はその単純さで注目されています：ディレクトリ全体が最初の数キロバイトに収まり、展開の実装は些細なものです。
  • RES/HOG ファイル（Descent、1994 年） — Parallax Software の HOG 形式も同様に先頭ディレクトリ付きの名前付きファイルを連結し、同じ DOS プラットフォームの制約をターゲットとしていました。

REZ は主にその階層的な仮想ディレクトリツリーによってこれらの代替手段と一線を画しており、WAD や GRP のフラットランプモデルよりも小型の FAT ファイルシステムに近い構造を持ちます。この階層化により Monolith の内部アセットパイプラインはより扱いやすくなりました — アーティストがリソースを論理的なサブフォルダに整理できるようになった反面、より複雑なパーサーが必要となりました。OpenClaw にとって、階層構造は利点です：仮想パスが自己文書化されているため、別途マニフェストを維持することなくアセットのルックアップが簡単に行えます。

OpenClaw において REZ ベースのアセットパイプラインを維持し、アセットを現代的な形式（ZIP やカスタムの SQLite ストアなど）に変換しないという決定は意図的なものです。アセットを変換するには変換ツールの配布が必要となり、著作権コンテンツをプロジェクトの法的立場を複雑にする方法で変換する可能性があります。CLAW.REZ をそのままロードすることで、OpenClaw は純粋なクリーンルーム再実装であり続けることが確保されます。これは ゲーム保存の倫理で論じられている通りです。
