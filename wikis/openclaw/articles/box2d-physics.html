Box2D Physics in Platform Games

最終更新: 2026-04-08
Box2D is an open-source 2D rigid body simulation library by Erin Catto that powers the physics engine in OpenClaw, handling character movement, collision detection, projectile arcs, and environmental interactions in a faithful recreation of classic platformer behaviour.

Overview of Box2D

Box2D Physics in Platform Games refers to the application of Box2D — a widely-used, open-source 2D rigid body simulation library originally written by Erin Catto — in the context of side-scrolling platformer games. Box2D was introduced at the 2006 Game Developers Conference and has since become one of the most influential physics engines in the independent and hobbyist game-development ecosystem, underpinning engines and games such as Angry Birds and many Ludum Dare entries.

In OpenClaw, Box2D serves as the authoritative physics layer for the entire simulation. The engine creates a Box2D world object that advances one fixed time step per game tick, keeping physics updates decoupled from the rendering frame rate. All collidable entities — Captain Claw, enemies, platforms, ladders, springs, and projectiles — are represented as Box2D bodies. This separation of physics from rendering allows the SDL2 rendering layer to interpolate draw positions between physics steps, producing smooth motion even when the physics rate and display refresh rate differ.

Box2D's coordinate system uses metres as its unit of length and operates independently of pixel coordinates. OpenClaw therefore maintains a scale factor (pixels per metre) and transforms between physics space and screen space at both input and output boundaries of the physics step.

Core Concepts: Bodies, Fixtures, and Joints

Box2D models the physical world through three principal abstractions: rigid bodies, fixtures, and joints.

A rigid body (b2Body) represents a point mass with position, velocity, angular velocity, and mass properties. Bodies may be static (infinite mass, never moves — used for terrain), kinematic (moves under direct velocity control but is not pushed by forces — used for moving platforms), or dynamic (fully simulated under forces, gravity, and collisions — used for the player character and enemies).

A fixture (b2Fixture) attaches a collision shape to a body and carries material properties: density (which influences mass), friction coefficient, and restitution (bounciness). Multiple fixtures can be attached to one body, enabling composite collision hulls. The available shape types are:

  • Polygon (b2PolygonShape) — a convex polygon; the axis-aligned box is the most common specialisation.
  • Circle (b2CircleShape) — a disc defined by centre offset and radius; cheap to compute and useful for rolling objects.
  • Edge (b2EdgeShape) — a single line segment, used for individual terrain edges.
  • Chain (b2ChainShape) — a connected sequence of line segments sharing ghost vertices to prevent internal edge collisions; the standard choice for level terrain in OpenClaw.

Joints (b2Joint subclasses) constrain relative motion between two bodies. A distance joint keeps two anchor points a fixed length apart, a revolute joint allows rotation around a shared pivot, and a prismatic joint permits sliding along one axis. In platformers these are used for pendulum traps, conveyor mechanisms, and hinged doors.

Contact Listener Pattern in OpenClaw

Box2D provides a callback-based interface, b2ContactListener, that notifies application code when two fixtures begin or end contact, and when contact impulses are computed. OpenClaw subclasses this listener to drive game logic that depends on what is touching what.

The four virtual methods available are:

  • BeginContact(b2Contact*) — fires when two fixtures first overlap. OpenClaw uses this to detect when Claw lands on solid ground (setting an isGrounded flag), when a projectile touches an enemy, and when the player enters a water volume.
  • EndContact(b2Contact*) — fires when the overlap ceases. Used to clear the isGrounded flag when Claw walks off an edge, and to end water-physics mode when the character exits a water body.
  • PreSolve(b2Contact*, const b2Manifold*) — called before the impulse is computed, allowing the application to disable contact entirely. This is where one-way platform logic lives: if Claw is moving upward or his feet are above the platform surface, the contact is disabled for that step.
  • PostSolve(b2Contact*, const b2ContactImpulse*) — called after impulse resolution, providing the actual impulse magnitude. Useful for impact-damage calculations and sound-effect triggering based on collision force.

Fixtures carry a user data pointer (a raw void* or a typed pointer in newer Box2D versions) that OpenClaw sets to a pointer to the owning game entity. Inside contact callbacks, OpenClaw casts this pointer to its internal ActorComponent type to identify which game object was involved, then dispatches the appropriate game event through the event system.

Platform-Specific Physics Challenges

Platformer games impose requirements on physics that are unusual in other genres. Several challenges arise directly in Captain Claw's gameplay and must be handled carefully on top of Box2D's general-purpose simulation.

One-way platforms (platforms the character can jump up through but lands on top of) are implemented via the PreSolve callback. A fixture category flag marks a surface as one-directional. Each frame, the callback checks the contact normal direction and the player's vertical velocity; if the player is moving upward or the contact normal points downward from the platform's perspective, the contact is disabled. Because PreSolve is called every sub-step, the platform feels solid the moment the player's trajectory inverts.

Slopes create a problem known as ghost collisions: when a body slides along a chain of edge shapes, internal vertices can catch the shape and impart unexpected upward impulses. Box2D's chain shape addresses this by storing ghost vertices — the previous and next segment endpoints — so the solver treats adjacent edges as smoothly connected, preventing snagging.

Moving platforms are kinematic bodies whose velocity is set each frame by a scripted path. Because kinematic bodies transfer momentum, a character standing on a moving platform inherits its horizontal velocity via friction. OpenClaw supplements this by directly adding the platform's velocity delta to the character's velocity when the contact listener confirms the player is grounded on a kinematic surface, preventing the subtle slipping that pure friction sometimes introduces at low frame rates.

Conveyor belts are static bodies (they don't move themselves) whose surface friction is overridden in PreSolve using b2Contact::SetTangentSpeed, which injects a target tangential velocity into the contact, effectively dragging the player character in one direction without requiring the tile geometry to move.

Character Controller Design

The player character's physics body in OpenClaw uses a capsule approximation: a rectangular box fixture for the main body combined with a small circle fixture at the bottom. The circle acts as a foot sensor — its rounded edge allows the character to smoothly ride over small bumps in terrain without catching. A separate, slightly narrower box fixture tagged as a sensor (no collision response) detects the ground below the character, providing the isGrounded state without requiring the main body to be resting on a surface.

Direct velocity manipulation replaces force-based movement for a responsive feel. Instead of applying horizontal forces and relying on friction to decelerate, OpenClaw reads the player's input each frame and sets b2Body::SetLinearVelocity for the horizontal component directly, leaving the vertical component under Box2D's gravity. This produces the snappy, precise feel expected of a traditional platformer without the floaty overshoot that pure force application creates.

Two timing tricks common in quality platformers are also present:

  • Coyote time: after Claw walks off a ledge, a small timer (typically 6–10 frames) continues to allow a jump input to trigger a jump. The isGrounded flag is not immediately cleared when EndContact fires; instead it is cleared only after the coyote window expires without a new ground contact. This makes edge-traversal feel forgiving.
  • Jump buffering: if the player presses the jump button a few frames before landing, the jump fires immediately on landing rather than being dropped. A jump-buffer timer is set on the button press; if the character becomes grounded while the timer is still active, the jump is executed.

Enemy characters follow a simpler pattern: most patrol enemies use kinematic bodies whose velocity is reversed when a terrain-edge sensor detects a cliff or wall ahead, while airborne enemies and bosses use dynamic bodies with scripted force impulses.

Performance and SDL2 Integration

Box2D's collision detection pipeline runs in two stages. The broad phase uses a dynamic AABB tree (b2DynamicTree) to find pairs of bodies whose axis-aligned bounding boxes overlap; this reduces the O(n²) naive pair-testing cost to roughly O(n log n). Only overlapping pairs advance to the narrow phase, where the GJK and EPA algorithms compute exact contact points and penetration depths. Bodies that have been stationary for several steps are put to sleep by the solver, removing them from the simulation until a nearby event wakes them, which substantially reduces CPU cost in large levels.

OpenClaw integrates Box2D with the SDL2 rendering layer through a coordinate-mapping utility. The physics world uses a scale of approximately 50 pixels per metre so that character dimensions map to realistic Box2D units (a character roughly 1.8 m tall corresponds to about 90 pixels). Each frame, the rendering system reads b2Body::GetPosition() and b2Body::GetAngle(), converts from metre-space to pixel-space, and passes the result to the SDL2 sprite renderer.

The physics step is run at a fixed rate (typically 60 Hz) inside the game loop, while SDL2's rendering may run at a different rate on high-refresh displays. OpenClaw computes an alpha interpolation factor from the time accumulated since the last physics step and blends the previous and current physics positions to derive a smooth draw position, preventing the jitter that occurs when the render rate is not a perfect multiple of the physics rate. Debug rendering — drawing Box2D shape outlines directly via SDL2's line-drawing primitives — is available through a compile-time flag, useful for tuning hitboxes and diagnosing collision edge cases.
