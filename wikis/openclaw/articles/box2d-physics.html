プラットフォームゲームにおけるBox2D物理

最終更新: 2026-04-08
Box2D は Erin Catto によるオープンソースの 2D 剛体シミュレーションライブラリで、OpenClaw の物理エンジンを駆動しています。キャラクターの動き・衝突判定・発射物の弾道・環境との相互作用を処理し、クラシックなプラットフォームアクションの挙動を忠実に再現します。

Box2D の概要

プラットフォームゲームにおけるBox2D物理とは、Erin Catto によって書かれた広く使われるオープンソースの 2D 剛体シミュレーションライブラリ Box2D を、横スクロールプラットフォーマーゲームに応用することを指します。Box2D は 2006 年の Game Developers Conference で発表され、それ以降インディおよびホビー系のゲーム開発エコシステムで最も影響力のある物理エンジンのひとつとなり、Angry Birds をはじめ多くの Ludum Dare 出展作品などのエンジンやゲームを支えています。

OpenClaw において、Box2D はシミュレーション全体の権威ある物理レイヤーとして機能します。エンジンはゲームティックごとに固定タイムステップを進める Box2D の world オブジェクトを作成し、物理更新をレンダリングフレームレートから切り離して管理します。Captain Claw・敵・プラットフォーム・はしご・バネ・発射物など衝突可能なすべてのエンティティは Box2D ボディとして表現されます。物理とレンダリングを分離することで、SDL2 レンダリングレイヤーは物理ステップ間の描画位置を補間でき、物理レートとディスプレイリフレッシュレートが異なる場合でも滑らかな動きを実現します。

Box2D の座標系はメートルを長さの単位として使用し、ピクセル座標とは独立して動作します。そのため OpenClaw はスケールファクター（ピクセル/メートル）を維持し、物理ステップの入出力両方の境界で物理空間とスクリーン空間を相互変換します。

コアコンセプト：ボディ・フィクスチャ・ジョイント

Box2D は物理世界を 3 つの主要な抽象化でモデル化します：剛体（rigid body）・フィクスチャ（fixture）・ジョイント（joint）です。

剛体（b2Body）は位置・速度・角速度・質量プロパティを持つ点質量を表します。ボディは 静的（無限の質量を持ち動かない — 地形に使用）・キネマティック（直接速度制御で動くが力で押されない — 動くプラットフォームに使用）・動的（力・重力・衝突のもとで完全にシミュレートされる — プレイヤーキャラクターや敵に使用）のいずれかになります。

フィクスチャ（b2Fixture）はボディに衝突形状をアタッチし、密度（質量に影響）・摩擦係数・反発係数（バウンシネス）などのマテリアルプロパティを持ちます。1 つのボディに複数のフィクスチャをアタッチでき、複合衝突ハルを実現します。利用可能な形状タイプは以下の通りです：

  • Polygon（b2PolygonShape）— 凸多角形。軸揃えボックスが最も一般的な特殊化です。
  • Circle（b2CircleShape）— 中心オフセットと半径で定義される円盤。計算コストが低く、転がるオブジェクトに適しています。
  • Edge（b2EdgeShape）— 1 本の線分。個々の地形エッジに使用されます。
  • Chain（b2ChainShape）— ゴーストバーテックスを共有した連結線分列。内部エッジの衝突を防ぎます。OpenClaw のレベル地形の標準的な選択肢です。

ジョイント（b2Joint サブクラス）は 2 つのボディ間の相対運動を制約します。距離ジョイントは 2 つのアンカーポイントを固定距離に保ち、回転ジョイントは共有ピボット周りの回転を許可し、プリズマティックジョイントは 1 軸方向のスライドを許可します。プラットフォームゲームではこれらを振り子トラップ・コンベアメカニズム・ヒンジ扉に使用します。

OpenClaw におけるコンタクトリスナーパターン

Box2D はコールバックベースのインターフェース b2ContactListener を提供しており、2 つのフィクスチャが接触を開始または終了したとき、および接触インパルスが計算されたときにアプリケーションコードに通知します。OpenClaw はこのリスナーをサブクラス化して、何が何に触れているかに依存するゲームロジックを駆動します。

利用可能な 4 つの仮想メソッドは以下の通りです：

  • BeginContact(b2Contact*) — 2 つのフィクスチャが最初に重なったときに発火します。OpenClaw はこれを使って、Claw が地面に着地したとき（isGrounded フラグをセット）、発射物が敵に当たったとき、プレイヤーが水中に入ったときを検出します。
  • EndContact(b2Contact*) — 重なりが解消されたときに発火します。Claw がエッジから歩き出たときに isGrounded フラグをクリアしたり、キャラクターが水中から出たときに水中物理モードを終了するために使用されます。
  • PreSolve(b2Contact*, const b2Manifold*) — インパルスが計算される前に呼ばれ、アプリケーションが接触を完全に無効化できます。ここに一方通行プラットフォームのロジックが実装されています：Claw が上方向に動いているか、足がプラットフォーム面より上にある場合、そのステップでは接触が無効化されます。
  • PostSolve(b2Contact*, const b2ContactImpulse*) — インパルス解決後に呼ばれ、実際のインパルス大きさを提供します。衝突力に基づく衝撃ダメージ計算や効果音のトリガーに役立ちます。

フィクスチャは ユーザーデータ ポインタ（生の void* または新しい Box2D バージョンではタイプドポインタ）を持ちます。OpenClaw はこれを所有するゲームエンティティへのポインタに設定します。コンタクトコールバック内で OpenClaw はこのポインタを内部の ActorComponent 型にキャストして関係するゲームオブジェクトを特定し、イベントシステムを通じて適切なゲームイベントをディスパッチします。

プラットフォーム固有の物理課題

プラットフォームゲームは他のジャンルでは珍しい要件を物理に課します。Captain Claw のゲームプレイでは複数の課題が直接発生し、Box2D の汎用シミュレーションの上に慎重に対処する必要があります。

一方通行プラットフォーム（キャラクターが下から飛び越えられるが上から乗ることができるプラットフォーム）は PreSolve コールバックで実装されます。フィクスチャのカテゴリフラグが面を一方向として示します。毎フレーム、コールバックは接触法線の方向とプレイヤーの垂直速度を確認します。プレイヤーが上方向に動いているか、接触法線がプラットフォームの視点から下方向を向いている場合、接触は無効化されます。PreSolve は各サブステップごとに呼ばれるため、プレイヤーの軌道が反転した瞬間にプラットフォームが固体として感じられます。

斜面はゴーストコリジョンと呼ばれる問題を引き起こします：ボディがエッジ形状の連鎖に沿ってスライドするとき、内部頂点がボディを引っかけて予期せぬ上方向のインパルスを与えることがあります。Box2D のチェーン形状はゴーストバーテックス（前後のセグメントのエンドポイント）を格納することでこれに対処し、ソルバーが隣接するエッジをスムーズに連結されたものとして扱い、引っかかりを防ぎます。

動くプラットフォームはキネマティックボディで、スクリプト化されたパスによって毎フレーム速度が設定されます。キネマティックボディは運動量を伝達するため、動くプラットフォーム上に立つキャラクターは摩擦によってその水平速度を受け取ります。OpenClaw はコンタクトリスナーがプレイヤーがキネマティックサーフェス上に接地していることを確認した際にプラットフォームの速度デルタをキャラクターの速度に直接加算することでこれを補完し、低フレームレートで純粋な摩擦が生じる微妙なスリップを防ぎます。

コンベアベルトは静的ボディ（自分自身は動かない）で、PreSolve 内で b2Contact::SetTangentSpeed を使って表面摩擦をオーバーライドします。これにより接触に目標の接線速度が注入され、タイルのジオメトリを動かすことなくプレイヤーキャラクターを一方向に引っ張る効果が生まれます。

キャラクターコントローラーの設計

OpenClaw のプレイヤーキャラクターの物理ボディはカプセル近似を使用しています：メインボディ用の矩形ボックスフィクスチャと底部の小さな円フィクスチャを組み合わせたものです。円は足センサーとして機能し、その丸みのあるエッジによってキャラクターが地形の小さな段差をスムーズに乗り越えられます。若干幅が狭い別のボックスフィクスチャがセンサー（衝突応答なし）としてタグ付けされ、メインボディが表面に乗っていることを要求せずにキャラクター下方の地面を検出して isGrounded 状態を提供します。

レスポンシブな感触を実現するため、力ベースの移動の代わりに直接速度操作を採用しています。水平力を適用して摩擦による減速に頼るのではなく、OpenClaw は毎フレームプレイヤーの入力を読み取り、水平コンポーネントに対して直接 b2Body::SetLinearVelocity を設定し、垂直コンポーネントは Box2D の重力のもとに置きます。これにより、純粋な力の適用が生み出すフワフワとした行き過ぎを生じさせることなく、従来のプラットフォーマーに期待されるキビキビとした精密な感触が実現されます。

高品質なプラットフォーマーに共通する 2 つのタイミングテクニックも実装されています：

  • コヨーテタイム：Claw が崖の端から歩き出した後、短時間（通常 6〜10 フレーム）はジャンプ入力でジャンプが実行できます。EndContact が発火しても isGrounded フラグはすぐにクリアされず、コヨーテウィンドウが期限切れになるか新たな地面接触が確認されるまでクリアされません。これにより崖の端の移動が寛容に感じられます。
  • ジャンプバッファリング：着地の数フレーム前にジャンプボタンを押すと、着地と同時にジャンプが即座に発動されます。ボタン押下時にジャンプバッファタイマーが設定され、タイマーが有効な間にキャラクターが接地状態になるとジャンプが実行されます。

敵キャラクターはよりシンプルなパターンに従います。ほとんどのパトロール敵は、前方の地形エッジセンサーが崖や壁を検出すると速度を反転させるキネマティックボディを使用しており、空中の敵やボスはスクリプト化された力のインパルスを持つ動的ボディを使用しています。

パフォーマンスと SDL2 統合

Box2D の衝突検出パイプラインは 2 段階で動作します。ブロードフェーズは動的 AABB ツリー（b2DynamicTree）を使って軸揃えバウンディングボックスが重なるボディのペアを見つけます。これにより素朴な O(n²) のペアテストコストが概ね O(n log n) に削減されます。重なっているペアだけがナローフェーズに進み、GJK および EPA アルゴリズムで正確な接触点と貫通深さが計算されます。数ステップ静止しているボディはソルバーによってスリープ状態に置かれ、近くのイベントで起こされるまでシミュレーションから除外されるため、大規模なレベルでも CPU コストが大幅に削減されます。

OpenClaw は座標マッピングユーティリティを通じて Box2D と SDL2 レンダリングレイヤーを統合します。物理ワールドは約 50 ピクセル/メートルのスケールを使用し、キャラクターの寸法が現実的な Box2D 単位にマッピングされます（約 1.8m の身長のキャラクターは約 90 ピクセルに相当）。毎フレーム、レンダリングシステムは b2Body::GetPosition() と b2Body::GetAngle() を読み取り、メートル空間からピクセル空間に変換して結果を SDL2 スプライトレンダラーに渡します。

物理ステップはゲームループ内で固定レート（通常 60Hz）で実行される一方、SDL2 のレンダリングは高リフレッシュレートのディスプレイでは異なるレートで実行される場合があります。OpenClaw は最後の物理ステップからの経過時間からアルファ補間係数を計算し、前後の物理位置をブレンドして滑らかな描画位置を導出することで、レンダリングレートが物理レートの完全な倍数でない場合に発生するジッターを防ぎます。デバッグレンダリング — SDL2 のライン描画プリミティブで Box2D の形状アウトラインを直接描画 — はコンパイル時フラグで利用可能で、ヒットボックスの調整や衝突エッジケースの診断に役立ちます。
