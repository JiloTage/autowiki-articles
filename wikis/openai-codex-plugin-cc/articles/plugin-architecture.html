Claude Codeプラグインアーキテクチャ

最終更新: 2026-04-01
Claude Codeプラグインシステムの技術的な構造を解説する記事。plugin.jsonマニフェスト、プラグインサーバープロセス、フックシステムとの統合、スラッシュコマンド定義の仕組みを、openai/codex-plugin-ccの実装を具体例として紹介する。

概要

Claude Codeプラグインアーキテクチャは、Anthropicが設計したClaude Codeの拡張機構であり、サードパーティが独自のコマンド・ツール・ワークフロー統合をClaude Codeに追加できるようにするシステムである。プラグインはローカルプロセスとして動作し、Claude Codeのホストプロセスと標準化されたプロトコルで通信する。

このアーキテクチャは以下の4つの中心的なコンポーネントから構成される。plugin.jsonマニフェスト（プラグインの宣言的定義）、プラグインサーバープロセス（実行エンジン）、フックシステム（Claude Codeイベントへの統合）、そしてスラッシュコマンド定義（ユーザーインターフェース層）である。openai/codex-plugin-ccはこのアーキテクチャの公式実装例として広く参照されている。

図1: Claude Codeホストプロセス、プラグインサーバープロセス、外部サービスの三層構造

plugin.jsonマニフェスト

プラグインの宣言的定義はすべてplugin.json（codex-plugin-ccでは.codex-plugin/plugin.json）に記述される。このファイルはClaude Codeのマニフェストローダーが起動時に読み込み、プラグインの登録・機能提供を行う。

マニフェストの主要フィールド

plugin.jsonは以下の主要フィールドで構成される。

  • id: プラグインの一意識別子（例: "openai/codex-plugin-cc"）
  • name: 表示名
  • version: セマンティックバージョン
  • description: 短い説明文
  • server: サーバープロセスの起動コマンドとエントリポイント
  • commands: 提供するスラッシュコマンドの定義配列
  • hooks: 購読するClaude Codeフックイベントのリスト
  • permissions: プラグインが要求するパーミッション（ネットワーク・ファイルシステムアクセス等）

マニフェスト定義例

codex-plugin-ccのマニフェストは以下のような構造を持つ。 

{
  "id": "openai/codex-plugin-cc",
  "name": "Codex Plugin for Claude Code",
  "version": "1.2.0",
  "description": "OpenAI CodexをClaude Codeから直接呼び出す公式ブリッジプラグイン",
  "server": {
    "command": "node",
    "args": ["dist/server.js"],
    "env": { "CODEX_CONFIG": "~/.codex/config.toml" }
  },
  "commands": [
    { "name": "codex:review", "description": "コードレビューをCodexに委譲" },
    { "name": "codex:rescue", "description": "バグ修正タスクをバックグラウンドで実行" }
  ],
  "hooks": ["PreToolUse", "PostToolUse"],
  "permissions": ["network", "process"]
}

プラグインサーバープロセス

プラグインサーバーはClaude Codeとは別の独立したローカルプロセスとして動作する。Claude Codeはプラグインのインストール・有効化時にサーバープロセスを起動し、標準入出力（stdio）またはUnixドメインソケット経由でJSON-RPCプロトコルを用いて通信する。

サーバーライフサイクル

  • 起動: Claude Code起動時またはプラグイン有効化時にserver.commandが実行される
  • ハンドシェイク: サーバーはマニフェスト情報をホストに送信し、ケーパビリティをネゴシエートする
  • 常駐: セッション中はプロセスが常駐し、コマンド呼び出しやフックイベントを待機する
  • 終了: Claude Code終了時またはプラグイン無効化時にプロセスが終了する

プロセス間通信プロトコル

ホストとプラグインサーバー間の通信はJSON-RPCベースのメッセージングで行われる。各メッセージにはmethod（RPC名）、params（引数）、id（リクエスト追跡用）が含まれる。非同期タスク（バックグラウンドジョブ）については、プラグインサーバーが即時にjobIdを返し、進捗はポーリングまたはプッシュ通知で伝達される。

フックシステムとの統合

Claude Codeのフックシステムは、ユーザー操作やツール実行の前後に外部プロセスやプラグインへ制御を渡す仕組みである。フック統合を活用することで、プラグインはClaude Codeのワークフローに深く組み込まれた自動化を実現できる。

図2: フックシステムを通じたプラグインの自動ワークフロー統合。PreToolUseで事前検査、PostToolUseで自動レビューを実行できる。

主要フックタイプ

  • PreToolUse: ツール実行前に呼び出される。プラグインはここでリクエストを検査し、ブロックまたは通過させる判断を行える
  • PostToolUse: ツール実行後に呼び出される。codex-plugin-ccはこのフックでコードレビューゲートを実装している
  • SessionStart / SessionEnd: セッションの開始・終了時に呼び出される。初期化・クリーンアップ処理に利用する
  • FileSave: ファイル保存時に呼び出される。保存前のバリデーションや自動フォーマットに利用できる

codex-plugin-ccでのフック活用例

codex-plugin-ccはPostToolUseフックを利用してコードレビューゲートを実装している。ファイル書き込みツールの実行後にフックが発火し、プラグインサーバーが変更内容をCodexへ送信して自動レビューを実行する。レビュー結果はClaude Codeの出力パネルにインライン表示される。この仕組みにより、フック統合を通じた継続的コード品質チェックが実現されている。

スラッシュコマンド定義

スラッシュコマンドはユーザーがClaude Codeのチャットインターフェースから直接プラグイン機能を呼び出すためのインターフェース層である。各コマンドはplugin.jsonのcommands配列に宣言し、プラグインサーバーのコマンドハンドラに実装する。

コマンド定義スキーマ

各コマンドエントリは以下のフィールドを持つ。

  • name: コマンド名（プレフィックスはプラグインIDから自動付与: codex:review）
  • description: コマンドの説明（スラッシュコマンド補完候補に表示）
  • parameters: 引数スキーマ（JSONSchema形式）
  • async: バックグラウンド非同期実行の可否
  • streaming: ストリーミング応答の可否

コマンドルーティング

ユーザーが/codex:reviewなどのコマンドを入力すると、Claude Codeのコマンドディスパッチャがプラグインレジストリを参照し、対応するプラグインサーバーへJSON-RPCリクエストを送信する。プラグインサーバーは処理結果をMarkdown形式またはリッチテキスト形式で返し、Claude Codeが出力として表示する。

プラグインマーケットプレイスでインストールされたプラグインのコマンドは自動的にClaude Codeのオートコンプリートに登録され、ユーザーは/を入力するだけで利用可能なコマンドをブラウズできる。

セキュリティモデルとパーミッション

Claude Codeプラグインはサンドボックス化された環境で動作し、plugin.jsonのpermissionsフィールドで宣言されたパーミッションのみが許可される。ユーザーはプラグインインストール時に要求パーミッションを確認し、明示的に承認する必要がある。

  • network: 外部ネットワークアクセスの許可（OpenAI APIへの通信等）
  • process: サブプロセスの起動許可（ローカルCodex CLIの呼び出し等）
  • filesystem:read: ファイルシステムの読み取りアクセス
  • filesystem:write: ファイルシステムの書き込みアクセス
  • env: 環境変数へのアクセス許可

codex-plugin-ccはnetworkとprocessパーミッションのみを要求しており、ファイルシステムへの直接アクセスは行わない。これによりセキュリティリスクを最小限に抑えながら、プラグインエコシステムの信頼性を維持している。
