codex:status / codex:result — ジョブ管理コマンド

最終更新: 2026-04-01
codex:status および codex:result は、openai/codex-plugin-cc が提供するスラッシュコマンドであり、codex:rescue などが生成したバックグラウンドジョブの進行状況監視・結果取得・キャンセル操作を担う非同期ワークフロー管理システムの中核をなす。

概要

codex:status（コーデックス・ステータス）と codex:result（コーデックス・リザルト）は、openai/codex-plugin-cc プラグインが提供する一対のジョブ管理コマンドである。これらのコマンドは、codex:rescue コマンドや将来の他のバックグラウンド実行コマンドが生成する非同期ジョブを追跡・管理するために設計された。

OpenAI Codex のクラウド上で実行されるバックグラウンドジョブは、完了までに数十秒から数分を要することがある。このため、ユーザーがClaude Code上での開発作業を継続しつつ、任意のタイミングでジョブの状態を確認し、完了後に結果を取得できる仕組みが不可欠である。codex:status がリアルタイムのステータス照会を、codex:result が完了済みジョブの成果物取得をそれぞれ担う。また、codex:cancel サブコマンドによる中断操作も同一のジョブ管理システムを通じて提供される。

これらのコマンドは デュアルエージェント開発 パターンの中核コンポーネントであり、Claude Code と OpenAI Codex の二つの AI エージェントが協調動作する際のハンドオフ機構として機能する。

図1: ジョブ管理システムの全体像。codex:rescue が生成したジョブを、status・result・cancel コマンドが追跡・取得・中断する。

codex:status コマンド

/codex:status は、バックグラウンドジョブの現在の実行状態をリアルタイムで照会するコマンドである。codex:rescue 実行時に返却された job_id を引数として渡すことで、ジョブの進行状況を取得できる。

基本構文

/codex:status <job_id>
/codex:status                  # 直近のジョブのステータスを確認（job_id省略時）
/codex:status --all            # 全アクティブジョブの一覧表示
/codex:status --watch <job_id> # 完了まで自動ポーリング（30秒間隔）

ジョブステータスの種別

ジョブは以下の5つのステータスを持つ。ステータスは単方向に遷移し、終端状態（done / failed / cancelled）に達したジョブは72時間後に自動削除される。

  • pending — キューに登録済み、まだ実行開始前
  • running — Codex クラウド上で現在実行中
  • done — 正常完了。/codex:result で結果を取得可能
  • failed — 実行失敗。エラー詳細は /codex:result で確認可能
  • cancelled — /codex:cancel によってキャンセル済み

ステータス出力の読み方

/codex:status <job_id> を実行すると、Claude Code のチャット上に構造化された出力が表示される。主なフィールドは以下の通りである。

  • job_id: ジョブの一意識別子（例: cjob_a1b2c3d4）
  • status: 現在のステータス文字列
  • progress: 進捗率（0〜100%）。running 状態でのみ有効
  • elapsed_sec: ジョブ開始からの経過秒数
  • estimated_remaining_sec: 推定残り時間（算出可能な場合のみ）
  • created_at: ジョブ登録日時（ISO 8601形式）
  • model: 実行に使用されたCodexモデルのバージョン
図2: ジョブステータス遷移。pending → running の後、正常終了で done、失敗で failed、キャンセルで cancelled に遷移する。

codex:result コマンド

/codex:result は、done 状態に到達したジョブの成果物をClaude Codeのチャット上に取得・表示するコマンドである。ジョブが failed の場合はエラー詳細が、pending または running の場合は「まだ完了していない」旨のメッセージが表示される。

基本構文

/codex:result <job_id>
/codex:result                  # 直近の完了ジョブの結果を取得
/codex:result --apply <job_id> # 結果のパッチを即時適用（確認ダイアログあり）
/codex:result --format json <job_id> # JSON形式で出力

結果オブジェクトの構造

done ジョブの結果オブジェクトは以下のフィールドを含む。codex:rescue が生成したジョブの場合、バグ修正・調査に特化した構造が返される。

  • job_id: ジョブ識別子
  • status: "done"
  • diagnosis: 問題の原因分析（自然言語テキスト）
  • patch: 修正案の Unified diff 形式パッチ（null の場合あり）
  • confidence: Codex の確信度スコア（0.0〜1.0。0.7未満で警告表示）
  • alternatives: 別の修正アプローチのリスト（省略可能）
  • warnings: 修正適用時の注意点リスト（省略可能）
  • model: 実行に使用した Codex モデル名
  • elapsed_sec: 実際の実行所要時間（秒）

パッチの適用

--apply フラグを指定すると、取得したパッチをClaude Codeのエディタ統合機能経由でワンクリック適用できる。適用前には差分プレビューが表示され、ユーザーが確認してから最終的に適用を決定する。confidence スコアが 0.7 未満の場合、追加の警告ダイアログが表示される。

プラグインアーキテクチャ の設計上、パッチ適用はClaude Code の内部 diff エンジンを呼び出す形で実装されており、コンフリクトが発生した場合はマージエディタが起動する。

codex:cancel によるジョブキャンセル

/codex:cancel は、pending または running 状態のジョブを中断するコマンドである。Claude Code 側でタスクを自己解決した場合など、OpenAI Codex のリソース使用量を節約するために積極的に活用することが推奨される。

基本構文

/codex:cancel <job_id>
/codex:cancel --all            # 全アクティブジョブを一括キャンセル

キャンセル時の挙動

running 状態のジョブをキャンセルした場合、Codex クラウド側での実行は即座に停止されるが、すでに部分的に生成された結果があれば /codex:result で部分結果として取得できる（partial: true フラグが付与される）。pending 状態のジョブはキュー登録が取り消される。

非同期ワークフロー管理の設計

codex:status と codex:result が構成する非同期ワークフロー管理システムは、デュアルエージェント開発 パターンを実現するための中核機構である。Claude Code（Anthropic）と Codex（OpenAI）という二つの AI エージェントが非同期に協調し、開発者の生産性を最大化する。

ポーリング戦略

プラグインは以下の指数バックオフ戦略でステータスをポーリングする（--watch モード時）。

  • 初回確認: ジョブ登録後 10 秒
  • 2回目以降: 前回の待機時間 × 1.5 倍（最大 60 秒間隔）
  • 完了検知後: 自動的にポーリングを停止し、チャット上に通知を表示

セッションをまたいだ job_id の有効性

job_id はローカルの ~/.claude/codex-jobs.json に永続化されるため、Claude Code のセッションを再起動した後でも /codex:status や /codex:result でジョブを追跡できる。ジョブの有効期限は登録から 72 時間であり、期限切れのジョブを照会した場合は expired エラーが返される。

複数ジョブの並列管理

プラグインは同時に最大 5 つのアクティブジョブを管理できる。/codex:status --all を実行することで、全ジョブの一覧と各ステータスをまとめて確認できる。同時実行上限を超えてジョブを登録しようとした場合、最も古い pending ジョブが自動的にキャンセルされ、新しいジョブが登録される。

図3: デュアルエージェント開発ワークフロー。codex:rescue でタスクを委譲し、codex:status で進捗を確認、codex:result で結果を取得する。

エラーハンドリングとトラブルシューティング

/codex:status および /codex:result は、複数のエラーケースを想定した堅牢なエラーハンドリングを実装している。

よくあるエラーと対処法

  • job_not_found: 指定した job_id が存在しない。job_id の綴りを確認するか、/codex:status --all で有効なジョブ一覧を参照する。
  • job_expired: ジョブが 72 時間の有効期限を超えて削除された。codex:rescue で新たにタスクを委譲する。
  • not_done_yet: /codex:result を pending / running 状態のジョブに対して実行した。/codex:status --watch で完了を待機してから再実行する。
  • api_error: OpenAI API との通信エラー。プラグインのAPIキー設定とプラグインアーキテクチャの接続設定を確認する。
  • quota_exceeded: API の利用上限に達した。OpenAI アカウントのクォータを確認する。

デバッグモード

詳細なデバッグ情報を取得したい場合は、--verbose フラグを追加する。これによりAPIリクエスト・レスポンスの詳細、ポーリング回数、実際のリクエストペイロードサイズなどが出力される。 

/codex:status --verbose <job_id>
/codex:result --verbose <job_id>
