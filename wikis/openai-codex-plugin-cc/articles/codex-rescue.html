codex:rescue — タスク委譲機能

最終更新: 2026-04-01
codex:rescueは、openai/codex-plugin-ccが提供するスラッシュコマンドであり、Claude Code上からバグ修正・調査タスクをOpenAI Codexにバックグラウンドで委譲する機能を実装している。ユーザーはメインのコーディング作業を中断することなく、複雑なデバッグ作業を並列実行させることができる。

概要

codex:rescue（コーデックス・レスキュー）は、openai/codex-plugin-ccプラグインが提供するスラッシュコマンドの一つであり、コマンド名が示す通り「救助（rescue）」の役割を担う。すなわち、Claude Codeのユーザーが手詰まりに陥ったバグや難解な調査タスクを、OpenAI Codexに委譲して解決を試みる機能である。

委譲されたタスクはOpenAI側のクラウド環境でバックグラウンド実行されるため、ユーザーはClaude Code上での主たる開発作業を継続できる。タスクの進行状況はcodex:status / codex:resultコマンドを通じて確認・取得できる。

図1: /codex:rescue 実行フロー。タスクはバックグラウンドで実行され、job_idで管理される。

使用方法

/codex:rescueコマンドは、Claude Codeのチャット入力欄からスラッシュコマンドとして呼び出す。以下の形式で使用する。

基本構文

/codex:rescue [問題の説明]

問題の説明を省略した場合、プラグインは現在開いているファイルや直前のエラーメッセージを自動的にコンテキストとして収集し、タスク説明を補完する。明示的に説明を渡すことで、委譲内容をより精密にコントロールできる。

オプションと引数

  • --file <path> — 特定ファイルをコンテキストとして指定する
  • --error <message> — エラーメッセージを直接渡す
  • --branch <name> — 作業対象のGitブランチを指定する（デフォルトは現在のブランチ）
  • --timeout <seconds> — ジョブのタイムアウト秒数を指定する（デフォルト: 300）

使用例

/codex:rescue TypeError: Cannot read property 'map' of undefined が src/components/List.tsx の35行目で発生している

/codex:rescue --file src/utils/parser.ts --error "Stack overflow in recursive call"

/codex:rescue --branch feature/auth 認証トークンのリフレッシュロジックでレースコンディションが疑われる

バックグラウンド実行の仕組み

/codex:rescueの最大の特徴は、タスクをOpenAI Codexのクラウド環境に委譲してバックグラウンドで実行する点にある。このアーキテクチャにより、ユーザーはClaude Code上での主たる開発作業を継続しながら、重い調査・修正タスクを並列処理させることができる。

ジョブのライフサイクル

コマンド実行後、プラグインは以下の手順でタスクを処理する。

  1. コンテキスト収集: 現在のファイル内容・スタックトレース・Gitの差分などを自動収集する
  2. ペイロード構築: 収集したコンテキストをOpenAI APIが解釈できる形式に変換する
  3. ジョブ登録: OpenAI APIにリクエストを送信し、job_idを受け取る
  4. 非同期実行: Codexクラウド上でバグ解析・修正案生成が進行する
  5. 結果取得: codex:resultコマンドで完了時に結果を取得する

job_idによる管理

ジョブ登録後、プラグインはClaude Codeのチャット上にjob_idを表示する。この識別子は以下の用途に使用できる。

  • /codex:status <job_id> — 進行状況（pending / running / done / failed）を確認する
  • /codex:result <job_id> — 完了したジョブの修正案・調査結果を取得する
  • /codex:cancel <job_id> — 実行中のジョブをキャンセルする

job_idの管理にはcodex:status / codex:resultの専用記事も参照のこと。

コンテキスト収集戦略

/codex:rescueが送信するペイロードの品質は、収集するコンテキストの精度に大きく依存する。プラグインアーキテクチャの設計上、プラグインはClaude Codeの内部状態へのアクセス権を持ち、以下の情報を自動収集する。

自動収集される情報

  • 現在アクティブなファイルの内容（最大8,000トークン相当）
  • 直前のエラーメッセージとスタックトレース
  • 現在のGitブランチ名とdiff（最大200行）
  • 関連するpackage.jsonまたはpyproject.tomlの依存情報
  • Claude Codeの直近の会話ターン（最大5ターン）

コンテキストの制限

OpenAI APIへのペイロードは合計で最大32,000トークンに制限される。収集したコンテキストがこの上限を超える場合、プラグインは優先度順にトリミングを行う。優先順位は「エラーメッセージ > アクティブファイル > Gitdiff > 会話履歴」の順である。

図2: コンテキスト収集の優先度。上限に達した場合は優先度の低いものからトリミングされる。

出力形式と結果の活用

/codex:rescueが完了すると、/codex:result <job_id>で以下の形式の結果を取得できる。

結果の構造

  • diagnosis: バグの原因分析（自然言語による説明）
  • patch: 修正案のUnified diff形式パッチ
  • confidence: Codexの確信度スコア（0.0〜1.0）
  • alternatives: 別の修正アプローチの候補リスト（任意）
  • warnings: 修正適用時の注意点（任意）

結果の適用

取得したpatchはClaude Codeのエディタ統合機能を通じてワンクリックで適用可能である。ただし、自動適用前にClaude Codeがdiffプレビューを表示するため、ユーザーが内容を確認してから適用を決定できる。confidence スコアが0.7未満の場合は警告メッセージが表示される。

他コマンドとの連携

/codex:rescueは単体で使用するほか、他のスラッシュコマンドと組み合わせることで効果的なデバッグワークフローを構築できる。

  • rescue + status: 委譲後にステータスポーリングを設定し、完了時に通知を受け取る
  • rescue + adversarial-review: 修正案を対抗的レビューにかけ、品質を二重確認する
  • rescue + cancel: Claude Code側で問題を自己解決できた場合にジョブを途中でキャンセルする

プラグインアーキテクチャの設計上、これらのコマンド間は共通のジョブキューを通じてステート共有されており、セッションをまたいでもjob_idが有効である（最大72時間）。
