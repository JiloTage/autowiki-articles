npm / GitHub Packagesによるプライベートスキル配布

最終更新: 2026-03-30
GitHub Packagesのプライベートnpmレジストリを活用してClaude Code Skillsを社内に配布する方法。セマンティックバージョニングによる厳密なバージョン管理と、package.jsonによる依存関係の記録が可能。

概要

GitHub Packagesを使ったnpm配布は、GitHubのプライベートリポジトリをnpmレジストリとして活用し、Claude Code Skillsを組織内に配布する方法だ。Skills共有方法の概要で示した4つのアプローチの中で最も成熟したバージョン管理機能を提供する。

前提条件

この方法を実装するには以下が必要となる：

  • GitHub OrganizationまたはPersonal Accountのプライベートリポジトリ
  • GitHub Personal Access Token（packages:read, packages:write スコープ）
  • 各開発者の環境にNode.js / npm インストール済み

パッケージリポジトリのセットアップ

リポジトリ構造

Skillsを格納するプライベートリポジトリを以下の構造で作成する：

claude-skills/
├── package.json
├── README.md
├── commands/
│   ├── deploy.md
│   ├── review-pr.md
│   ├── generate-tests.md
│   └── run-ci.md
└── scripts/
    └── postinstall.js

package.jsonの設定

パッケージをGitHub Packagesに公開するための `package.json` 設定：

{
  "name": "@myorg/claude-skills",
  "version": "1.0.0",
  "description": "社内共有Claude Code Skills",
  "private": false,
  "publishConfig": {
    "registry": "https://npm.pkg.github.com",
    "access": "restricted"
  },
  "scripts": {
    "postinstall": "node scripts/postinstall.js"
  },
  "files": [
    "commands/",
    "scripts/"
  ]
}

インストール後自動リンクスクリプト

インストール後にSkillsを自動的に `.claude/commands/` に配置する `scripts/postinstall.js`：

const fs = require('fs');
const path = require('path');

const projectRoot = process.env.INIT_CWD || process.cwd();
const claudeCommandsDir = path.join(projectRoot, '.claude', 'commands');
const skillsDir = path.join(__dirname, '..', 'commands');

// .claude/commands/ ディレクトリを作成
fs.mkdirSync(claudeCommandsDir, { recursive: true });

// 各Skillファイルにシンボリックリンクを作成
const skills = fs.readdirSync(skillsDir).filter(f => f.endsWith('.md'));
for (const skill of skills) {
  const target = path.join(skillsDir, skill);
  const link = path.join(claudeCommandsDir, skill);
  if (!fs.existsSync(link)) {
    fs.symlinkSync(target, link);
    console.log(`Linked: ${skill}`);
  }
}

パッケージの公開

認証設定（.npmrc）

パッケージ公開者のホームディレクトリに `~/.npmrc` を設定する：

//npm.pkg.github.com/:_authToken=ghp_YOUR_GITHUB_TOKEN
@myorg:registry=https://npm.pkg.github.com

公開コマンド

# バージョンを更新してパブリッシュ
npm version patch  # または minor / major
npm publish

利用者側のインストール

利用者の.npmrc設定

プロジェクトルートに `.npmrc` ファイルを作成（Gitにコミット可能）：

@myorg:registry=https://npm.pkg.github.com

認証トークンはホームディレクトリの `~/.npmrc` に記述し、リポジトリにはコミットしない：

//npm.pkg.github.com/:_authToken=${GITHUB_TOKEN}

インストールコマンド

# プロジェクトに追加
npm install --save-dev @myorg/claude-skills

# または特定バージョンを固定
npm install --save-dev @myorg/claude-skills@1.2.0

CI/CD連携

GitHub ActionsでGITHUB_TOKENを使うと、追加設定なしでインストールできる：

- name: Install Claude Skills
  run: npm install
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

バージョン管理戦略

セマンティックバージョニング（semver）を活用することで、破壊的変更を安全に管理できる：

  • patch（1.0.x）: バグ修正、表現の改善
  • minor（1.x.0）: 新しいSkillの追加
  • major（x.0.0）: 既存Skillのインターフェース変更・削除

メリット・デメリット

この方法の主なメリットは、明示的なバージョン固定（`package.json` の `devDependencies`）により、チーム全員が同じSkillバージョンを使用できる点だ。また、`npm outdated` で更新確認ができ、`npm update` で一括更新も容易。デメリットは初期設定コストの高さと、開発者全員がGitHub Personal Access Tokenを設定する必要がある点だ。セットアップ自動化スクリプトでトークン設定を自動化することを推奨する。
