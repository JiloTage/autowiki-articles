モノレポ構成でのClaude Code Skills管理

最終更新: 2026-03-30
すべてのプロジェクトを単一リポジトリで管理するモノレポ構成において、Claude Code Skillsを一元管理する方法。高い可視性と即時反映が可能だが、リポジトリ規模拡大に伴う管理コストも考慮が必要。

概要

モノレポ（monorepo）構成とは、複数のプロジェクト・パッケージを単一のGitリポジトリで管理する手法だ。この構成を採用している組織では、Claude Code Skillsも同じリポジトリ内に配置することで、Skills共有方法の概要で示した他のアプローチよりもシームレスな管理が実現できる。

モノレポ内のSkills配置

構造例

典型的なモノレポ内でのSkills配置は以下のような形になる：

myorg-monorepo/
├── .claude/
│   └── commands/           # モノレポ全体で共有するSkills
│       ├── deploy.md
│       ├── review-pr.md
│       └── run-ci.md
├── packages/
│   ├── frontend/
│   │   └── .claude/
│   │       └── commands/   # frontendパッケージ固有のSkills
│   │           └── storybook.md
│   ├── backend/
│   │   └── .claude/
│   │       └── commands/   # backendパッケージ固有のSkills
│   │           └── migrate-db.md
│   └── shared/
├── apps/
│   ├── web/
│   └── mobile/
└── tools/
    └── scripts/
図1: モノレポにおけるSkillsのスコープ階層

Claude CodeのスコープルールとSkills探索

Claude Codeは作業ディレクトリから上位ディレクトリへ順に `.claude/commands/` を探索し、Skillsを収集する。つまりモノレポ内の各パッケージで作業する際、以下のSkillsが自動的に利用可能になる：

  1. 現在のディレクトリの `.claude/commands/`（パッケージ固有）
  2. 親ディレクトリの `.claude/commands/`（モノレポ全体共有）
  3. ホームディレクトリの `~/.claude/commands/`（個人設定）

Nx / Turborepoとの統合

NxやTurborepoなどのモノレポツールを使っている場合、Skills管理をさらに自動化できる。

Nx Generatorによる自動配置

# Nxジェネレータでパッケージ作成時にSkillsも自動配置
nx generate @myorg/workspace:library my-lib --add-claude-skills

ワークスペース設定でのSkills参照

# turbo.json でSkills関連タスクを定義
{
  "pipeline": {
    "install-skills": {
      "outputs": [".claude/commands/**"]
    }
  }
}

スキルのバージョニングとレビュー

モノレポの最大のメリットは、SkillsへのPRがプロジェクトコードのPRと同じフローでレビューできる点だ。変更の影響範囲がリポジトリ内で即座に可視化され、CIでテストも実行できる。

Skillsの自動テスト例

# .github/workflows/test-skills.yml
name: Test Claude Skills
on: [push, pull_request]
jobs:
  validate-skills:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate skill files
        run: |
          for skill in .claude/commands/*.md; do
            # frontmatterの存在確認
            if ! grep -q "^---" "$skill"; then
              echo "Missing frontmatter: $skill"
              exit 1
            fi
          done

メリット・デメリット

モノレポアプローチの最大の強みは、Skills変更とコード変更を1つのコミット・PRで原子的に行える点だ。Skills定義がコードと同期したままの状態が保たれる。一方で、モノレポ自体の管理コストが高く、リポジトリ規模が大きくなるにつれてCI実行時間の増加など課題も生じる。すでにモノレポを採用している組織なら追加コストなく導入できるが、そうでない場合はオーバーエンジニアリングになる可能性が高い。他のアプローチと組み合わせたセットアップ自動化スクリプトとの併用も検討すること。
