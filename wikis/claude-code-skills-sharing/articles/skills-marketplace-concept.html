Claude Code プラグインマーケットプレイスの現状と活用ガイド

最終更新: 2026-03-31
Claude Code には公式のプラグインシステムとマーケットプレイスが存在する。/plugin コマンドによるインストール、公式・コミュニティ・組織内の三層構造、カスタムマーケットプレイスの作成方法を解説する。

Claude Code プラグインシステムの概要

Claude Code には 公式プラグインシステム が実装されており、スラッシュコマンド・サブエージェント・MCPサーバー・フックをひとつの配布単位（プラグイン）にパッケージして共有できる。Skills共有方法の概要で紹介した手動的なアプローチを大きく超え、/plugin コマンドひとつで検索・インストール・管理が完結する。

プラグインが含むことができるコンポーネントは以下の4種類である。

  • スラッシュコマンド（Skills）: .claude/commands/ 配下のMarkdownファイル。繰り返し実行するタスクを定型化する。
  • サブエージェント: 特定の開発タスク（コードレビュー、テスト生成など）に特化した目的型エージェント。
  • MCPサーバー: GitHub、Slack、Figmaなど外部ツールへの接続を提供するModel Context Protocolサーバー。
  • フック: ツール呼び出し前後などのライフサイクルイベントにカスタム処理を追加する仕組み。

/plugin コマンドの使い方

/plugin コマンドはタブ型UIを提供し、以下の操作が行える。

図1: /plugin コマンドのタブUI（Discover・Installed・Marketplaces・Errors）

CLIからのインストール

プラグインはCLIから直接インストールすることも可能だ。

# 公式マーケットプレイスからインストール
/plugin install <plugin-name>@claude-plugins-official

# コミュニティマーケットプレイスを追加
/plugin marketplace add <user-or-org>/<repo-name>

# 追加されたマーケットプレイスから検索・インストール
/plugin install <plugin-name>

三層マーケットプレイス構造

Claude Code のプラグインエコシステムは、公開範囲と管理主体によって三つの層に分類できる。

図2: Claude Code プラグインエコシステムの三層構造

公式マーケットプレイス（Anthropic管理）

Anthropicが運営する公式マーケットプレイス（claude-plugins-official）は、Claude Code 起動時に自動的に登録される。フロントエンドデザイン、GitHub連携、Slack統合、コード解析など多様なプラグインが公開されており、/plugin コマンドから即時インストールできる。

開発したプラグインを公式マーケットプレイスに提出する場合は以下の窓口を利用する。

  • claude.ai 経由: claude.ai/settings/plugins/submit
  • Console経由: platform.claude.com/plugins/submit

Anthropicの審査プロセスを経て、セキュリティ・品質が確認されたプラグインのみが掲載される。

コミュニティマーケットプレイス

コミュニティが主体となるマーケットプレイスは、Git リポジトリに .claude-plugin/marketplace.json ファイルを配置することで誰でも構築できる。/plugin marketplace add user/repo コマンドで追加し、そこに収録されたプラグインを通常の /plugin install でインストールできる。

代表的なコミュニティリソースとして awesome-claude-code（GitHub）がある。スラッシュコマンド・スキル・フック・プラグインをキュレーションした包括的なリスト集であり、コミュニティが継続的に更新している。

組織内マーケットプレイス

企業・組織内部のプライベートマーケットプレイスは、プライベートリポジトリで運営する。セントラルリポジトリアプローチやモノレポ構成をプラグイン形式に昇格させた形態だ。組織のマネージド設定を使えば、全メンバーのClaude Codeにプラグインを自動配布できる。

カスタムマーケットプレイスの構築

コミュニティや組織向けの独自マーケットプレイスを構築するには、Gitリポジトリに以下の構成を用意する。

my-team-marketplace/
├── .claude-plugin/
│   └── marketplace.json   ← マーケットプレイス定義
├── plugins/
│   ├── code-review/
│   │   ├── PLUGIN.md
│   │   └── commands/
│   └── deploy-helper/
│       ├── PLUGIN.md
│       └── commands/
└── README.md

marketplace.json の構造

{
  "name": "My Team Marketplace",
  "description": "社内Claude Code プラグイン集",
  "plugins": [
    {
      "name": "code-review",
      "title": "コードレビュー支援",
      "description": "PRレビューを自動化するプラグイン",
      "path": "plugins/code-review"
    },
    {
      "name": "deploy-helper",
      "title": "デプロイ自動化",
      "description": "デプロイフローを一元管理するプラグイン",
      "path": "plugins/deploy-helper"
    }
  ]
}

このリポジトリを GitHub に公開（または社内のプライベートリポジトリに配置）した後、チームメンバーが /plugin marketplace add <org>/my-team-marketplace を実行すると、プラグインが検索可能になる。

プラグイン配布 vs 直接共有の使い分け

プラグインシステムを使う方法と、npmパッケージ・Gitサブモジュール・セントラルリポジトリなどの直接共有方法は、用途によって使い分ける。

  • プラグイン配布: Claude Code エコシステム内での発見性を高めたい場合、スラッシュコマンド以外の機能（MCPサーバー・フック）も含める場合、広く公開したい場合に適している。
  • npm / GitHub Packages: バージョン管理の厳密性が求められる場合、CI/CDパイプラインとの統合が必要な場合に適している。
  • Gitサブモジュール・セントラルリポジトリ: 小〜中規模チームで追加ツール導入を避けたい場合、既存のGitワークフローをそのまま活用したい場合に適している。

セキュリティとガバナンス

プラグインはコマンド実行・ファイル操作・ネットワークアクセスなど強力な操作を行う可能性があるため、マーケットプレイス選択とインストール判断は慎重に行う必要がある。

権限宣言と最小権限の原則

各プラグインが要求する権限は allowed-tools フィールドで宣言され、インストール前に確認できる。最小権限の原則に基づき、必要最小限の権限のみを要求するプラグインを選択することが推奨される。公式マーケットプレイス掲載のプラグインはAnthropicのセキュリティ審査を通過しているため、コミュニティプラグインより信頼度が高い。

ソース確認

コミュニティマーケットプレイスから取得するプラグインは、インストール前にソースコードを確認し、不審な操作が含まれていないかをチェックすることが重要だ。Skillsの品質管理とコードレビュープロセスに記載する審査チェックリストを適用するとよい。
