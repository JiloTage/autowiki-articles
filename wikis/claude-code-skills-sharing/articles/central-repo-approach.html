セントラルリポジトリからのスキル配布

最終更新: 2026-03-30
最もシンプルなClaude Code Skills共有方法。Skillsを集約した専用プライベートリポジトリを作成し、各開発者がクローンして手動またはスクリプトでセットアップする。追加ツール不要で即日導入できる。

概要

セントラルリポジトリアプローチは、Claude Code Skillsを一か所に集めたプライベートGitHubリポジトリを「Single Source of Truth」として運用し、各開発者がそこからクローンして利用する最もシンプルな方法だ。Skills共有方法の概要で示した4つのアプローチの中で最も導入が速い。

このアプローチが適している状況

  • チーム規模が小さい（5〜15名程度）
  • Git以外の追加ツールを導入したくない
  • バージョン管理の厳密さよりも導入速度を優先
  • Claude Code導入の初期フェーズ（後から他アプローチに移行可能）

セットアップ手順

Skills専用リポジトリの作成

GitHubでプライベートリポジトリ（例: `myorg/claude-skills`）を作成し、以下の構造でSkillsを配置する：

claude-skills/
├── README.md          # 使い方ガイド
├── CHANGELOG.md       # 変更履歴
├── commands/
│   ├── deploy.md
│   ├── review-pr.md
│   ├── generate-tests.md
│   └── run-migrations.md
└── install.sh         # ワンコマンドインストールスクリプト

インストールスクリプト（install.sh）

#!/bin/bash
# Claude Code Skills インストールスクリプト
set -euo pipefail

SKILLS_REPO="git@github.com:myorg/claude-skills.git"
SKILLS_DIR="$HOME/.claude-skills"
COMMANDS_DIR="$HOME/.claude/commands"

# Skills リポジトリをクローン（または更新）
if [ -d "$SKILLS_DIR" ]; then
  echo "Updating claude-skills..."
  git -C "$SKILLS_DIR" pull --ff-only
else
  echo "Cloning claude-skills..."
  git clone "$SKILLS_REPO" "$SKILLS_DIR"
fi

# グローバルコマンドディレクトリを作成
mkdir -p "$COMMANDS_DIR"

# 各Skillファイルをシンボリックリンク
for skill in "$SKILLS_DIR/commands/"*.md; do
  skill_name=$(basename "$skill")
  link="$COMMANDS_DIR/$skill_name"
  if [ -L "$link" ]; then
    rm "$link"  # 既存リンクを更新
  fi
  ln -s "$skill" "$link"
  echo "  Linked: $skill_name"
done

echo "Done! Skills available in Claude Code."

グローバル配置 vs プロジェクト配置

グローバル配置（推奨）

`~/.claude/commands/` に配置することで、すべてのプロジェクトでSkillsが利用可能になる。組織全員が使う汎用Skillsはこの方法が適している：

install.sh でのリンク先: ~/.claude/commands/

プロジェクト配置

特定プロジェクトのみに適用したいSkillsは、プロジェクトルートの `.claude/commands/` に配置する：

install.sh でのリンク先: ./PROJECT_ROOT/.claude/commands/

更新通知の仕組み

Skillsが更新されたとき、開発者に通知する仕組みが重要だ。以下のアプローチが考えられる：

  • Slackチャンネル通知: GitHub ActionsでリポジトリにPushされたときにSlack通知
  • CHANGELOG.md: 変更内容を記録し、PRのdescriptionで案内
  • 自動更新スクリプト: cron job（月次など）でシステム的に更新

メリット・デメリット

このアプローチの最大の強みは導入の速さだ。リポジトリを作成してinstall.shを書けば、その日のうちに全チームに展開できる。バージョン固定が弱い（常にlatest）点はデメリットだが、小規模チームではむしろ常に最新Skillsを使えるメリットになる。チームが成長したらGitサブモジュールアプローチやnpm / GitHub Packagesに移行することを検討する。セットアップ自動化スクリプトと組み合わせることで、開発者のオンボーディングコストを最小化できる。
