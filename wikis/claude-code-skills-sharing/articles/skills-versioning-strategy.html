Skillsリポジトリのバージョニング戦略とリリース管理

最終更新: 2026-03-31
Claude Code Skillsを共有リポジトリで管理する際のバージョニング戦略を体系的に解説する。セマンティックバージョニング、Gitタグ戦略、CHANGELOG自動生成、ブランチ戦略、リリースノートの作成まで、安定した配布基盤の構築方法を紹介する。

概要

Skillsリポジトリのバージョニング戦略とリリース管理は、Claude Code Skillsを社内で安定かつ継続的に共有するための重要な基盤だ。Skills共有方法の概要で紹介した各アプローチに共通して適用できる考え方であり、特にGitサブモジュールアプローチやnpm / GitHub Packagesアプローチにおいてバージョニングは中核的な役割を担う。適切なバージョン管理なしには、どのプロジェクトがどのSkillsを使っているかの追跡が困難になり、破壊的変更の影響範囲の把握も難しくなる。

セマンティックバージョニングの適用

Skillsリポジトリにはセマンティックバージョニング（SemVer）の適用が強く推奨される。SemVerは MAJOR.MINOR.PATCH の形式でバージョンを表現し、各数値の意味が明確に定義されている。

図1: SemVerの各数値が示す変更の種類と対応するバージョンアップ例

Skillsにおける破壊的変更の例としては、Skillのスラッグ（ファイル名）変更、必須引数の追加、動作の根本的な変更などが挙げられる。後方互換な変更には、新しいSkillファイルの追加や、既存Skillへのオプション引数追加が含まれる。

Gitタグ戦略

バージョンの記録にはGitタグを使用する。アノテーテッドタグ（annotated tag）はタグ作成者・日時・メッセージを記録できるため、ライトウェイトタグより推奨される。

タグの作成と命名規則

# アノテーテッドタグの作成（推奨）
git tag -a v2.4.1 -m "fix: deploy skillのコメント誤記を修正"

# タグをリモートにプッシュ
git push origin v2.4.1

# 全タグをまとめてプッシュ
git push origin --tags

# タグ一覧の確認
git tag -l "v*" --sort=-version:refname

命名規則は v{MAJOR}.{MINOR}.{PATCH} に統一し、プレリリース版には v2.5.0-beta.1 や v2.5.0-rc.1 のサフィックスを付与する。

サブモジュールでのバージョン固定

Gitサブモジュールアプローチでは、タグを使ったバージョン固定が重要なプラクティスだ：

# サブモジュールを特定バージョンに固定
cd .claude/shared-commands
git checkout v2.4.1
cd ../..
git add .claude/shared-commands
git commit -m "chore: claude-skills を v2.4.1 に更新"

ブランチ戦略

Skillsリポジトリのブランチ戦略は、複雑なGit Flowよりもシンプルな構成が適している。推奨構成は main ブランチと stable ブランチの2本立てだ。

図2: main（開発）とstable（リリース済み）の2ブランチ構成

各ブランチの役割

main ブランチは日常的な開発の主軸であり、新しいSkillの追加・修正がすべてここに集約される。stable ブランチはリリース済みの安定版を示し、リリースタイミングで main からマージする。チームによっては stable ブランチを省略し、タグだけで管理するシンプルな構成も有効だ。

CHANGELOGの自動生成

バージョン間の変更内容を記録するCHANGELOGは、手動管理よりも自動生成ツールを活用することで継続的なメンテナンスが容易になる。コミットメッセージにConventional Commits形式を採用することがその前提条件となる。

Conventional Commitsの採用

# Conventional Commits 形式の例
feat: review-pr スキルに --draft オプションを追加
fix: deploy スキルの環境変数参照エラーを修正
docs: README にバージョニングポリシーを追記
refactor: generate-tests スキルの引数処理を整理
BREAKING CHANGE: commit スキルの引数形式を変更

git-cliffによるCHANGELOG自動生成

git-cliffはRust製のCHANGELOG生成ツールで、Conventional Commitsを解析してMarkdown形式のCHANGELOGを自動生成する：

# cliff.toml の基本設定例
[changelog]
header = "# Changelog\n\n"
body = """
## {{ version }} - {{ timestamp | date(format="%Y-%m-%d") }}
{% for group, commits in commits | group_by(attribute="group") %}
### {{ group | upper_first }}
{% for commit in commits %}
- {{ commit.message }}{% endfor %}
{% endfor %}
"""

[git]
conventional_commits = true
commit_parsers = [
  { message = "^feat", group = "Features" },
  { message = "^fix", group = "Bug Fixes" },
  { message = "^docs", group = "Documentation" },
  { message = "^refactor", group = "Refactoring" },
]
# バージョンタグ付きでCHANGELOGを生成
git cliff --tag v2.5.0 -o CHANGELOG.md

# 最新タグから現在のHEADまでの差分を表示
git cliff --unreleased

リリースワークフロー

バージョンリリースの手順を標準化することで、誰がリリースを担当しても一貫した品質を維持できる。以下のワークフローをチームで共有し、セットアップ自動化スクリプトに組み込むことを推奨する。

リリースチェックリスト

#!/bin/bash
# release.sh - Skillsリポジトリのリリーススクリプト
set -e

VERSION=$1
if [ -z "$VERSION" ]; then
  echo "Usage: ./release.sh v2.5.0"
  exit 1
fi

echo "=== Skillsリポジトリ リリース: $VERSION ==="

# 1. mainブランチであることを確認
BRANCH=$(git branch --show-current)
if [ "$BRANCH" != "main" ]; then
  echo "Error: mainブランチで実行してください（現在: $BRANCH）"
  exit 1
fi

# 2. ローカルが最新であることを確認
git pull origin main

# 3. CHANGELOGを更新
git cliff --tag $VERSION -o CHANGELOG.md
git add CHANGELOG.md
git commit -m "chore: CHANGELOG を $VERSION 用に更新"

# 4. タグを作成
git tag -a $VERSION -m "release: $VERSION"

# 5. stableブランチをマージ
git checkout stable
git merge main --no-ff -m "release: $VERSION を stable にマージ"
git checkout main

# 6. リモートにプッシュ
git push origin main stable
git push origin $VERSION

echo "=== リリース完了: $VERSION ==="

リリースノートの作成

GitHubのReleasesページにリリースノートを公開することで、利用者はバージョンアップ時の変更点を容易に確認できる。CHANGELOGから自動生成する方法が効率的だ：

# GitHub CLI を使ったリリース作成
# CHANGELOGから最新バージョンのノートを抽出してリリース
NOTES=$(git cliff --latest --strip all)

gh release create v2.5.0 \
  --title "v2.5.0 - 新規Skill追加とバグ修正" \
  --notes "$NOTES" \
  --target stable

バージョンポリシーの文書化

バージョニングポリシーはSkillsリポジトリのREADMEまたは専用の VERSIONING.md に明文化し、チーム全体が参照できるようにする。特に以下の項目を明示することが重要だ。

サポートポリシー

MAJORバージョンのサポート期間（例：「現行MAJOR版と1つ前のMAJOR版をサポート」）を定めることで、利用者がアップグレードを計画しやすくなる。セントラルリポジトリアプローチやモノレポ構成を採用している場合でも、このポリシーは有効だ。

非推奨化プロセス

Skillを廃止する際には、MINORバージョンで非推奨警告を追加し、次のMAJORバージョンで削除するという段階的なプロセスを採用する。非推奨のSkillファイルの冒頭に以下のような警告コメントを追記することで、利用者への通知が可能だ：


# deploy (非推奨)
