Quickwitクエリ言語

最終更新: 2026-04-05
QuickwitクエリはElasticsearch Query DSLと高い互換性を持ちながら、クラウドネイティブな分散検索エンジン向けに最適化されたクエリ言語・APIである。全文検索・フィルター・集計・スクロールなど多彩な機能を提供し、既存のElasticsearchエコシステムとの親和性を保ちつつQuickwit固有のパフォーマンス特性を活かす設計となっている。

概要

QuickwitのQuickwitクエリ言語は、Elasticsearch Query DSLをベースとしたクエリインターフェースを提供する。これにより、既存のElasticsearch/OpenSearchユーザーは最小限の変更でQuickwitへ移行できる。一方で、Quickwitはオブジェクトストレージ上のイミュータブルなスプリット構造を前提としたクエリプランナーを持ち、Apache Tantivyを通じた高速な全文検索と列指向ストレージを活用する独自の最適化が施されている。

クエリは大きく3つのカテゴリに分類される: 全文検索クエリ（matchやmatch_phraseなど）、フィルタークエリ（term、range、boolなど）、集計クエリ（terms、histogram、date_histogramなど）。これらはJSON形式のクエリDSLとして記述し、REST APIを通じてQuickwitに送信する。

図1: Quickwitのクエリ処理フロー。クライアントからJSON Query DSLがREST APIに届き、クエリプランナーがスプリット剪定と並列実行計画を立てた上でApache Tantivyによる検索が実行される。

全文検索クエリ

全文検索クエリはテキストフィールドに対してトークン化・アナライズを経た検索を行う。QuickwitではApache Tantivyのアナライザー設定に基づいてクエリテキストが処理される。

matchクエリ

matchクエリは最も基本的な全文検索クエリである。指定したフィールドに対してテキストを分析し、含まれるトークンのいずれかが一致するドキュメントを返す（デフォルトはOR論理）。

{
  "query": {
    "match": {
      "message": {
        "query": "connection timeout error",
        "operator": "AND"
      }
    }
  }
}

operatorを"AND"に設定すると、すべてのトークンが含まれるドキュメントのみが返される。デフォルトの"OR"はいずれかのトークンが含まれれば一致とみなす。また、fuzzinessパラメータを設定することでファジーマッチも可能である。

match_phraseクエリ

match_phraseクエリはフレーズとして連続するトークンが同じ順序で出現するドキュメントを検索する。ログ解析でエラーメッセージの特定フレーズを探す際に有効である。

{
  "query": {
    "match_phrase": {
      "message": "disk quota exceeded"
    }
  }
}

slopパラメータを追加すると、フレーズ内のトークン間に許容する位置ずれ数を指定できる。

query_stringクエリ

query_stringクエリはLucene構文に基づいた表現力の高いクエリ文字列をサポートする。複数フィールドへのクロスフィールド検索、ワイルドカード、正規表現などが使用できる。

{
  "query": {
    "query_string": {
      "query": "level:ERROR AND service:\"auth-service\"",
      "default_field": "message"
    }
  }
}

フィルタークエリ

フィルタークエリはスコアリングを行わず、ドキュメントが条件に一致するかどうかのみを判定する。スコア計算が不要なため全文検索クエリより高速であり、Quickwitのクエリプランナーはフィルター条件を使ってスプリット剪定を積極的に行う。

termクエリ

termクエリは特定フィールドの値が完全に一致するドキュメントを検索する。キーワードフィールドやIDフィールドに対して使用する。

{
  "query": {
    "term": {
      "status": "error"
    }
  }
}

複数の値に対する一致検索にはtermsクエリを使用する。

rangeクエリ

rangeクエリは数値・日時フィールドの範囲検索に使用する。Quickwitの列指向ストレージを活用し効率的に処理される。タイムスタンプフィールドへのrange指定は、スプリット剪定に直結するため検索パフォーマンスに大きく影響する。

{
  "query": {
    "range": {
      "timestamp": {
        "gte": "2026-04-01T00:00:00Z",
        "lt":  "2026-04-02T00:00:00Z",
        "format": "strict_date_optional_time"
      }
    }
  }
}

boolクエリ

boolクエリは複数のクエリを論理的に組み合わせる複合クエリである。以下の4つの節を持つ:

  • must: すべての条件が一致し、スコアに寄与する
  • filter: すべての条件が一致するが、スコアには寄与しない（キャッシュ効率が高い）
  • should: いずれかの条件が一致すると好ましい（スコア加点）
  • must_not: いずれの条件にも一致しない（除外条件）
{
  "query": {
    "bool": {
      "must": [
        { "match": { "message": "authentication failed" } }
      ],
      "filter": [
        { "term":  { "service": "gateway" } },
        { "range": { "timestamp": { "gte": "now-1h" } } }
      ],
      "must_not": [
        { "term": { "level": "debug" } }
      ]
    }
  }
}

実務上はfilter節でタイムスタンプや固定条件を絞り込み、must節で全文検索スコアリングを行うパターンが最も効率的である。

集計クエリ

集計クエリはログ分析やモニタリングダッシュボードで不可欠な機能である。QuickwitはElasticsearch互換の集計APIを提供し、オブザーバビリティワークフローで使われる主要な集計タイプをサポートする。

terms集計

terms集計はフィールドの値ごとにドキュメント数（またはメトリクス値）を集計し、上位N件を返す。エラーの発生元サービスや頻出ログレベルの分布を把握する際に使用する。

{
  "aggs": {
    "top_services": {
      "terms": {
        "field": "service",
        "size": 10
      }
    }
  }
}

histogramおよびdate_histogram集計

histogramは数値フィールドを指定した間隔でバケット化し、date_histogramは時刻フィールドをカレンダー単位（minute / hour / day など）でバケット化する。時系列ログのビジュアル化に直接対応する。

{
  "aggs": {
    "events_over_time": {
      "date_histogram": {
        "field": "timestamp",
        "calendar_interval": "1h",
        "time_zone": "Asia/Tokyo"
      },
      "aggs": {
        "error_count": {
          "filter": { "term": { "level": "ERROR" } }
        }
      }
    }
  }
}

サブ集計（nested aggregation）を使うことで、時間帯ごとのエラー件数など複合的な分析が一度のリクエストで実現できる。

メトリクス集計

数値フィールドに対する統計集計として、min、max、avg、sum、stats（上記すべてを同時に返す）が利用できる。レイテンシ分布やリソース使用量の集計に有効である。

{
  "aggs": {
    "latency_stats": {
      "stats": {
        "field": "response_time_ms"
      }
    }
  }
}

クエリプランナーの最適化

Quickwitのクエリプランナーはオブジェクトストレージ上のスプリット構造を前提とした独自の最適化を行う。この設計はオブジェクトストレージ上の検索における高コストなI/Oを最小化するために重要である。

図2: クエリプランナーによるスプリット剪定の例。timestampのrange条件を使ってメタデータだけで対象外スプリットを除外し、必要なスプリットのみを実際にダウンロードして検索する。

スプリット剪定

各スプリットはメタデータとして時刻範囲・タグなどを保持している。クエリプランナーはこのメタデータをメタストアから参照し、クエリのrange条件やタグフィルターと照合することで、実際にダウンロードが必要なスプリットの数を最小化する。これは特にオブジェクトストレージのリクエスト課金コストを直接削減する効果がある。

並列実行

選択されたスプリットは複数のサーチャーノードに分散され、並列に検索される。各サーチャーは担当スプリットのサブセットに対してローカル検索を実行し、結果を返す。クエリプランナーは最終的にすべての部分結果をマージして、クライアントへのレスポンスを構築する。

スクロールAPIとsearch_after

大量の検索結果を分割して取得するために、QuickwitはElasticsearch互換のページング機構を提供する。

スクロールAPI

スクロールAPIは検索コンテキストをサーバー側に保持し、複数回のリクエストで連続的にドキュメントを取得する仕組みである。大量データのエクスポートや全件スキャンに向いている。

POST /api/v1/my-index/_search?scroll=1m
{
  "query": { "match_all": {} },
  "size": 1000,
  "sort": [{ "timestamp": "asc" }]
}

レスポンスに含まれる_scroll_idを使って後続リクエストを送信する。スクロールコンテキストは指定した有効期間（上記では1分）が過ぎると自動的に破棄される。

search_after

search_afterはステートレスなページングを実現するパラメータである。スクロールAPIと異なりサーバー側でコンテキストを保持しないため、リソース消費が少なく長期間にわたるページングに適している。

{
  "query": { "match": { "level": "ERROR" } },
  "size": 100,
  "sort": [
    { "timestamp": "asc" },
    { "_id": "asc" }
  ],
  "search_after": ["2026-04-05T12:34:56Z", "doc-0000abc"]
}

前のページの最後のドキュメントのソート値をsearch_afterに渡すことで次のページを取得する。ソートキーには一意性を保証するための_idを含めることが推奨される。

QuickwitネイティブAPIの特徴

QuickwitはElasticsearch互換APIに加えて、Quickwit固有の機能を持つネイティブAPIも提供する。

ネイティブクエリ構文

ネイティブクエリ構文では、Lucene構文に近いシンプルなクエリ文字列でフィールド検索・範囲検索・ブール演算を記述できる。これはKibana Dev Tools等の外部依存なしに素早くクエリを試したい場面で有用である。

OpenTelemetryログ・トレースへの最適化

QuickwitはOpenTelemetryのログ・トレースデータモデルに最適化されたインデックス設定を提供する。resource_attributesやspan_attributesなどのネストしたJSONフィールドに対するクエリも効率的に処理できる。詳細はOpenTelemetry統合を参照。

取り込みとクエリの分離

Quickwitは取り込みの最終確認（コミット）を経てからスプリットがクエリ対象になるため、厳密な整合性モデルを持つ。Ingest APIには即時コミットモード（commit=force）と非同期コミットモードがあり、用途に応じてレイテンシとスループットのバランスを調整できる。詳細はデータ取り込みソースを参照。
