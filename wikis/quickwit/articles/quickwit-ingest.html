Quickwitのデータ取り込みソース

最終更新: 2026-04-05
QuickwitはKafka、Apache Pulsar、S3/ファイル、REST APIなど複数のデータソースからドキュメントを取り込む柔軟な取り込みパイプラインを提供します。各ソースにはそれぞれ固有の設定・最適化オプションがあり、ユースケースに応じて選択できます。

概要

Quickwitのデータ取り込みソースは、Quickwitがインデックスを構築する際にドキュメントを受け取るための入口である。Quickwitのインデクサーノードがこれらのソースからデータを読み取り、スプリット（Split）を生成してオブジェクトストレージへアップロードする。取り込みソースはインデックス設定ファイル（YAML）内の sources セクションで定義される。

各ソースはポーリングまたはプッシュ型のいずれかの方式でデータを受け取る。ストリーミングソース（Kafka・Pulsar）はオフセットを追跡しながら継続的にデータを取り込み、バッチソース（S3・ファイル）は指定されたパスを一括処理する。Ingest REST APIはクライアントからのプッシュを受け付けるリアルタイム取り込み手段である。

図1: Quickwitのデータ取り込みフロー。複数のソースからインデクサーがデータを受け取り、スプリットを生成してオブジェクトストレージに保存する。

Kafkaソース

Kafkaソースは、Apache Kafkaからのストリーミングデータ取り込みに特化したQuickwitの主要な取り込み手段である。大量のログやイベントデータをリアルタイムに近い形で処理するために設計されている。

パーティション管理

QuickwitのKafkaソースは、Kafkaトピックの各パーティションをインデクサーノードへ均等に割り当てる。パーティションの割り当てはQuickwitのコントロールプレーンが管理し、インデクサーノードの増減に応じてリバランスが自動的に行われる。各パーティションは独立したシャードとして処理されるため、スループットのスケールアウトが容易である。

オフセット追跡

Kafkaのコンシューマーオフセットはメタストアに保存される。これにより、インデクサーの再起動や障害後も取り込みの重複・欠落を防ぐことができる。Quickwitはat-least-onceの配送セマンティクスを採用しており、重複排除はユニークドキュメントIDを利用して実現する。

設定例（インデックスYAML）:

sources:
  - source_id: kafka-logs
    source_type: kafka
    params:
      topic: application-logs
      client_params:
        bootstrap.servers: "kafka:9092"
        group.id: "quickwit-indexer"

Apache Pulsarソース

Apache PulsarソースはPulsarブローカーへのサブスクリプションを通じてメッセージを取り込む。Pulsarの持つ地理的レプリケーション（Geo-Replication）機能やマルチテナント機能と組み合わせることで、分散環境での取り込みが可能になる。

Pulsarソースは Shared または Exclusive サブスクリプションタイプをサポートする。QuickwitはPulsarのメッセージIDをオフセットとして使用し、チェックポイントをメタストアに保持する。設定は source_type: pulsar と params ブロックでブローカーURL・トピック・サブスクリプション名を指定する。

S3/ファイルソース

S3ソースおよびファイルソースは、既存のファイルやオブジェクトストレージ内のデータをバッチで取り込む際に使用する。ログアーカイブのバックフィルや、一時的な大量データ投入に適している。

S3ソースの設定

S3ソースはバケット名とプレフィックス（またはグロブパターン）を指定する。QuickwitはS3オブジェクトのリストを取得し、各オブジェクトを順次ダウンロードして処理する。処理済みオブジェクトはメタストアに記録され、再処理を防ぐ。

サポートするファイル形式は ndjson（改行区切りJSON）が基本であり、gzip圧縮にも対応する。

ローカルファイルソース

ファイルソースはローカルパスまたは標準入力からデータを読み込む。CI/CDパイプラインやワンショットのデータ移行シナリオに適している。source_type: file と params.filepath でファイルパスを指定する。

Ingest REST API

Ingest APIは、HTTPクライアントから直接QuickwitへドキュメントをPOSTするプッシュ型の取り込み手段である。ストリーミングソースのセットアップが不要なシンプルな統合や、アプリケーションからの直接送信に適している。

エンドポイントと使用方法

Ingest APIの主なエンドポイントは POST /api/v1/{index_id}/ingest である。リクエストボディにはNDJSON形式のドキュメントを含める。1回のリクエストで複数ドキュメントをバルク送信できる。

Ingest APIには wal（Write-Ahead Log）バッファが存在し、受信したドキュメントはまずメモリバッファに格納される。バッファが一定量（バイト数またはドキュメント数）に達するか、コミット間隔タイムアウトが到来すると、スプリットとしてコミットされる。

取り込みレイテンシとコミット設定

Quickwitのインデックス取り込みは、ElasticsearchのNear-Real-Time（NRT）とは異なり、スプリット単位でコミットされるため、デフォルトで数秒〜数十秒のレイテンシが発生する。このトレードオフにより、高いスループットと低いオブジェクトストレージコストを実現している。

コミット間隔は indexing_settings.commit_timeout_secs で調整可能である。値を小さくするとレイテンシは下がるが、小さなスプリットが多数生成されるため、マージ処理（Merge）の負荷が増す。本番環境では60秒程度が推奨される。

エージェント連携

Quickwitは多様なログ転送エージェントと統合できる。これらのエージェントはIngest APIまたはElasticsearch互換のBulk APIを使用してデータを送信する。

Vector

Vectorは高性能なオブザーバビリティデータパイプラインツールである。Quickwitへの送信には elasticsearch シンクを使用し、エンドポイントにQuickwitのBulk APIアドレスを指定する。Vectorのトランスフォーム機能でフィールド変換・フィルタリングを行いながらQuickwitへ転送できる。

Fluent Bit

Fluent Bitは軽量なログプロセッサ・フォワーダーであり、Kubernetes環境でのDaemonSetとしての利用が一般的である。QuickwitへはElasticsearchアウトプットプラグインを設定し、Host にQuickwitのホスト、Index にインデックスIDを指定する。

Logstash

LogstashはElasticが開発するデータ収集・処理パイプラインツールである。QuickwitはElasticsearch互換APIを提供しているため、Logstashの elasticsearch アウトプットプラグインをそのまま利用できる。既存のLogstashパイプラインからの移行が容易である。

ドキュメントマッピングとスキーマ設定

Quickwitでは取り込むドキュメントのスキーマを ドキュメントマッピング（Document Mapping）として定義する。マッピングはインデックス設定の doc_mapping セクションで記述する。

フィールド型

主なフィールド型は以下の通りである:

  • text: 全文検索対象の文字列フィールド。トークナイザーを指定可能
  • keyword: 完全一致検索・集計に使用する文字列フィールド
  • i64 / u64 / f64: 数値フィールド。範囲クエリや集計に利用
  • datetime: タイムスタンプフィールド。ログのタイムライン検索に不可欠
  • ip: IPv4/IPv6アドレスフィールド。CIDR範囲クエリに対応
  • json: 動的なJSONオブジェクト。スキーマが不定のログに有用
  • bool: 真偽値フィールド

タイムスタンプフィールドの設定

timestamp_field はQuickwitが時刻ベースのプルーニング（Pruning）を行うための特別なフィールドである。このフィールドを正しく設定することで、時刻範囲クエリのパフォーマンスが大幅に向上する。ログ管理のユースケースでは必ず設定することが推奨される。

ドキュメントマッピングの設定例:

doc_mapping:
  timestamp_field: timestamp
  field_mappings:
    - name: timestamp
      type: datetime
      input_formats: [unix_timestamp, rfc3339]
      fast: true
    - name: message
      type: text
      tokenizer: default
    - name: level
      type: keyword
    - name: host
      type: keyword
    - name: extra
      type: json
