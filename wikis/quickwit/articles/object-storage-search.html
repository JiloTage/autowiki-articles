オブジェクトストレージ上の検索

最終更新: 2026-04-05
オブジェクトストレージ上の検索とは、Amazon S3・Google Cloud Storage・Azure Blob Storageなどのクラウドオブジェクトストレージを一次検索インデックスの保存先として活用するアーキテクチャであり、ストレージとコンピュートを分離することで大規模データに対するコスト効率の高い全文検索を実現する技術的アプローチである。

概要

オブジェクトストレージ上の検索（Object Storage-based Search）は、従来のローカルSSD依存型検索エンジンに代わり、クラウドオブジェクトストレージを検索インデックスの永続化レイヤーとして採用するアーキテクチャパターンである。Quickwitはこのアーキテクチャを中心に設計されており、S3などの安価で耐久性の高いストレージ上にイミュータブルなインデックスセグメントを蓄積することで、ペタバイト規模のデータに対して経済的な検索を実現する。

このアーキテクチャは、Snowflake・Databricks・Apache Icebergなどのデータレイク系プラットフォームが採用するストレージ・コンピュート分離モデルと思想を共有しており、クラウドネイティブな検索の主流アプローチとして急速に普及しつつある。

ストレージとコンピュートの分離

従来の検索エンジン（Elasticsearchなど）はインデックスをノードのローカルディスクに保存するため、ストレージとコンピュートが密結合している。これに対してオブジェクトストレージ上の検索では、両者を明確に分離する。

従来モデルの課題

ローカルSSD依存のアーキテクチャでは、次の課題が生じる:

  • 過剰プロビジョニング: ストレージ要件に合わせてコンピュートも過剰に確保する必要がある
  • スケーリングの非対称性: 書き込み負荷とクエリ負荷が時間帯によって大きく異なるが、どちらかに合わせると他方が非効率になる
  • ノード障害リスク: ローカルディスク故障時のデータ復旧コストが高い
  • レプリケーションコスト: 可用性確保のためにデータの複製が必要となり、実効ストレージコストが増大する

分離モデルの利点

ストレージとコンピュートを分離することで、次のメリットが得られる:

  • 独立スケーリング: 取り込み負荷増大時はインデクサーだけ、クエリ負荷増大時はサーチャーだけをスケールアウトできる
  • ゼロダウンタイム拡張: 新しい検索ノードはオブジェクトストレージに接続するだけで即時稼働する
  • サーバーレス親和性: クエリノードをスポットインスタンスやFaaS基盤で運用できる
図1: ストレージとコンピュートの分離。インデクサー（N台）とサーチャー（M台）は独立してスケールでき、どちらもオブジェクトストレージのスプリットを参照する。

イミュータブルインデックスセグメント（Split）

オブジェクトストレージ上の検索アーキテクチャの核心は、インデックスをイミュータブルなセグメント（Splitと呼ぶ）として管理する点にある。一度書き込まれたスプリットは変更されず、不要になれば削除される。

スプリットの内部構造

各スプリットは自己完結したアーカイブ（通常.splitまたは.zstd圧縮ファイル）であり、以下を含む:

  • 転置インデックス: 全文検索用の用語→文書IDマッピング（Apache Tantivy形式）
  • 列指向ストレージ: 集計・フィルタリング最適化されたカラムデータ（Arrow/Parquet類似形式）
  • メタデータ: タイムレンジ、フィールドスキーマ、統計情報（クエリのプルーニングに使用）

イミュータブル設計により、オブジェクトストレージの「一度書いたら変えない」という特性と完全に一致する。並行書き込みの競合が発生せず、ガベージコレクション（古いスプリットの削除）も単純なオブジェクト削除で済む。

スプリットのライフサイクル

スプリットは次のライフサイクルを経る: Staged（一時領域への書き込み） → Published（メタストアへの登録） → MarkedForDeletion（削除候補） → Deleted（オブジェクトストレージから削除）。マージ操作では複数の小さなスプリットが1つの大きなスプリットに統合され、検索効率が向上する。

キャッシュ戦略：ホットデータとコールドデータ

オブジェクトストレージはローカルSSDより読み取りレイテンシが高いため、効果的なキャッシュ戦略が性能の鍵を握る。

データ階層化（ホット / コールド）

検索エンジンのワークロードでは、直近のデータへのアクセスが大半を占める（いわゆるホットデータ）。古いデータ（コールドデータ）は低頻度でアクセスされる。この特性を活かし:

  • ホットデータ（直近数時間〜数日）: サーチャーのローカルSSDにキャッシュし、ほぼオンメモリに近い速度で応答
  • コールドデータ（数週間〜数年前）: オブジェクトストレージからオンデマンドで読み込み（レイテンシは高めだが頻度も低い）

キャッシュ退避とウォームアップ

LRU（Least Recently Used）ベースのキャッシュ退避により、サーチャーのディスク容量を超えたスプリットは自動的に削除される。クラスタ再起動後や新ノード追加時のキャッシュウォームアップは性能に影響するため、スプリットのプリフェッチやレプリカ配置の最適化が実装されている。

ParquetおよびArrow形式との関係

オブジェクトストレージ上の検索において、列指向フォーマットの選択は重要な設計決定である。

Apache Parquet

Apache ParquetはHive/Spark由来の列指向ストレージフォーマットで、オブジェクトストレージへの効率的なデータ保存が強みである。スノーフレーク型のデータウェアハウスやDatabricksのDelta Lakeなど、多くのデータレイク基盤がParquetを採用している。全文検索用の転置インデックスを含まないため、ログ検索よりもOLAP集計ワークロードに向いている。

Apache Arrow

Apache Arrowはメモリ上の列指向フォーマット（IPC形式）であり、異なるシステム間でのゼロコピーデータ交換を実現する。Quickwitの内部では、スプリット内の列指向ストレージにArrow類似の形式が採用されており、集計クエリを高速化する。Parquetファイルを直接クエリする機能も追加されつつあり、オブジェクトストレージ上のデータレイクとシームレスに統合できる。

転置インデックス vs. 純粋な列指向

Quickwit（およびApache Tantivyベースのスプリット）は転置インデックスと列指向ストレージを組み合わせており、Parquet単体では難しい全文検索を実現しつつ、集計にも対応する。これはElasticsearch / OpenSearchのアーキテクチャと同様の思想だが、永続化先がオブジェクトストレージである点が決定的な差異である。

類似アーキテクチャとの比較

オブジェクトストレージ上の検索は、他の著名なデータプラットフォームとも思想を共有する。

Snowflake / Databricks

Snowflakeはストレージとコンピュートを分離した先駆的なデータウェアハウスである。クエリエンジン（仮想ウェアハウス）は独立してスケールし、S3上のParquetファイルを読み込む。Databricks（Delta Lake）も同様のアーキテクチャを採用しており、ACID トランザクションをオブジェクトストレージ上で実現する。これらはOLAP集計に特化しており、全文検索は苦手である点でQuickwitと補完的な関係にある。

Apache Iceberg / Delta Lake

Apache IcebergやDelta Lakeはオブジェクトストレージ上のテーブルフォーマット仕様である。スナップショット管理、スキーマ進化、タイムトラベルクエリなど高度なデータ管理機能を提供する。Quickwitのスプリット管理モデルはこれらとコンセプトが近く、将来的にIcebergテーブルとの統合が検討される分野である。

Loki / Prometheus（参考対比）

Grafana Lokiもオブジェクトストレージをバックエンドとしたログアグリゲーションシステムだが、転置インデックスを持たずラベルベースの検索に限定される。全文検索が必要なユースケースではQuickwitが優位であり、オブザーバビリティスタックの選択においてトレードオフが生じる。

コストモデルの比較

オブジェクトストレージを採用することで、検索基盤のコスト構造は根本から変わる。

ストレージコスト

Amazon S3 Standardのストレージ単価はローカルSSDの約10〜20分の1（1 GB あたり月額 $0.023 vs $0.20〜$0.40）である。大量のログを長期保存する場合、このコスト差は数百万ドル規模に達することがある。さらにS3 Glacierなどのアーカイブクラスを組み合わせることで、コールドデータのコストをさらに削減できる。

コンピュートコスト

クエリが来ない夜間・週末などの低負荷時間帯に検索ノードをスケールダウン（または停止）できるため、コンピュートコストも削減可能である。従来のElasticsearchでは常時全ノードを起動しておく必要があったが、オブジェクトストレージ上の検索では「使った分だけ」の課金モデルに近づける。スポットインスタンスの活用も容易である。

ネットワーク・APIコスト

オブジェクトストレージの読み取りAPIコスト（S3 GET: $0.0004/1000リクエスト）と、クロスリージョン転送コストが追加発生する。特にスプリット読み込みが頻繁な場合にコストが積み上がるため、効果的なキャッシュはコスト最適化の観点からも重要である。検索ノードとオブジェクトストレージを同一リージョンに配置することで転送コストをほぼゼロにできる。

総保有コスト（TCO）の比較

Elasticsearchとの比較において、Quickwit公式のベンチマークでは同等のデータ量に対して総コストが1/10以下になるケースも報告されている。ただし、低レイテンシが要求されるリアルタイム検索ではElasticsearchが依然として優位な面もあり、ユースケースに応じた選択が求められる。
