OpenTelemetry統合

最終更新: 2026-04-05
QuickwitはCNCFプロジェクトであるOpenTelemetryとのネイティブ統合を提供し、OTLPプロトコルを通じてログ・トレース・メトリクスを直接受信・保存・検索できるオブザーバビリティバックエンドとして機能します。

概要

OpenTelemetry統合は、Quickwitのオブザーバビリティバックエンドとしての中核機能である。OpenTelemetry（OTel）はCloud Native Computing Foundation（CNCF）が管理するオープンスタンダードで、テレメトリデータ（ログ・トレース・メトリクス）の収集・変換・エクスポートを統一するフレームワークを提供する。

Quickwitはオブザーバビリティユースケースに特化した設計を持ち、OTLP（OpenTelemetry Protocol）をネイティブサポートすることで、OpenTelemetry CollectorやSDKからのデータを直接受け入れる。これにより、従来のElasticsearch/Jaegerスタックをより低コストのQuickwitで代替することが可能になる。

OTLPプロトコルとネイティブサポート

OTLPはOpenTelemetryが定義するテレメトリデータの転送プロトコルである。gRPCおよびHTTP/JSON形式をサポートし、Quickwitは両方のトランスポートに対応している。

OTLPエンドポイント

Quickwitが提供するOTLPエンドポイントは以下の通りである:

  • トレース（gRPC）: http://<quickwit-host>:7281/otlp/v1/traces
  • ログ（gRPC）: http://<quickwit-host>:7281/otlp/v1/logs
  • トレース（HTTP）: http://<quickwit-host>:7280/api/v1/otel-traces-v0_7/ingest
  • ログ（HTTP）: http://<quickwit-host>:7280/api/v1/otel-logs-v0_7/ingest

QuickwitはOTLPデータを受信すると、専用のビルトインインデックス（otel-traces-v0_7、otel-logs-v0_7）に自動的に格納する。これらのインデックスはOpenTelemetryのデータモデルに最適化されたスキーマを持ち、trace_id・span_id・service_nameなどの標準フィールドに対して効率的な検索が可能である。

OpenTelemetry Collectorの設定

OpenTelemetry CollectorをQuickwitと連携させる場合、Collectorの設定ファイルにQuickwitへのOTLPエクスポーターを追加する。以下は代表的な設定例である。

トレースの転送設定

以下のYAML設定は、OTLPレシーバーでトレースを受信し、QuickwitのgRPCエンドポイントへ転送する構成である:

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 5s
    send_batch_size: 1000

exporters:
  otlp/quickwit:
    endpoint: <quickwit-host>:7281
    tls:
      insecure: true

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/quickwit]
    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/quickwit]

ログのフィルタリングと変換

ログパイプラインではProcessorを使用して、不要なフィールドの削除や属性の追加・変換が可能である。filterプロセッサを使用することで特定のサービスやログレベルのみを転送するフィルタリングもできる。バッチプロセッサ（batch）はスループット向上のために推奨される設定であり、Quickwit側のインデックス効率を高める。

ログ・トレース・メトリクスの取り込みフロー

OpenTelemetryは三種類のテレメトリシグナルを定義している。Quickwitはそれぞれに対して異なる対応状況を持つ。

ログの取り込み

ログはOTLP経由でotel-logs-v0_7インデックスに格納される。このインデックスはOpenTelemetryログデータモデルに準拠したスキーマを持ち、以下のフィールドが標準的に保存される:

  • timestamp: ログ発生時刻（ナノ秒精度）
  • severity_text / severity_number: ログレベル（INFO, WARN, ERROR等）
  • body: ログメッセージ本文（全文検索対象）
  • service.name: サービス識別子（Resource Attributeより）
  • trace_id / span_id: ログとトレースの相関に使用
  • attributes: カスタム属性（JSON形式でフラット化）

トレースの取り込み

分散トレーシングデータはotel-traces-v0_7インデックスに格納される。スパン単位でドキュメント化され、trace_idによる検索で一つのトレースに属するすべてのスパンを取得できる。Quickwitのトレースインデックスはスパンの開始・終了時刻、parent_span_id、ステータス、タグなどを保存し、分散トレーシングの根本原因分析に活用できる。

メトリクスの現状

メトリクスシグナルについては、Quickwitは現時点でネイティブなOTLPメトリクス取り込みをサポートしていない。メトリクスはPrometheus等の専用ツールで管理し、Quickwitはログとトレースのバックエンドとして機能させるアーキテクチャが推奨される。Grafanaを活用することで、Quickwit（ログ・トレース）とPrometheus（メトリクス）を統合した統一オブザーバビリティダッシュボードを構築できる。

Jaeger互換APIとの関係

Quickwitは分散トレーシングのフロントエンドとしてJaeger UIをサポートするため、Jaeger互換のクエリAPIを実装している。これにより、既存のJaegerインフラを持つ組織はUIを変更せずにバックエンドをQuickwitに移行できる。

Jaeger APIエンドポイント

Quickwitが提供するJaeger互換エンドポイントは以下の通りである:

  • サービス一覧: GET /api/v1/jaeger/api/services
  • トレース検索: GET /api/v1/jaeger/api/traces?service=<name>
  • トレース詳細: GET /api/v1/jaeger/api/traces/<trace_id>

Jaeger UIの設定でQuickwitのAPIエンドポイントを指定することで、既存のJaegerダッシュボードをそのまま流用できる。OpenTelemetry CollectorにJaegerレシーバーを設定している場合、CollectorがデータをOTLP形式に変換してQuickwitへ転送するパターンも可能である。

GrafanaおよびZipkinとの統合パターン

Quickwitはエコシステムの主要ツールとの統合を提供しており、既存のオブザーバビリティスタックへのドロップイン置換として機能する。

Grafanaとの統合

Quickwit公式のGrafanaデータソースプラグインを使用することで、Grafanaダッシュボードからログ・トレースのクエリと可視化が可能になる。設定手順は以下の通りである:

  • GrafanaにQuickwitプラグインをインストールする（grafana-cli plugins install quickwit-quickwit-datasource）
  • Grafana UIからデータソースを追加し、QuickwitのURLを設定する
  • インデックス名（otel-logs-v0_7など）を指定してクエリを作成する

Grafana Explore機能を使用すれば、ログとトレースをリンクして相関分析ができる。trace_idフィールドを活用したログ→トレースのジャンプ機能により、障害発生時の根本原因分析が効率化される。

Zipkinとの統合

Zipkin形式のトレースデータはOpenTelemetry CollectorのZipkinレシーバーを経由してQuickwitに取り込むことができる。CollectorがZipkin形式からOTLP形式への変換を担うため、Quickwit側の設定変更は不要である。Zipkin UIを引き続き使用したい場合は、CollectorがQuickwitとZipkinバックエンドの両方にエクスポートするファンアウト構成が可能である。

SDKでのインストルメンテーション

アプリケーションにOpenTelemetry SDKを組み込むことで、トレース・ログをQuickwitに直接送信できる。OpenTelemetry SDKはJava・Python・Go・Node.js・.NETなど主要言語に対応している。

SDKエクスポーター設定

SDKの設定でOTLPエクスポーターのエンドポイントをQuickwit（またはOpenTelemetry Collector）に向けることでテレメトリデータが送信される。環境変数による設定が最も汎用的な方法である:

# OTLPエクスポーターエンドポイント（Collectorを経由する場合）
OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4318

# または直接Quickwitに送信する場合
OTEL_EXPORTER_OTLP_ENDPOINT=http://quickwit-host:7280

# サービス名の設定
OTEL_SERVICE_NAME=my-application
OTEL_RESOURCE_ATTRIBUTES=deployment.environment=production

自動インストルメンテーション

多くのOpenTelemetry SDKは自動インストルメンテーション機能を提供しており、コード変更なしにHTTPリクエスト・データベースクエリ・メッセージングなどのトレースを自動収集できる。JavaエージェントやPythonのopentelemetry-instrumentコマンドを使用することで、既存アプリケーションへの最小限の変更でQuickwitへのテレメトリ送信を開始できる。より詳細なデータ収集には、データ取り込みソースの設定と組み合わせることが推奨される。
