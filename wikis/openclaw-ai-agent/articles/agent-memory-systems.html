エージェントメモリシステム

最終更新: 2026-04-08
エージェントメモリシステムとは、AIエージェントが情報を保持・検索・更新するための仕組みの総称であり、短期的なコンテキスト管理から長期的な知識ストアまでを包含する。OpenClawはインコンテキスト記憶・外部ベクトルDB・構造化キャッシュを組み合わせたハイブリッドメモリアーキテクチャを採用し、複雑なタスクにおける継続的な推論を可能にする。

メモリの種類

AIエージェントが扱うメモリは大きく3つのカテゴリに分類される。インコンテキスト記憶（In-Context Memory）はモデルの入力ウィンドウに直接含まれる情報であり、最も即座にアクセスできる一方、容量はコンテキストウィンドウサイズによって制約される。外部記憶（External Memory）はデータベースやファイルシステムなど外部ストアに永続化された情報であり、容量上限がなく長期保存に適している。キャッシュ記憶（Cached Memory）は計算コストの高い中間表現をあらかじめ保存しておく仕組みで、応答速度の向上に貢献する。

これら3種類のメモリは相互補完的に機能する。エージェントアーキテクチャの観点では、タスクの性質やリソース制約に応じて適切なメモリ層を選択・組み合わせることが性能と信頼性の鍵となる。

短期記憶とコンテキストウィンドウ管理

短期記憶の実体はLLMのコンテキストウィンドウである。エージェントはユーザーとのメッセージ履歴、ツール呼び出し結果、思考ステップなどをすべてコンテキストとして保持しながら推論を進める。コンテキストウィンドウには上限があるため、長時間・多ステップのタスクでは管理戦略が不可欠となる。

代表的なコンテキスト管理手法には以下がある。

  • スライディングウィンドウ: 古いメッセージを順次削除し、直近の会話のみを保持する。
  • 要約圧縮: 古い会話履歴をLLM自身に要約させ、要約テキストとしてコンテキストに組み込む。
  • 重要度フィルタリング: タスク目標に関連性の低いメッセージを優先的に削除する。
  • 外部オフロード: 詳細情報を外部記憶に書き出し、必要時にのみ取り込む。

OpenClawでは、セッションごとにコンテキスト使用率をモニタリングし、閾値（デフォルト80%）を超えた時点で自動的に要約圧縮を実行するメカニズムを持つ。

長期記憶とベクトルデータベース（RAG）

長期記憶は外部ストアに永続化された知識であり、セッションをまたいで参照可能な情報を提供する。実装の中心となるのがベクトルデータベースとRAG（Retrieval-Augmented Generation）パターンである。

RAGでは、保存したいテキストをエンベディングモデルで数値ベクトルに変換し、ベクトルDBに格納する。エージェントが情報を必要とした際、クエリを同様にベクトル化して類似度検索を行い、上位k件のチャンクを取得してコンテキストに注入する。この仕組みにより、数百万件を超えるドキュメントから関連情報を数十ミリ秒で検索することが可能となる。

OpenClawが対応する主なベクトルDBバックエンドは以下の通りである。

  • ローカルファイルベース（デフォルト）: 小規模タスク向け。追加依存なし。
  • Chroma: 中規模・プロトタイプ向けの軽量OSSベクトルDB。
  • Qdrant: 高スループット・大規模本番環境向け。フィルタリング機能が豊富。
  • pgvector: PostgreSQL拡張。既存RDBとの統合が容易。

マルチエージェントシステムでは、複数のエージェントが同一ベクトルDBを共有し、互いの知識を活用する構成が一般的に取られる。

エピソード記憶とセマンティック記憶

認知科学に由来する区分として、エージェントのメモリはエピソード記憶とセマンティック記憶に大別される。

エピソード記憶は「いつ・何をしたか」という具体的な出来事の記録である。エージェントにとっては過去のタスク実行ログ、ツール呼び出し履歴、エラーと回復のパターンなどが該当する。エピソード記憶を参照することで、エージェントは同種のタスクで以前うまくいった戦略を再利用したり、失敗パターンを避けたりできる。

セマンティック記憶は「何が真実か」という概念・事実・関係性の知識体系である。ドメイン知識ドキュメント、ツール仕様書、ユーザー設定などが含まれる。セマンティック記憶は比較的安定しており、更新頻度は低い。

OpenClawはこの区分に対応する2つのストアを持ち、それぞれ異なる検索インデックスと有効期限ポリシーを適用している。エピソード記憶はTTL（Time-To-Live）による自動削除が有効であり、古いエピソードがストアを圧迫しない設計となっている。

OpenClawのメモリ実装

OpenClawのメモリシステムはMemoryManagerクラスを中心に構成される。このクラスはインコンテキスト・キャッシュ・外部という3層のメモリを抽象化し、エージェントコアに統一的なAPIを提供する。エージェントアーキテクチャにおける他のコンポーネント（プランナー、ツール実行器など）はすべてこのAPIを通じてメモリにアクセスする。

主要なインターフェースを以下に示す。

  • memory.store(key, value, tier="external"): 指定した層に情報を保存する。
  • memory.retrieve(query, top_k=5): セマンティック検索で関連情報を取得する。
  • memory.forget(key): 指定エントリを削除する。
  • memory.summarize_context(): 現在のコンテキストを圧縮して短期記憶の使用量を削減する。
  • memory.snapshot(): 現在のメモリ状態をシリアライズし、セッションを再開可能にする。

また、OpenClawはメモリ書き込みツールをエージェント自身に公開している。エージェントはタスク遂行中に重要な発見や中間結果を自発的に外部メモリに書き込むことができ、これにより長時間タスクでの情報損失が防止される。AIエージェント概論でも述べられているように、メモリへの自律的な書き込みはエージェントの自己組織化能力の基盤となる。

メモリ管理の課題と最適化

エージェントメモリシステムには複数の技術的課題が存在する。

ハルシネーションによる誤情報の蓄積: エージェントが誤った情報を外部メモリに書き込むと、以降のセッションでその誤情報が参照され続ける危険がある。OpenClawはメモリ書き込み時に信頼度スコアを記録し、低スコアのエントリは検索結果の優先度を下げる仕組みを採用している。

メモリの整合性維持: 長期運用の中でメモリ内の情報が陳腐化したり矛盾したりする問題がある。定期的なガベージコレクションと矛盾検出パイプラインによって整合性を保つ設計が求められる。

プライバシーと情報漏洩: 複数ユーザーが同一エージェントを利用する環境では、ユーザーAの情報がユーザーBに漏洩しないよう、テナント分離とアクセス制御が必須となる。

検索レイテンシの最適化: 大規模なベクトルDBでは検索レイテンシが問題になる場合がある。階層型インデックス（HNSW）や近似最近傍探索（ANN）を活用し、精度とレイテンシのバランスを調整する。

OpenClawはこれらの課題に対して設定可能なポリシーを提供しており、タスクのリスクレベルや性能要件に応じてメモリ戦略をカスタマイズできる。詳細はエージェントアーキテクチャを参照。
