ツール使用アーキテクチャ

最終更新: 2026-04-08
ツール使用アーキテクチャは、AIエージェントが外部関数やAPIを呼び出す仕組みを定義する設計パターンであり、JSON Schemaによるツール定義、LLMによるツール選択、ツール実行サイクル、エラーハンドリングを組み合わせて自律的なタスク遂行能力を実現する。

ツール使用の概念と歴史

AIエージェントにおける「ツール使用」とは、大規模言語モデル（LLM）が自然言語の推論能力を超えて、プログラム的な外部機能（ファイル操作、Web検索、データベースクエリ、APIコールなど）を呼び出す能力を指す。 これは、LLM単体では実現できなかったリアルタイム情報取得・副作用を伴うアクション・精密な計算処理を可能にする。

初期のLLMは会話生成に特化しており、外部システムとの連携は手動のプロンプトエンジニアリングに依存していた。 2023年以降、OpenAIのFunction Callingの公開をきっかけに、LLMが構造化されたツール呼び出しを出力するパラダイムが標準化された。 その後、AnthropicのTool Use API、GoogleのFunction Callingなど各社が対応し、現在では主要なLLMプロバイダーの共通機能となっている。

OpenClaw は、このツール使用パラダイムを中心に設計された AIエージェントフレームワークであり、 型安全なツールレジストリと動的なツール選択メカニズムを特徴とする。

JSON Schemaによるツール定義

ツール使用の基盤となるのが、JSON Schema形式によるツール定義である。 各ツールは以下の要素を持つスキーマとして記述される：

  • name: ツールの一意な識別子（例: search_web、read_file）
  • description: ツールの機能説明（LLMがツールを選択する際の参照情報）
  • parameters: 入力パラメータのJSON Schemaオブジェクト（型・必須フィールド・説明を含む）
  • returns（省略可）: 出力形式の説明

以下はファイル読み取りツールのスキーマ例である： 

{
  "name": "read_file",
  "description": "指定したパスのファイルを読み込み、内容を返す",
  "parameters": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "読み込むファイルの絶対パス"
      },
      "encoding": {
        "type": "string",
        "enum": ["utf-8", "utf-16", "binary"],
        "default": "utf-8"
      }
    },
    "required": ["path"]
  }
}

LLMはこのスキーマ情報をシステムプロンプトまたは専用のAPI引数として受け取り、 ユーザーリクエストの文脈に応じて適切なツールと引数を選択する。 スキーマの記述精度が高いほどツール選択の精度が向上するため、 descriptionフィールドの品質管理がエージェント性能に直結する。

ツール選択メカニズム

LLMがどのツールを呼び出すかを決定するメカニズムは、 モデルのトレーニングとプロンプト設計の両方に依存する。 主要な選択戦略を以下に示す。

強制選択と自動選択

  • auto（自動）: LLMがタスクの文脈に基づいて、ツールを呼び出すか直接応答するかを自律的に判断する。最も一般的な設定。
  • any（必須）: 少なくとも1つのツールを呼び出すことを強制する。検索や計算が必ず必要なユースケースに適用。
  • tool（指定）: 特定のツールの呼び出しを強制する。構造化データ抽出など、出力フォーマットを統一したい場合に有効。
  • none（禁止）: ツール呼び出しを禁止し、テキスト応答のみを許可する。

並列ツール呼び出し

多くのモデルは依存関係のない複数のツールを並列で呼び出す能力を持つ。 例えば「A市とB市の天気を比較して」という指示に対し、 二つの天気取得ツールを同時に呼び出すことで応答時間を短縮できる。 OpenClawのエグゼキュータはこの並列呼び出しをネイティブサポートしており、 Promise.allベースの非同期実行プールを採用している。

エージェントアーキテクチャの観点からは、 ツール選択の精度はシステムプロンプトにおけるツール説明の品質と、 ツール数の多寡に大きく影響される。 ツールが50個を超える場合は動的ツールフィルタリング（タスク文脈に応じたサブセット提供）が推奨される。

ツール実行サイクルとエラーハンドリング

ツール実行サイクルはエージェントランタイムの中核処理であり、以下のステップで構成される：

  1. パース: LLMのtool_useブロックからツール名と引数を抽出する
  2. バリデーション: JSON Schemaに対する引数の型・必須チェック
  3. ディスパッチ: ツールレジストリから対応する実装関数をルックアップして呼び出す
  4. タイムアウト制御: 設定した最大実行時間を超えた場合はキャンセルしエラーを返す
  5. 結果シリアライズ: 実行結果をJSON化してtool_resultとして会話履歴に追記する

エラーハンドリングとフォールバック

ツール実行では様々なエラーが発生しうる。OpenClawは以下の4層エラーハンドリング戦略を採用している：

  • バリデーションエラー: スキーマ不適合の引数を受け取った場合、実行前にエラーをtool_resultとして返し、LLMに再試行を促す。
  • 実行エラー: ツール内部での例外（ネットワーク障害・権限エラーなど）はキャッチされ、構造化エラーメッセージとしてLLMに通知される。
  • タイムアウトフォールバック: タイムアウト時は部分結果があればそれを返し、なければデグレード応答（「情報を取得できませんでした」など）を生成する。
  • リトライポリシー: 一時的な障害（レート制限・5xxエラー）に対しては指数バックオフリトライを適用する。最大リトライ回数はツール定義に設定可能。

特に重要なのが「エラーの透過性」の原則である。ツール実行の失敗をランタイムが隠蔽するのではなく、 LLMに詳細なエラー情報を伝えることで、モデル自身が代替戦略（別ツールの使用・ユーザーへの確認など）を選択できるようにする。

OpenClawのツールレジストリ実装

OpenClawのツールレジストリは、ツールの登録・管理・提供を担う中央コンポーネントである。 MCPプロトコルとの統合により、 外部MCPサーバーが提供するツールも同一インターフェースで扱える設計となっている。

レジストリの主要機能

  • 型安全なツール登録: TypeScriptのジェネリクスを活用し、入力スキーマと実装関数の型整合性をコンパイル時に保証する。
  • 動的ロード: プラグイン形式でツールモジュールを実行時に追加・削除できる。 エージェントを停止せずにツールセットを更新可能。
  • スコープ管理: ツールにスコープ（read-only、write、dangerousなど）を付与し、 エージェントの権限設定に応じてフィルタリングする。
  • メトリクス収集: 各ツールの呼び出し回数・平均実行時間・エラー率を自動収集し、 パフォーマンスモニタリングに活用する。
  • LLM向けスキーマ生成: 登録されたツール群から、各LLMプロバイダーのAPI形式に適したスキーマを動的に生成する（Anthropic形式・OpenAI形式など）。

ツールレジストリのアーキテクチャは、OpenClawが採用する エージェントアーキテクチャの拡張性を担保する根幹コンポーネントであり、 ユーザーが独自のドメイン固有ツールを追加することで、 任意のワークフローに特化したエージェントを構築できる。
