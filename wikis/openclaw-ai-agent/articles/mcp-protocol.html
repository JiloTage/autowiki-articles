Model Context Protocol（MCP）

最終更新: 2026-04-08
Model Context Protocol（MCP）は、Anthropicが2024年11月に発表したオープン標準プロトコルで、AIエージェントと外部ツール・データソースをホスト/クライアント/サーバーの三層アーキテクチャで接続し、LLMアプリケーションのコンテキスト管理とツール統合を標準化する。

概要と目的

Model Context Protocol（MCP）は、OpenClaw AIエージェントをはじめとするAIエージェントフレームワークと外部ツール・データソースを接続するための標準化されたインタフェース仕様である。Anthropicが2024年11月に公開し、オープンソースとして提供された。

それ以前のLLMアプリケーションは、各ツールやサービスとの統合を独自のアダプタやAPIラッパーで実装しており、ツールが増えるほど統合コストが指数的に増大するという課題があった。MCPはこの問題を「USB-Cポート」に例えられる統一インタフェースで解決することを目指す。開発者がMCP準拠のサーバーを一度実装すれば、MCP対応の任意のホスト（エージェントや統合開発環境など）から利用できるようになる。

MCPの主な目的は次の三点に集約される。第一に、LLMとコンテキストプロバイダー間の通信プロトコルの標準化。第二に、ツール・リソース・プロンプトの発見（ディスカバリー）と呼び出しの均一化。第三に、セキュリティモデルとアクセス制御の一元化である。

MCPアーキテクチャ

MCPは「ホスト」「クライアント」「サーバー」の三層から構成される。以下の図はその関係を示す。

ホスト (Host)

ホストはLLMと対話するアプリケーション本体を指す。Claude DesktopやOpenClawのようなエージェントフレームワーク、あるいは統合開発環境（IDE）がホストに相当する。ホストはユーザーからの入力を受け取り、LLMに推論を依頼しながら、内部のMCPクライアントを通じてサーバーとやり取りする。

クライアント (Client)

MCPクライアントはホスト内部に組み込まれ、MCPサーバーとの一対一接続を管理するプロトコル仲介コンポーネントである。クライアントはサーバーとの能力ネゴシエーション（ハンドシェイク）を行い、利用可能なリソース・ツール・プロンプトを発見する。LLMがツール呼び出しを要求した際、クライアントがリクエストを適切なサーバーに中継し、応答をLLMに返す。

サーバー (Server)

MCPサーバーは実際の機能（ファイル読み書き、データベースクエリ、外部API呼び出しなど）を提供する軽量プロセスである。サーバーはリソース・ツール・プロンプトの三種類のプリミティブを公開し、クライアントからの要求に応じて処理を実行して結果を返す。サーバーとクライアントの通信はstdio（ローカルプロセス間通信）またはHTTP+SSE（リモート接続）で行われる。

主要コンポーネント

MCPサーバーが公開できるプリミティブには三種類ある。これらを組み合わせることで、LLMは外部環境を読み取り、操作し、一貫したプロンプトで誘導されることが可能になる。

Resources（リソース）

Resourcesは、LLMのコンテキストウィンドウに読み込むことができるデータや情報を表す。ファイル内容、データベースのレコード、ウェブページのスナップショット、アプリケーションのログなどが該当する。リソースはURIで識別され、テキストまたはバイナリ形式で提供される。LLMはリソースを「読む」ことで最新の外部情報をコンテキストに取り込める。

Tools（ツール）

Toolsは、LLMが副作用を伴う操作を実行するためのインタフェースである。ツールユース（Tool Use）アーキテクチャに対応し、ファイルの書き込み、APIへのPOSTリクエスト、コードの実行、データベースへの書き込みなどが含まれる。各ツールはJSON Schemaで入力仕様が定義されており、LLMはスキーマを参照して適切な引数を生成できる。

Prompts（プロンプト）

Promptsはサーバーが定義する再利用可能なプロンプトテンプレートである。ユーザーや開発者が特定のタスクに最適化されたプロンプトをサーバー側で管理し、クライアントに提供できる。これにより、複数のアプリケーション間でプロンプトエンジニアリングの成果を共有・再利用することが容易になる。

MCPサーバーの実装例

MCPサーバーはPython・TypeScript・Rust等の言語で実装できる。以下はPython SDKを使ったファイル読み取りツールの概念的な例である。 

from mcp.server import Server
from mcp.types import Tool, TextContent

server = Server("file-server")

@server.list_tools()
async def list_tools():
    return [
        Tool(
            name="read_file",
            description="ファイルの内容を読み取る",
            inputSchema={
                "type": "object",
                "properties": {
                    "path": {"type": "string", "description": "ファイルパス"}
                },
                "required": ["path"]
            }
        )
    ]

@server.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "read_file":
        with open(arguments["path"]) as f:
            content = f.read()
        return [TextContent(type="text", text=content)]

# stdio トランスポートで起動
server.run_stdio()

公式SDKの他に、コミュニティが提供するMCPサーバーも多数存在し、GitHub・Slack・PostgreSQL・Puppeteer（ウェブ操作）など主要なサービス向けのサーバーが公開されている。

MCPのユースケース

MCPは幅広いユースケースに適用できる。代表的なシナリオを以下に示す。

  • 開発支援エージェント: IDEに組み込まれたエージェントがMCP経由でコードベース・ターミナル・デバッガを操作し、自律的にバグを修正する。
  • データ分析: LLMがMCP越しにデータベースをクエリし、結果を解釈してレポートを生成する。
  • マルチエージェント協調: マルチエージェントシステムにおいて、各エージェントがMCPサーバー経由で共有リソースにアクセスし、情報を一貫して参照・更新する。
  • ナレッジ管理: 社内文書をリソースとして公開するMCPサーバーを構築し、複数のエージェントが最新情報を参照できる環境を整備する。

OpenClawとMCPの統合

OpenClawはMCPをネイティブサポートし、設定ファイルにMCPサーバーのエンドポイントを列挙するだけで複数のサーバーを同時に利用できる。OpenClawのエージェントループ（Think-Act-Observe）において、MCPツールは組み込みツールと同等の扱いを受け、ツールユースアーキテクチャの標準フローで呼び出される。

OpenClawとMCPの統合で特筆すべき点として、コンテキスト継続性がある。MCPのリソースはエージェントの作業メモリに統合され、長期にわたるタスクでも外部状態を常に参照可能にする。また、マルチエージェント構成では、親エージェントと子エージェントが同一のMCPサーバーを共有することで、分散した作業の結果を一貫した形で集約できる。

なお、LangChainもMCPサポートを追加しており、LangChainベースのアプリケーションとOpenClawとの間で同一のMCPサーバーを共用することが可能である。これにより、エコシステム横断での再利用性が向上している。

他のツール統合規格との比較

MCPの位置付けを理解するために、既存の統合手法と比較する。

規格・手法 標準化 双方向性 コンテキスト管理 主な用途
MCP オープン標準 あり（双方向） リソース+ツール+プロンプト 汎用AIエージェント統合
OpenAPI / REST 業界標準 なし（リクエスト/レスポンス） なし ウェブサービスAPI
Function Calling（OpenAI） プロプライエタリ 一方向（LLM→ツール） 限定的 OpenAIモデル専用統合
LangChain Tool フレームワーク固有 一方向 チェーン内のみ LangChainエコシステム
Plugins（旧OpenAI） 廃止済み なし なし ChatGPTプラグイン

MCPの最大の強みは、リソース・ツール・プロンプトを一つのプロトコルで統合し、サーバーとクライアントの両方向で情報を流せる双方向性にある。また、特定のLLMやフレームワークに依存しないオープン標準として設計されているため、LangChainやOpenClawなど複数のフレームワークが採用している。
