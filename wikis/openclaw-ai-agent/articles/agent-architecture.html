エージェントアーキテクチャ設計パターン

最終更新: 2026-04-08
エージェントアーキテクチャ設計パターンとは、AIエージェントの推論・計画・行動・自己修正の各フェーズを構造化するための再利用可能な設計手法の体系である。OpenClawではこれらのパターンを組み合わせ、高い信頼性と拡張性を持つエージェントシステムを構築している。

概要

AIエージェントは単なる言語モデルの呼び出しにとどまらず、外部ツールとのインタラクション、複数ステップにわたる計画立案、記憶の管理、そして自己の行動に対する反省といった複雑なオーケストレーションを必要とする。 これらの要求に応えるため、研究コミュニティと産業界では複数の設計パターンが提案・実装されてきた。

OpenClaw フレームワークは、以下に示す主要パターンを統合的に採用しており、ユースケースに応じて柔軟なパターン選択・組み合わせを可能にする。 詳細なエージェントの概念については AIエージェント概論 を参照されたい。

ReAct（推論＋行動）パターン

ReAct（Reasoning + Acting） パターンは、Yao et al.（2022）が提案した手法であり、言語モデルが「思考（Thought）→ 行動（Action）→ 観察（Observation）」のサイクルを繰り返すことで複雑なタスクを解決する。

従来のチェーン・オブ・ソートが推論のみに留まるのに対し、ReActは推論と外部ツール呼び出しを交互に行う点が特徴である。これにより、エージェントは計算、情報検索、コード実行といった外部能力を推論プロセスに統合できる。 ツール使用アーキテクチャとの親和性が特に高いパターンである。

ReActのフロー

OpenClawでのReAct実装

OpenClawは各ターンの出力を構造化フォーマット（thought・action・action_inputフィールド）でパースし、 ツールディスパッチャに渡す。観察結果はプロンプト履歴にフィードバックされ、次の推論ステップへ入力される。 短期・長期の記憶統合については エージェント記憶システム を参照。

Plan-and-Execute（計画実行）パターン

Plan-and-Execute パターンは、タスクを「計画フェーズ」と「実行フェーズ」に明確に分離する。計画フェーズでは上位モデルが全体のサブタスクリストを生成し、実行フェーズでは各サブタスクを順次（または並列に）実行するエグゼキュータが担当する。

ReActが一ステップずつ動的に判断するのに対し、Plan-and-Executeは計画を先行して立案するため、長期的・多ステップタスクへの適性が高い。ただし計画の修正には再計画フェーズが必要になる場合がある。

マルチエージェントシステム では、このパターンを拡張して複数のエグゼキュータエージェントが並列にサブタスクを処理する構成が一般的である。

Reflexion（自己反省）パターン

Reflexion（Shinn et al., 2023）は、エージェントが過去の行動と結果を「反省（reflection）」し、その洞察を言語的フィードバックとしてエピソード記憶に格納する手法である。 次のエピソードではその反省が参照されるため、同じ失敗を繰り返さない適応的な行動が可能になる。

反省サイクル

  1. タスク実行: エージェントが通常のReActループで行動する
  2. 評価: 評価モジュールが成功・失敗・部分的成功を判定する
  3. 反省生成: 評価結果を受けてLLMが反省テキストを生成する
  4. 記憶格納: 反省テキストをエピソード記憶（長期バッファ）に保存する
  5. 次エピソードへ注入: 保存された反省を次回のシステムプロンプトに挿入する

OpenClawのReflexion実装は エージェント記憶システム のエピソード記憶層と密接に統合されており、反省内容はJSONエントリとして管理されて自動的にコンテキストウィンドウに注入される。

MRKL（Modular Reasoning, Knowledge and Language）

MRKL（Modular Reasoning, Knowledge and Language） システム（Karpas et al., 2022）は、LLMをルーターとして複数の専門モジュール（計算機、データベース、APIなど）に問い合わせを振り分けるアーキテクチャである。

各モジュールは特定のドメインに特化しており、LLMは自身のパラメトリック知識とモジュールからの離散的な知識（Neuro-Symbolic な情報）を組み合わせて回答を生成する。 これはReActの「ツール呼び出し」概念の原型ともいえる。

OpenClawでのパターン実装

OpenClaw は上記パターンを単純に採用するのではなく、各パターンの強みを組み合わせたハイブリッドアーキテクチャを採用している。

ハイブリッドアーキテクチャの構成

レイヤー 採用パターン 役割
戦略層 Plan-and-Execute 長期タスクの分解と優先度付け
実行層 ReAct + MRKL ツール選択と逐次的な推論・行動
改善層 Reflexion 失敗からの学習と反省の蓄積

実行層ではReActのダイナミックな推論能力とMRKLのモジュール化設計を統合し、ツール呼び出しを型安全なスキーマ（JSON Schema）で記述することで信頼性を高めている。 詳しくは ツール使用アーキテクチャ を参照。

パターンの切り替えメカニズム

OpenClawではタスクの複雑度スコアを事前に推定し、スコアに応じてパターンを動的に選択する：

  • 低複雑度（スコア < 0.4）: シンプルなReActループ
  • 中複雑度（0.4 ≤ スコア < 0.75）: ReAct + MRKL
  • 高複雑度（スコア ≥ 0.75）: Plan-and-Execute + ReAct + Reflexion

パターン選択の指針

実際のシステム設計においてどのパターンを選択するかは、以下の軸で判断することが推奨される。

選択マトリクス

判断軸 ReAct Plan-and-Execute Reflexion MRKL
短期タスク ◎ △ ○ ○
長期・複雑タスク △ ◎ ○ △
失敗からの学習 △ △ ◎ △
多様なツール統合 ○ ○ △ ◎
実装の容易さ ◎ ○ △ ○

一般的な推奨事項

  • プロトタイプ段階では ReAct から始め、要件に応じて拡張する
  • 本番環境で信頼性が求められる場合は Plan-and-Execute を検討する
  • 反復的な改善や自律学習が必要な場合は Reflexion を追加する
  • ドメイン特化モジュールが多数存在する場合は MRKL ベースのルーティング層を設ける
  • マルチエージェントシステム では、Plan-and-ExecuteとMRKLを組み合わせ、各エージェントに特化した役割を割り当てることが効果的である
