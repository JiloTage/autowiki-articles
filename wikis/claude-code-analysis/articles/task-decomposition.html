タスク分解戦略

最終更新: 2026-03-31
コーディネータモードにおけるタスク分解戦略は、複雑な作業を独立した並行実行可能な単位に分割するための体系的手法である。自己完結型プロンプトの設計・タスク依存グラフの構築・発見フェーズの運用・結果統合の方法論を包括し、マルチエージェント処理の効率を最大化する。

タスク分解の原則

タスク分解戦略の基盤となる3つの原則は、独立性・並行性・依存関係管理である。コーディネータモードはこれらの原則に従ってタスクをワーカーエージェントに割り当てる。

独立性の原則

各タスクは他のタスクの完了を待たずに実行できる形に設計する。タスクが共有状態を変更する場合、そのタスクは独立していないと判断し、順次実行スケジュールに移動させる。独立性を確保するためには以下の点を避ける必要がある。

  • 同一ファイルへの同時書き込み
  • 互いの出力に依存した処理
  • 共有リソースへの競合アクセス

並行性の最大化

「並列性はスーパーパワー」という設計哲学のもと、独立したタスクは常に並行起動する。ワーカーは非同期で動作するため、コーディネータは複数のワーカーを同時に起動し、それぞれが自律的に処理を進めることができる。

依存関係管理

タスク間に依存関係がある場合、先行タスクの完了後に後続タスクを起動するよう制御フローを設計する。依存関係は有向非巡回グラフ（DAG）として表現し、トポロジカル順序に従って実行する。

自己完結型プロンプトの設計

ワーカーエージェントは親コーディネータのコンテキストにアクセスできないため、割り当てプロンプトには実行に必要な情報をすべて含める必要がある。この自己完結性はコーディネータモードの最重要原則の一つである。

含めるべき情報

  • タスクの目的と完了条件: 何を達成すべきかを明確に記述する
  • 対象ファイルやリソースのパス: 相対パスではなく絶対パスを使用する
  • 前提となるコンテキスト: 発見フェーズで収集した仕様・制約・既存コード情報
  • 期待する出力形式: XML フォーマットの要件を明記する
  • 使用可能なツール: ワーカーが利用できるツールの範囲を示す

プロンプトのアンチパターン

「前の指示を参照してください」「先ほど説明したように」などの前のコンテキストへの参照は、自己完結型プロンプトでは機能しない。各ワーカーはゼロから情報を読み込むため、必要な情報は常にプロンプト内に再掲する。

タスク依存グラフの構築

複雑なタスクを処理する前に、コーディネータはタスク間の依存関係を有向非巡回グラフ（DAG）として明示化する。このグラフに基づいて並行実行できるタスクと順次実行すべきタスクを判別する。

グラフ構築のステップ

  1. 全タスクをリストアップし、それぞれの入力と出力を定義する
  2. あるタスクの出力が他タスクの入力になっている場合、依存エッジを引く
  3. 入次数0のタスク（依存なし）を並行実行候補とする
  4. 依存タスクが完了次第、後続タスクをキューに追加する

コーディネータの発見フェーズ

実際のワーカー起動前に、コーディネータは発見フェーズを実施する。このフェーズではコーディネータ自身がツールを使用して必要な情報を収集し、自己完結型プロンプトの作成に必要な仕様を統合する。

発見フェーズで行うこと

  • コードベース調査: 関連ファイルの構造・API・インターフェースを把握する
  • 依存関係の確認: 既存のモジュール間依存関係を調べてタスク分割の境界を決める
  • 制約の特定: テスト要件・スタイルガイド・命名規則などの制約を収集する
  • 仕様の統合: 収集した情報を各ワーカーへのプロンプトに組み込める形に整理する

発見フェーズとワーカー起動の境界

発見フェーズはコーディネータが単独で実行する。調査が完了し十分な情報が揃った段階でワーカーを起動する。発見が不十分な状態でワーカーを起動すると、プロンプトに不足情報が生じ、ワーカーの成果物が不正確になるリスクがある。

XMLタスク結果フォーマット

ワーカーからコーディネータへの通知は、以下のXMLフォーマットでラップされたユーザーメッセージとして届く。このフォーマットはコーディネータが結果を自動的に解析して統合するための標準インターフェースである。

<task_id>{タスクID}</task_id>
<status>{completed|failed|timeout}</status>
<summary>{結果の要約（1-3文）}</summary>
<result>{詳細な結果・生成物・変更内容}</result>

各フィールドの役割

  • task_id: コーディネータがどのタスクの結果かを識別するためのID
  • status: タスクの完了状態。failed や timeout の場合、コーディネータはリトライや代替戦略を検討する
  • summary: コーディネータが全体統合時に参照する要約。後続タスクのプロンプトに埋め込む際に使用する
  • result: 成果物の詳細。コーディネータが最終報告を作成する際の素材となる

ワーカーへの効果的な作業割り当て

コーディネータはエージェントアーキテクチャのAgentToolを介してワーカーを起動する。効果的な作業割り当てには以下の考慮が必要である。

タスク粒度の調整

タスクが細かすぎると通信オーバーヘッドが増大し、粗すぎると並行化の恩恵が得られない。一般的なガイドラインとして、1つのワーカーが5〜20分で完了できる作業量を目安とする。

ワーカーの再利用判断

コーディネータはワーカーを再利用するか新規生成するかを状況に応じて判断する。

  • 再利用が適切な場合: 同じファイルへの連続操作、前のタスク結果に直接依存する後続作業、コンテキストの重複度が高い場合
  • 新規生成が適切な場合: タスク領域が全く異なる場合、コンテキストが発散してキャッシュ効率が低い場合

ツール制限の考慮

コーディネータの内部ツール（チーム管理・メッセージング）はワーカーからアクセスできない。ワーカーに割り当てるプロンプトでは、ワーカーが利用可能なツールの範囲内で完結するタスクを設計する。

結果統合戦略

すべてのワーカーが完了した後、コーディネータは各ワーカーの結果を統合して最終的な成果物を作成する。結果統合は単純な結合ではなく、矛盾の解消・品質検証・全体的な一貫性の確保を含む。

統合のステップ

  1. 結果収集: 全ワーカーの <result> と <summary> を収集する
  2. 矛盾検出: 複数のワーカーが同一リソースを変更した場合、変更の整合性を確認する
  3. 品質検証: 各ワーカーの成果物が当初の要件を満たしているか評価する
  4. 統合実行: 成果物を統合し、必要に応じてコーディネータ自身が補完作業を行う
  5. 最終報告: 全体の作業結果をユーザーに報告する

失敗したタスクへの対応

ワーカーが failed や timeout を返した場合、コーディネータは以下の戦略を取る。

  • 追加情報を補強したプロンプトで同じタスクを再試行する
  • タスクをより小さい粒度に分割して再分解する
  • コーディネータ自身がそのタスクを引き受ける

詳細なプロンプト設計の知識は、失敗タスクのリカバリー戦略においても重要な役割を果たす。
