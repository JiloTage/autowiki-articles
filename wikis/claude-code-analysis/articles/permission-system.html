パーミッションシステム

最終更新: 2026-03-31
Claude Code のパーミッションシステムは、ツール実行の可否をリアルタイムに判定する多層構造のアクセス制御機構である。パーミッションモードの選択、ツールごとの checkPermissions() 呼び出し、エージェントスコープごとのツールセット制限、MCP チャネルのパーミッション管理などを通じて、安全で細粒度のアクセス制御を実現する。

概要

Claude Code のパーミッションシステムは、ユーザーが明示的に承認していない操作が自動実行されることを防ぐための仕組みである。全ツール実行の前に checkPermissions() が呼び出され、その結果に応じてツールの実行が許可・拒否・保留される。パーミッション設計は「最小権限の原則」に基づいており、必要な権限だけをエージェントごとに付与することで、ツールの誤用を防止する。

パーミッションシステムは以下のコンポーネントと密接に連携する。

  • ツールシステム: 各ツールが checkPermissions() メソッドを実装する
  • エージェントアーキテクチャ: エージェントごとのパーミッションスコーピング
  • QueryEngine: パーミッション拒否のトラッキング
  • MCP インテグレーション: MCP チャネルパーミッション管理

パーミッションモードの種類

Claude Code は起動オプションや設定に応じて複数のパーミッションモードを持つ。

モード説明主な用途
自動承認モード（autoApprove） 全ツール実行を自動的に承認する。ユーザーへの確認プロンプトが省略される。 CI/CD・バッチ処理・テスト環境
手動承認モード（manual） 破壊的操作（ファイル書き込み・シェル実行など）の前にユーザー確認を要求する。 インタラクティブな開発作業
読み取り専用モード（readOnly） 書き込み系ツールを完全に無効化し、読み取り操作のみ許可する。 コードレビュー・監査
ホワイトリストモード（allowList） 明示的に許可したツールのみ実行可能。デフォルト拒否の厳格なモード。 セキュリティ重視の環境
プランモード（plan） ツールの実行計画を作成するが実際には実行しない。ドライランに相当する。 変更の事前確認

パーミッションモードはセッション開始時に決定され、エージェント実行フローの「設定決定」フェーズでツールセットとともに確定される。サブエージェントはデフォルトで親エージェントのモードを継承するが、スコーピングによって上書きも可能である。

ツールごとの checkPermissions()

各ツールは Tool インターフェースの一部として checkPermissions() メソッドを実装している。このメソッドはツール実行の直前に呼び出され、現在のパーミッションコンテキストに基づいて実行の可否を判定する。

判定ロジックの主な要素は以下の通りである。

  • 操作種別: 読み取り（低リスク）・書き込み（中リスク）・実行（高リスク）の分類
  • 対象パス: ファイル操作の場合、アクセス先パスをプロジェクトルートと照合する
  • ツール固有ルール: BashTool のシェルコマンド検証、FileEditTool の上書き確認など
  • ユーザー設定: 永続的なホワイトリスト・ブラックリスト（settings.json で管理）

checkPermissions() が拒否を返した場合、実行はブロックされ、拒否イベントが QueryEngine の拒否トラッカーに記録される。インタラクティブモードでは、ユーザーへの確認プロンプトが表示される前に checkPermissions() が実行され、確認後に再評価される。

パーミッションスコーピング

パーミッションスコーピングは、エージェントごとに利用可能なツールセットを制限する仕組みである。エージェントアーキテクチャの「主要設計パターン」で述べられているように、ツールセットはエージェントごとに制限されながら SDK レベルのパーミッションは保持される。

スコーピングの主な目的は以下の通りである。

  • 最小権限の保証: タスクに不要なツールを持たないサブエージェントを生成できる
  • サンドボックス化: 特定の作業ディレクトリや操作範囲に限定したエージェントの構築
  • セキュリティ分離: 信頼度の低い入力を処理するエージェントから危険ツールを除外する

スコーピングはツールセットのマージ・重複排除メカニズムとも連携する。ビルトインツールが MCP ツールより優先されるルールにより、スコーピング設定が外部プラグインによって意図せず拡張されることを防ぐ。

管理者信頼ゲート

フロントマター機能（エージェント定義ファイルの先頭に記述されるメタデータ）をロードする前に、ソースの信頼性をチェックする「管理者信頼ゲート」が設けられている。

信頼ゲートは以下の順序でソースを評価する。

  1. ポリシー設定: 組織のポリシーから提供されたエージェントは最高信頼度
  2. プロジェクト設定: プロジェクトディレクトリ内の設定ファイルは高信頼度
  3. ユーザー設定: ユーザーのホームディレクトリ設定は中信頼度（ゲート通過が必要）
  4. プラグイン: 外部プラグインは検証必須
  5. 不明ソース: ゲートを通過できず、ロードが拒否される

この仕組みにより、悪意あるプロジェクトファイルや改ざんされたプラグインが危険なエージェント定義を注入するサプライチェーン攻撃を防ぐ。エージェントアーキテクチャのロード優先順位（ビルトイン→プラグイン→ユーザー→プロジェクト→フラグ→ポリシー）は、信頼ゲートの評価順序と整合している。

パーミッション拒否のトラッキング

QueryEngine は会話ライフサイクル管理の一部として、パーミッション拒否イベントをトラッキングする。拒否トラッカーは以下の情報を記録する。

  • 拒否されたツール名とその引数
  • 拒否の理由（モード制約・パスポリシー・ユーザー拒否など）
  • 拒否発生時のメッセージ ID（会話コンテキストとの紐付け用）
  • 累積拒否回数（セッション統計）

トラッキングデータは以下の用途に活用される。

  • モデルへのフィードバック: 拒否されたアクションをモデルに通知し、代替戦略を検討させる
  • テレメトリ: どのツールが頻繁に拒否されるかを収集し、UX 改善に活用する
  • デバッグ: パーミッション設定のトラブルシューティング時に拒否ログを参照する

拒否イベントは クエリループのエラーリカバリーメカニズムとも連携しており、連続した拒否が発生した場合に適切なフォールバック処理が行われる。

MCP チャネルパーミッション管理

MCP インテグレーションにおいて、各 MCP サーバーへの接続はチャネル単位でパーミッションが管理される。src/services/mcp/ のチャネルセキュリティモジュールが以下を担当する。

管理項目内容
チャネル認証 各 MCP サーバーへの接続時に OAuth / API キーなどで認証を実施する
ツール公開制御 MCP サーバーが公開するツールのうち、許可リストに含まれるものだけを登録する
スコープ制限 サーバーごとに許可する操作のスコープ（読み取り専用・書き込み可など）を設定する
接続ライフサイクル エージェント終了時に MCP 接続を解放し、リソースリークを防ぐ
依存フィルタリング 必要な MCP サーバーが利用可能な場合のみエージェントをロードする

MCP チャネルのパーミッション設定は settings.json の mcpServers セクションで定義され、エージェント実行フローの「設定決定」フェーズで読み込まれる。チャネルパーミッションの拒否も QueryEngine の拒否トラッカーに記録される。

設定との統合

パーミッションシステムの多くの設定は settings.json（ユーザー設定）および .claude/settings.json（プロジェクト設定）で永続化される。設定可能な主な項目を示す。

  • permissions.defaultMode: デフォルトのパーミッションモード
  • permissions.allowedTools: 常時自動承認するツールのホワイトリスト
  • permissions.deniedTools: 常時拒否するツールのブラックリスト
  • permissions.allowedPaths: アクセス許可するファイルパスパターン
  • mcpServers.*.permissions: MCP サーバーごとのチャネルパーミッション

プロジェクト設定はユーザー設定より優先され、ポリシー設定はすべてに優先する。この優先順序はエージェントアーキテクチャのロード優先順位と一致しており、システム全体で一貫したパーミッション管理が行われる。
