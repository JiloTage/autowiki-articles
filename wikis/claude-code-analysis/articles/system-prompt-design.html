システムプロンプト設計

最終更新: 2026-03-31
Claude Code のシステムプロンプト設計は、各ツールの prompt.ts ファイルと起動時に構築されるシステムプロンプトから成り、AI の行動を精密に制御する多層的な指示体系である。プロンプト構造・AI行動制御メカニズム・コンテキスト依存生成・AgentTool やコーディネータモードの特殊設計・キャッシュ最適化戦略にわたる包括的な設計パターンを持つ。

システムプロンプトの構造と役割

Claude Code のシステムプロンプト設計は、単一のプロンプトではなく複数の層から構成される階層的な指示体系である。最上位にはグローバルなシステムプロンプト（src/constants/prompts.ts 等）が存在し、その下に各ツール固有の prompt.ts ファイルが位置する。ツール説明文はツールが LLM のコンテキストに組み込まれる際に動的にアセンブルされ、全体として一貫したAI行動制御を実現する。

システムプロンプトの主な役割は以下の通りである。

  • 行動規範の確立: AIが何をすべきか、すべきでないかを明確に定義する
  • ツール使用の誘導: 各ツールの適切な使用タイミングと方法を指示する
  • セキュリティ境界の設定: 破壊的操作や不適切な動作を防止する
  • 品質基準の維持: 出力フォーマットや一貫性を担保する

prompt.ts ファイルの目的と設計パターン

Claude Code では各ツールが独立した prompt.ts ファイルを持つ。このファイルはツールの description() メソッドが返す文字列を定義するものであり、LLM がそのツールをどのように扱うべきかを指示する。ツールシステムの設計原則に従い、各ツールディレクトリは実装・UI・プロンプト・セキュリティ検証を明確に分離している。

代表的な設計パターンを以下に示す。

  • 禁止事項の明示: 「bash の grep/rg の代わりに必ず Grep ツールを使用する」のように、代替手段を明示しつつ禁止事項を列挙する
  • 条件分岐の記述: 「A の場合は X を、B の場合は Y を使用する」という形式で状況別の指示を提供する
  • 優先順位の付与: 複数の選択肢がある場合に推奨順序を示す
  • 例示による補強: 抽象的な指示に具体例を添えて理解を助ける

AI行動制御のメカニズム

Claude Code のシステムプロンプトは「何をすべきか（DO）」と「何をすべきでないか（DON'T）」の両面から AI の行動を制御する。この二元的な制御は特にセキュリティとユーザー体験に関わる領域で顕著である。

すべきこと（DO）の指示

  • 独立したコマンドは単一メッセージ内で複数ツール呼び出しとして並行実行する
  • 順次実行が必要な依存コマンドは && でチェーンして単一 Bash 呼び出しにまとめる
  • ファイル編集前に必ず FileReadTool で内容を読み取る
  • old_string の一意性を確保するため、2〜4行の隣接するコンテキストを含める
  • エージェント起動時には常に3〜5語の説明を含める

すべきでないこと（DON'T）の指示

  • 明示的に要求されない限り、破壊的 git コマンド（reset --hard、checkout . 等）を実行しない
  • 明示的に要求されない限り、コミットを作成しない
  • main/master への force-push は絶対に行わない
  • bash コマンドで grep/rg/find/cat 等を実行しない（専用ツールを使用する）
  • フォーク結果を捏造・予測しない（"Never fabricate or predict fork results"）
  • 著作権コンテンツを逐語的に複製しない

コンテキスト依存のプロンプト生成（description()メソッド）

各ツールの description() メソッドは静的な文字列を返すだけでなく、実行時のコンテキスト情報を受け取って動的にプロンプト文字列を生成できる設計になっている。これにより、ツールの説明がその時点のシステム状態やセッション設定を反映した内容になる。

コンテキスト依存生成の主な用途を示す。

  • サンドボックスモード: サンドボックスが有効な場合、BashTool の説明に「一時ファイルには $TMPDIR を使用する」という制約が追加される
  • パーミッション設定: 許可されている操作の範囲に応じてツール説明が変化する
  • MCP サーバー情報: 接続中の MCP サーバーに応じてツールセットが動的に変化する（MCP インテグレーション参照）
  • wiki ID 自動解決: ワーカーエージェントへのブリーフィング内容がコンテキストに応じて変化する

AgentTool のプロンプト設計（16.7KB）

AgentTool の prompt.ts は全ツール中最大の16.7KBを誇り、エージェントアーキテクチャの複雑さを反映した詳細な指示体系を持つ。その設計には並行実行・フォーク機能・エージェントブリーフィングの3つの主要テーマがある。

並行実行の強調

AgentTool のプロンプトは並行実行を最優先の原則として位置づける。「単一メッセージ内で複数のツール使用コンテントブロックを送信する」という指示は、逐次処理よりも並行処理を優先することを明確にしている。これにより、独立したサブタスクは同時に実行され、全体の処理時間が短縮される。

フォーク機能の制御

AgentTool のフォーク機能には特有の制約が設けられている。

  • "Don't peek": フォーク出力を覗き見しない。フォーク結果を待たずに次の処理を進める
  • "Never fabricate or predict fork results": フォーク結果を捏造・予測しない。実際の結果のみを使用する
  • エージェント結果をユーザーに表示する前に要約する

エージェントブリーフィング

AgentTool のプロンプトは「スマートな同僚が部屋に入ってきたようにエージェントにブリーフィングする」という比喩を用いて、エージェントへの指示の書き方を示す。すべてのプロンプトはワーカーが必要とするすべてを含む自己完結型でなければならない。

コーディネータモードのプロンプト設計

コーディネータモードのプロンプトは並行化を設計哲学の中心に置く。「並列性はあなたのスーパーパワー。ワーカーは非同期。可能な限り独立したワーカーを並行起動する」というメッセージは、コーディネータに期待される行動様式を端的に表している。

コーディネータプロンプトの主要な指示を以下に示す。

  • 自己完結型プロンプト: 「すべてのプロンプトはワーカーが必要とするすべてを含む自己完結型でなければならない」
  • ワーカー再利用戦略: コンテキスト重複が高い場合はワーカーを再利用する；コンテキストが分岐した場合は新規ワーカーを生成する
  • タスク結果フォーマット: <task_id>・<status>・<summary>・<result> を含む XML ラップメッセージで結果を受け取る
  • 独立性の確保: ワーカー間の依存関係を最小化し、最大限の並行化を実現する

プロンプトキャッシュ安定性の維持

Claude Code は Anthropic の API プロンプトキャッシュを積極的に活用してコストを削減している。プロンプトキャッシュが有効に機能するためには、同一の入力に対して同一のプロンプト文字列が生成される必要がある。このため、システムプロンプト設計にはキャッシュ安定性を維持するための工夫が組み込まれている。

キャッシュ安定性を支える設計上の決定を以下に示す。

  • ツール説明の不変性: ツールの説明文は実行ごとに変化しないように設計されており、API のプロンプトキャッシュヒット率を最大化する
  • 名前での重複排除: ツールセットのアセンブリ時に名前での重複排除を行い、同一ツールが複数回出現することを防ぐ
  • 順序の安定性: ツール一覧の順序を安定させることでキャッシュヒット率を高める
  • 動的コンテンツの分離: セッション固有の情報はシステムプロンプトの末尾や別のブロックに配置し、キャッシュ可能な静的部分を最大化する

この設計により、同一の設定を持つ複数のセッションが共通のシステムプロンプト部分を再利用でき、API コストの大幅な削減が実現される。プロンプトキャッシュ安定性はプロンプト設計の最重要考慮事項の一つである。
