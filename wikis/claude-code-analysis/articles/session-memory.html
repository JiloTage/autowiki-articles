セッションメモリ

最終更新: 2026-03-31
Claude Code のセッションメモリシステム（src/services/SessionMemory/）は10セクション構造のテンプレートを用いた永続的メモリ管理を提供する。セクションごとに約2000トークンの制限があり、並行 Edit ツール呼び出しで更新される。「Current State」セクションの継続的更新により会話の継続性を維持する。

セッションメモリ概要

セッションメモリは src/services/SessionMemory/ に実装されており、Claude Code が長時間のセッションや複数のセッションをまたいでコンテキストを維持するための仕組みである。構造化された10セクションのテンプレートを使用して、プロジェクトに関する重要な情報を整理・保持する。

10セクション構造

セッションメモリは以下の10セクションで構成される。

  1. Session Title: セッションの識別名
  2. Current State: 現在の作業状態（継続性のため常に最新に更新）
  3. Task Specification: タスクの仕様・要件
  4. Files and Functions: 関連ファイル・関数の一覧
  5. Workflow: 作業ワークフロー・手順
  6. Errors & Corrections: エラーと修正内容の記録
  7. Codebase Documentation: コードベースのドキュメント
  8. Learnings: セッション中に得た知見
  9. Key Results: 重要な成果・結果
  10. Worklog: 作業ログ

更新戦略

セッションメモリの更新は以下の原則に従う。

  • 並行 Edit 呼び出し: 複数セクションを並行して更新する
  • 厳格なヘッダー保持: セクションヘッダーは変更しない
  • トークン制限遵守: セクションあたり約2000トークンに収める
  • 選択的圧縮: 重要度の低い詳細を削除して新情報のための余地を作る
  • Current State の優先更新: 継続性確保のため必ず更新する

セッションメモリとコンテキスト圧縮の違い

セッションメモリとコンテキスト圧縮は異なる目的を持つ。

  • セッションメモリ: 構造化された長期情報の保持；セッションをまたいで永続する
  • コンテキスト圧縮: トークン制限への対応；会話履歴を要約して圧縮する

セッションメモリはコンテキスト圧縮が行われても保持される構造化データとして機能する。

クエリループとの統合

セッションメモリはクエリループのセットアップフェーズでプリフェッチされ、LLM への初期コンテキストとして提供される。これにより Claude は新しいセッションを開始する際にも前回の作業状態を即座に把握できる。

エージェントメモリスナップショット

エージェントシステムも独自のメモリ管理を持つ（src/tools/AgentTool/agentMemory.ts・agentMemorySnapshot.ts）。これらはエージェントの実行状態をスナップショットとして保存し、エージェントの再開時に復元する機能を提供する。セッションメモリとは独立して動作する。
