MCP OAuth 認証フロー

最終更新: 2026-03-31
Claude Code の MCP OAuth 認証フローは src/services/mcp/auth.ts（88 KB）に実装されており、OAuth 2.0 認可コードフローによる MCP サーバーへの安全なアクセスを提供する。トークンの取得・更新・失効処理に加え、チャネル許可リストとパーミッション管理を組み合わせた多層セキュリティモデルを採用している。

auth.ts の全体構成と役割

auth.ts（88 KB）は MCP インテグレーション における認証レイヤーの中核である。このファイルは OAuth 2.0 フローの完全な実装を提供し、認可コードの取得からアクセストークンの管理まで一貫して処理する。

主要な責務は以下の通りである：

  • 認可エンドポイントへのリダイレクト管理：ブラウザを介した認可コードフローの起動
  • トークン交換：認可コードをアクセストークン・リフレッシュトークンと交換
  • トークンライフサイクル管理：有効期限チェック・自動リフレッシュ・失効処理
  • セキュアなトークン保存：OS キーチェーンまたは暗号化ローカルストレージへの永続化
  • チャネルセキュリティとの統合：チャネル許可リストおよびパーミッションとの連携

OAuth 2.0 フローの実装詳細

auth.ts における OAuth 2.0 実装は認可コードフロー（Authorization Code Flow）を採用しており、以下の3ステップで構成される。

認可コード取得

フローの開始時、Claude Code はローカルポートでコールバックリスナーを起動し、認可エンドポイントへのリダイレクト URL を生成する。URL には client_id・scope・redirect_uri・state（CSRF 対策用ランダム文字列）・code_challenge（PKCE 対応）が含まれる。ユーザーのデフォルトブラウザを開いて認可画面を表示し、認証・同意後にコールバック URL へリダイレクトされた認可コードを受け取る。

アクセストークン取得

認可コードを受け取った後、auth.ts はトークンエンドポイントに POST リクエストを送信する。リクエストボディには grant_type=authorization_code・受け取った code・redirect_uri・client_secret（または PKCE の code_verifier）が含まれる。レスポンスとして access_token・token_type・expires_in・refresh_token・scope が返却される。

トークンリフレッシュ

アクセストークンの有効期限が近づいた場合、または API 呼び出し時に 401 エラーが返却された場合、auth.ts は自動的にリフレッシュフローを実行する。grant_type=refresh_token と refresh_token を送信して新しいアクセストークンを取得し、トークンストアを更新する。リフレッシュトークン自体も期限切れの場合は再認証フローへ誘導する。

トークン管理

取得したトークンの保存・更新・失効処理は auth.ts が一元管理する。

トークン保存

トークンはサーバーの識別子（ホスト名 + パスのハッシュ）をキーとしてセキュアストレージに保存される。macOS では Keychain、Linux では libsecret（Secret Service）、Windows では Credential Manager を優先的に使用する。セキュアストレージが利用できない環境ではファイルシステムへの暗号化保存にフォールバックする。保存されるデータには access_token・refresh_token・expires_at（Unix タイムスタンプ）・scope が含まれる。

トークン更新

各 API 呼び出し前に auth.ts は保存済みトークンの有効期限を確認する。expires_at - current_time < buffer_seconds（通常 60〜300 秒のバッファ）の場合は事前にリフレッシュを実行する。この先読みリフレッシュにより、実際の API 呼び出し中にトークンが期限切れになるリスクを低減する。

トークン失効処理

MCP サーバーの削除時・ユーザーの明示的なログアウト時・認証エラーが継続する場合に、auth.ts はトークンを失効させる。失効時は認可サーバーの revocation エンドポイント（RFC 7009）に revoke リクエストを送信してサーバー側でも無効化し、ローカルストレージからも削除する。

チャネルセキュリティ

認証レイヤーはトークン管理だけでなく、チャネルレベルのアクセス制御とも連携している。

channelAllowlist.ts の役割

channelAllowlist.ts は MCP サーバーが接続を許可するチャネルの許可リストを管理する。チャネルとは Claude Code 内部のメッセージルーティング単位であり、特定のエージェントコンテキストや UI コンポーネントに対応する。許可リストは MCP サーバーごとに設定され、設定ファイルまたは動的なユーザー承認によって更新される。OAuth 認証済みのサーバーであっても、チャネルが許可リストに含まれていなければ通信は拒否される。

channelPermissions.ts によるパーミッション管理

channelPermissions.ts は細粒度のチャネルパーミッションを実装する。各 MCP サーバーに対して許可されるアクション（ツール実行・リソース読み取り・サンプリング要求など）をビットマスクまたは列挙型で管理する。パーミッションは OAuth スコープと対応しており、トークン取得時に要求したスコープ内でのみ操作が許可される。ツールシステムのアセンブリ時にチャネルパーミッションが評価され、許可されていないツールはプールに追加されない。

認証済み MCP サーバーへの安全なアクセス制御

OAuth 認証・チャネル許可リスト・チャネルパーミッションの3層が組み合わさることで、MCP サーバーへの多層防御が実現される。

レイヤー 実装ファイル チェック内容
OAuth 認証 auth.ts 有効なアクセストークンの存在
チャネル許可リスト channelAllowlist.ts 接続元チャネルが許可リストに含まれるか
チャネルパーミッション channelPermissions.ts 要求操作がスコープ内のパーミッションを持つか

各 API 呼び出しはこの3層を順番に通過する。いずれかのレイヤーで拒否された場合は適切なエラーコードが返却され、エージェントアーキテクチャのエラーハンドリング機構によって上位レイヤーへ伝播される。

接続ライフサイクルとの統合

OAuth 認証フローは useManageMCPConnections.ts が管理する接続ライフサイクルと密接に統合されている。接続確立時に auth.ts の認証状態が確認され、未認証の場合は自動的に OAuth フローが起動される。

接続の状態遷移は以下の通りである：

  1. DISCONNECTED：サーバー未接続の初期状態
  2. AUTHENTICATING：OAuth フロー進行中（ブラウザ認可待ち）
  3. TOKEN_ACQUIRED：トークン取得完了・接続試行中
  4. CONNECTED：認証済み・API 呼び出し可能
  5. TOKEN_EXPIRED：トークン期限切れ・リフレッシュ試行中
  6. ERROR：認証失敗・再試行またはユーザー介入が必要

エージェントの終了時には runAgent.ts のクリーンアップフェーズでトークンのリフレッシュスケジューラが停止し、接続が適切に解放される。セッション間でのトークン持続は設定により制御可能であり、ユーザーが毎回の認証を省略できる。
