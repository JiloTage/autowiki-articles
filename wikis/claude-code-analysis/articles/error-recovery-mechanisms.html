エラーリカバリー機構

最終更新: 2026-03-31
Claude Code のエラーリカバリー機構は、クエリループ（src/query.ts）のフェーズ4として実装されており、プロンプト長超過・最大出力トークン数到達・ネットワーク障害・ツール実行失敗・パーミッション拒否など多様な障害モードに対する体系的な回復戦略を提供する。各エラータイプごとに専用のリカバリーパスが定義されており、可能な限りユーザーセッションを中断せずに処理を継続できる設計になっている。

エラーリカバリーの全体像

エラーリカバリー機構は クエリループ のフェーズ4に位置し、モデル呼び出し後に発生する回復可能なエラーを分類・処理する責務を持つ。エラーは大別すると「コンテキスト制限エラー」「API/ネットワークエラー」「ツール実行エラー」「操作エラー」の4カテゴリに分類される。

以下のダイアグラムはエラーリカバリーフロー全体を示す。

プロンプト長超過のリカバリー

API が prompt_too_long エラーを返した場合、クエリループ はただちに コンテキスト圧縮 パイプラインを起動する。このエラーはコンテキストウィンドウの上限を超えたことを示しており、メッセージ履歴を削減しなければ同一リクエストを再送しても同じ結果になる。

リカバリーは次の3段階で行われる。

  1. コンパクションドレイン: 既にキューに入っているコンパクション操作を先に完了させる。これにより重複圧縮を防ぐ。
  2. リアクティブ圧縮: エラーをトリガーとして即時にコンパクションを実行する。古いメッセージの snip・microcompact・collapse を適用してトークン数を削減する。
  3. ループ再試行: 圧縮後のコンテキストで同一のモデル呼び出しを再実行する。圧縮後もまだ長すぎる場合はさらなる圧縮を試みる。

autocompact が有効な場合、コンテキストが一定閾値（通常95%）を超えた時点で予防的にコンパクションが起動されるため、reactive な圧縮が必要になる頻度は低減される。

最大出力トークン数到達時の継続処理

max_tokens ストップシーケンスでレスポンスが打ち切られた場合、クエリループはトークンエスカレーション戦略を適用する。これはモデルの出力が完結していないにもかかわらず、トークン制限に達したことを意味する。

具体的な処理は以下の通りである。

  • 継続リクエスト: 現在の未完了レスポンスをアシスタントメッセージとしてコンテキストに追加し、「続きを生成してください」という暗黙の継続クエリを送る。
  • モデルエスカレーション: フォールバックモデルが設定されている場合、より大きなコンテキストウィンドウや出力制限を持つモデルへ切り替えることがある。
  • ストリーミング結合: 継続レスポンスは元のレスポンスとシームレスに結合されて UI に表示される。

ネットワークエラー・APIエラーのリトライ戦略

ネットワーク障害や一時的な API エラーは src/services/api/withRetry.ts（28.2 KB）が処理する。このモジュールはクエリループから独立しており、API 呼び出し層でラップされている。

リトライ戦略の主要パラメータを以下に示す。

  • 対象エラー: HTTP 429（レート制限）・5xx系エラー・ネットワークタイムアウト・接続リセット
  • 非対象エラー: 4xx系クライアントエラー（認証失敗・パラメータエラーなど）はリトライせずに即時伝播
  • バックオフ戦略: 指数バックオフ（Exponential Backoff）＋ジッター。初回リトライまでの待機時間は短く、リトライ回数が増えるにつれて指数的に延長される。
  • 最大リトライ回数: 設定可能。デフォルトでは数回のリトライ後にエラーをクエリループへ伝播する。
  • フォールバックモデル: 主要モデルへのリクエストが繰り返し失敗する場合、フォールバックモデルへ自動切り替えすることでサービス継続性を維持する。

ツール実行エラーの処理

ツールシステム でエラーが発生した場合、クエリループはそれをセッション終了要因とは扱わない。代わりにエラー情報を tool_result メッセージとして LLM に返し、次のターンで LLM 自身がリカバリーを判断できるようにする。

ツールエラー処理の詳細フロー。

  1. ツール実行時に例外が発生すると、エラーメッセージとスタックトレース（デバッグモード時）が収集される。
  2. エラーは {"type": "tool_result", "is_error": true, "content": "...エラー詳細..."} 形式でコンテキストに追加される。
  3. LLM は次のターンでこのエラーを認識し、代替アプローチの試行・ユーザーへの説明・処理の中断などを自律的に判断する。
  4. 並行ツール実行（isConcurrencySafe() が true のツール群）では、一部のツールが失敗しても他のツールの実行は継続される。

パーミッション拒否時の処理フロー

QueryEngine はパーミッション拒否をトラッキングする専用の状態を持つ。ツールがユーザーによって拒否された場合、その情報はセッション全体で保持され、同一ツールへの再試行を抑制する。

パーミッション拒否のハンドリング。

  • インタラクティブ承認: デフォルトではツール実行前にユーザーへ承認を求める。ユーザーが拒否した場合、拒否済みリストに追加される。
  • 拒否済みツールの扱い: 拒否済みとしてマークされたツールは、同一セッション内での再呼び出し時に即座に is_error: true の tool_result を返す（再度プロンプトを表示しない）。
  • LLM へのフィードバック: 拒否情報は tool_result を通じて LLM に伝わり、LLM は代替手段を検討するか、作業の断念をユーザーに説明する。
  • スコープ分離: サブエージェント（AgentTool）は独自のパーミッションスコープを持ち、親エージェントの拒否リストを継承しない設計になっている。

abort/キャンセル処理

ユーザーが Ctrl+C などでセッションを中断した場合、またはプログラム的に abort シグナルが発火した場合、クエリループは安全なクリーンアップを実施してから終了する。

using ステートメント（Symbol.asyncDispose）を活用したリソース管理により、ジェネレータのどの終了パスでも確実にクリーンアップが実行される。具体的には以下が保証される。

  • 進行中のツール実行の中断: abort シグナルは実行中のツールに伝播され、可能な場合はツール実行がキャンセルされる。
  • ストリーミングの停止: API ストリームが即座にクローズされ、部分的なレスポンスはバッファから破棄される。
  • セッション状態の保存: abort 前の会話履歴は セッションメモリ に保存され、次回の再開時に利用できる。
  • 非同期リソースの解放: ファイルハンドル・ネットワーク接続などのリソースが確実に解放される。
