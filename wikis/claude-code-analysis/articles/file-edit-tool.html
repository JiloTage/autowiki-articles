FileEditTool の設計

最終更新: 2026-03-31
FileEditTool は Claude Code におけるファイル精密編集の中核ツールであり、old_string/new_string 方式による差分編集と事前読み取り強制によって安全かつ正確なファイル変更を実現する。FileReadTool との連携フロー・一意性検証・replace_all オプションなど、設計上の重要な原則を詳しく解説する。

FileEditTool の概要

FileEditTool は Claude Code のファイル操作ツール群の中でも最も頻繁に使用される編集専用ツールである。ツールシステムの一部として、ファイル全体を上書きする FileWriteTool とは異なり、既存ファイルの特定の文字列を検索・置換する差分編集アプローチを採用している。これにより、大きなファイルの一部だけを変更する際に不要なデータの再送信を避けつつ、精密なコントロールを実現する。

FileEditTool の実装は src/tools/FileEditTool/ ディレクトリ以下に配置されており、FileEditTool.ts（コア実装）・prompt.ts（ツール説明・指示）・UI.tsx（ターミナルUI）・utils.ts（ヘルパー関数）の標準ディレクトリ構成を持つ。

事前読み取り要求の設計原則

FileEditTool には、編集を実行する前に対象ファイルを必ず FileReadTool で読み取ることを強制するという重要な設計原則がある。この仕組みはプロンプトレベルの指示として実装されており、AI が FileReadTool を使わずに FileEditTool を呼び出した場合はエラーが返される。

この制約には以下の設計上の意図がある。

  • 正確性の保証: ファイルの最新内容を把握した上で old_string を指定することで、意図しない箇所への変更を防ぐ
  • 競合の検出: 外部プロセスによってファイルが変更されている場合、読み取り時点と編集時点の内容の乖離を検出できる
  • コンテキストの維持: プロンプト設計の観点から、AI がファイル全体のコンテキストを把握した上で編集箇所を決定できる

この「先読み強制」パターンは、FileEditTool の prompt.ts に以下のような指示として明示されている: 「You MUST have read the file before editing it. If you haven't read the file yet, use the Read tool first.」

old_string / new_string 方式による精密編集

FileEditTool の入力スキーマは3つのパラメータで構成される。

  • file_path: 編集対象ファイルの絶対パス
  • old_string: ファイル内で検索する文字列（空文字列の場合は新規ファイル作成）
  • new_string: old_string を置換する新しい文字列

このアプローチは「行番号ベース編集」と比較して以下の利点を持つ。

  • 行番号はファイルの変更によってずれるが、文字列は変更内容そのものを表現する
  • 意図した変更箇所を自然言語に近い形で表現でき、誤操作のリスクが低い
  • 小さな差分のみを送信するため、トークン効率が高い

old_string に空文字列を渡した場合、FileEditTool は新規ファイル作成モードで動作し、new_string の内容でファイルを新規作成する。これにより FileWriteTool の機能を代替できる場面もあるが、通常の新規ファイル作成には FileWriteTool を使うことが推奨される。

一意性検証

FileEditTool の最重要な安全機構の一つが一意性検証である。old_string がファイル内に複数存在する場合、FileEditTool はエラーを返して編集を中止する。

一意性検証によって生じるエラーへの対処法としては、以下が推奨される。

  • old_string に前後の文脈行を追加して一意な文字列を構成する
  • 意図的にすべての出現箇所を置換したい場合は replace_all オプション（後述）を使用する

replace_all オプションの動作

FileEditTool には replace_all というオプションパラメータが存在し、true に設定すると一意性検証をスキップしてファイル内のすべての old_string 出現箇所を new_string で置換する。

このオプションの主なユースケースは以下の通りである。

  • 変数・関数のリネーム: コードベース内の特定のシンボル名をファイル全体で統一的に変更する
  • 一括テキスト置換: 設定ファイルや文書ファイル内の特定フレーズを全箇所で更新する
  • コードスタイルの統一: 特定のパターンをすべて新しいパターンで置き換える

replace_all は強力な操作であるため、意図せぬ箇所への変更を避けるために old_string は十分に具体的な文字列を指定することが重要である。

ファイル書き込み後の検証

FileEditTool は編集を実行した後、書き込み結果の検証を行う。検証プロセスには以下が含まれる。

  • 差分の表示: UI.tsx によって編集前後の差分がターミナルに表示され、ユーザーが変更内容を視覚的に確認できる
  • ファイル読み取り推奨: 編集後に再度 FileReadTool でファイルを読み取ることが推奨され、意図した変更が正しく反映されているかを確認する
  • エラーハンドリング: ファイルシステムへの書き込み失敗（権限エラー・ディスクフルなど）が発生した場合、詳細なエラーメッセージが返される

プロンプト設計の観点から、FileEditTool の prompt.ts は「編集後は必ずファイルを読み返して変更内容を確認せよ」という指示を含んでいる。これは特にインデントや改行など、視覚的に把握しにくい変更が意図通りかを確認するために重要である。

FileWriteTool との使い分け

Claude Code のファイル書き込みツールは FileEditTool と FileWriteTool の2種類があり、それぞれ異なる用途に最適化されている。

観点 FileEditTool FileWriteTool
操作の種類 差分編集（部分置換） 全体上書き
事前読み取り 必須（強制） 推奨（強制ではない）
トークン効率 高（差分のみ送信） 低（全内容を送信）
適用場面 既存ファイルの一部変更 新規ファイル作成・完全な書き直し
誤操作リスク 低（一意性検証あり） 高（既存内容を完全に上書き）

一般的なルールとして、既存ファイルの変更には FileEditTool を、新規ファイルの作成には FileWriteTool を使用することが推奨される。ただし、ファイルの大部分を書き直す場合は FileWriteTool の方が効率的な場合もある。
