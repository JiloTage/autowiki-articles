エージェントライフサイクル

最終更新: 2026-03-31
Claude Code のエージェントライフサイクルは runAgent.ts が定義する6フェーズ（初期化・状態準備・設定決定・フック/スキル登録・ストリーミング処理・クリーンアップ）で構成される。エージェントの再開・サブエージェントのフォーク・メモリスナップショット管理など、複雑な実行モデルを詳しく解説する。

概要

エージェントライフサイクルとは、Claude Code の エージェントアーキテクチャ におけるエージェントの生成から破棄までの一連のフェーズを指す。中核実装ファイルは src/tools/AgentTool/runAgent.ts（35.8 KB）であり、エージェントが安全・効率的・一貫性を持って動作するための詳細なフロー管理を提供する。

以下の状態遷移図はエージェントライフサイクルの全体像を示す。

6フェーズの詳細

フェーズ1: 初期化

エージェントが起動されると最初に行われる処理である。ユニークなエージェント ID（UUID）が生成され、テレメトリシステムへの登録が行われる。このエージェント ID はライフサイクル全体を通じてトレース・ロギング・デバッグの基準となる識別子であり、サブエージェントがフォークされた際も親子関係をIDで追跡できる。

フェーズ2: 状態準備

ファイル状態のキャッシュを初期化し、パーミッションコンテキストを解決する。具体的には以下の処理が行われる。

  • ファイルシステムの現在状態をキャッシュして後続の操作を効率化する
  • 親エージェントやシェル環境から引き継ぐパーミッションコンテキストを解決する
  • 作業ディレクトリ・環境変数などのランタイム状態を確定する

フェーズ3: 設定決定

エージェントが使用するリソースと権限の範囲を決定するフェーズである。MCP インテグレーションとの接続確立もこのフェーズで行われる。

  • パーミッションモード: エージェントが持つ権限レベル（読み取り専用・フルアクセスなど）
  • ツールセット: このエージェントが利用可能な ツール の集合（親エージェントのツールセットからさらにスコープを絞ることが可能）
  • MCP接続: 必要な MCP サーバーへの接続を確立する

フェーズ4: フック/スキル登録

エージェントのふるまいをカスタマイズする拡張ポイントを登録するフェーズである。

  • フロントマターフック: エージェント定義ファイルのフロントマター（YAML/TOML ヘッダ）に記述されたフックを解析・登録する。登録前に管理者信頼ゲート（Admin Trust Gate）によりソースの信頼性が検証される
  • スキルプリロード: エージェントに紐づくスキルを初期メッセージとしてプリロードする。これにより プロンプト設計 で指定された特定能力がエージェントのコンテキストに組み込まれる

フェーズ5: ストリーミング処理

エージェントの主要な実行フェーズである。クエリループ を通じてモデルからのレスポンスをストリーミングし、ツール呼び出しの処理やマルチターン会話の管理を行う。

  • クエリループ: LLM へのリクエスト送信・ストリーミングレスポンス受信・ツール実行・継続判断のサイクルを繰り返す
  • インクリメンタルトランスクリプト記録: 会話の各ターンをリアルタイムで永続化し、障害時の回復や監査ログとして利用する
  • 中断処理: abort コントローラーによる中断シグナルを監視し、適切なタイミングで処理を停止できる

フェーズ6: クリーンアップ

エージェントが終了する際に確実にリソースを解放するフェーズである。このフェーズは正常終了・エラー終了・中断のいずれの場合も実行される。

  • MCP 接続の解放（サーバーとの接続を適切に閉じる）
  • フック登録のクリア（メモリリークを防止）
  • バックグラウンドタスクの終了待機
  • エージェント状態のガベージコレクション（GC）

エージェント再開（resumeAgent.ts）

resumeAgent.ts（9.7 KB）は中断されたエージェントを再開する機能を提供する。エージェントが中断した場合や長時間タスクの途中でセッションを継続する必要がある場合に使用される。

再開メカニズムの概要を以下の図に示す。

再開時は以下の手順で状態が復元される。

  1. 永続化されたトランスクリプト（会話履歴）を読み込む
  2. メモリスナップショット（agentMemorySnapshot.ts）から実行時状態を復元する
  3. MCP 接続・ツールセット・パーミッションコンテキストを再確立する
  4. フェーズ5（ストリーミング処理）から処理を再開する

サブエージェントフォーク（forkSubagent.ts）

forkSubagent.ts（8.7 KB）は親エージェントの状態からサブエージェントを生成する機能を提供する。コーディネータモード における並列処理の基盤となる仕組みである。

フォーク設計原則

サブエージェントフォークには2つの中核的設計原則がある。

  • Don't peek（覗き見禁止）: 親エージェントはフォークしたサブエージェントの内部実行状態を直接参照してはならない。サブエージェントの処理は独立したコンテキストで完結する必要がある
  • Never fabricate or predict fork results（捏造・予測禁止）: フォーク結果は実際の実行結果のみが正当であり、モデルがフォーク出力を推測・補完することは禁止される

コンテキスト分離

フォークされたサブエージェントのコンテキスト分離方針は以下の通りである。

  • 非同期サブエージェント: 親コンテキストとは切り離された独立した abort コントローラーを取得する。これにより親が終了してもサブエージェントが継続実行できる
  • 同期サブエージェント: 親の abort コントローラーを共有する。親がキャンセルされるとサブエージェントも連動して停止する
  • ツールセット制限: サブエージェントのツールセットは親エージェントのツールセットからさらに制限されうる（権限昇格は不可）

メモリスナップショット管理

agentMemory.ts（5.9 KB）と agentMemorySnapshot.ts（5.6 KB）がエージェントのランタイム状態を管理する。セッションメモリ システムとは異なり、エージェントメモリはエージェント実行インスタンスに紐づく一時的な状態を管理する。

スナップショットの構造

メモリスナップショットは以下のデータを含む。

  • 実行コンテキスト: 現在のフェーズ・ツール実行結果・中間計算結果
  • 会話状態: メッセージ履歴・ツール呼び出し履歴・モデルへの送信済みコンテキスト
  • リソース参照: オープン中のファイルハンドル・確立済みMCP接続のメタ情報

スナップショットのライフサイクル

スナップショットは以下のタイミングで作成・更新・破棄される。

  • 作成: フェーズ1（初期化）完了後、または再開時の状態復元時
  • 更新: ストリーミング処理中、各ターンの完了時にインクリメンタルに更新される
  • 破棄: フェーズ6（クリーンアップ）の GC 処理時に解放される
