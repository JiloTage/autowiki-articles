タスクライフサイクル

最終更新: 2026-04-09
MulticaのタスクライフサイクルはGo製タスクサービス（internal/service/task.go）が管理し、エンキュー→クレーム→開始→完了/失敗の各状態遷移でWebSocketイベントをブロードキャストしながらエージェントによる自律的なIssue解決を実現する。

タスクライフサイクルとは、MulticaプラットフォームにおいてIssueに割り当てられたエージェントタスクが「エンキュー（enqueue）」から「完了（complete）または失敗（fail）」へと至る一連の状態遷移プロセスを指す。バックエンドのinternal/service/task.goがオーケストレータとして機能し、各フェーズでWebSocketイベントをすべての接続クライアントへブロードキャストすることでリアルタイム同期を実現している。Issueのステータスはタスクのステータスと自動的に同期されるため、プロジェクトボード上で進捗が即座に反映される。

ライフサイクル状態の概要

Multicaのタスクは4つの主要状態と2つの終端状態から構成されるステートマシンとして管理される。状態遷移は一方向性であり、後退（ロールバック）はサポートされない。各状態は対応するIssueステータスと1対1で同期される。

タスク状態 Issueステータス 説明
enqueued 待機中 エージェントへの割り当てが確定し、デーモンによるクレームを待つ状態
claimed 割り当て済み 特定のデーモンインスタンスがタスクを排他的に取得した状態
started 進行中 エージェントCLIプロセスが起動し、実際の作業を実行している状態
completed 完了 エージェントCLIが正常終了し、成果物が報告された終端状態
failed 失敗 エラー・タイムアウト等によりエージェントCLIが異常終了した終端状態
図1: タスクライフサイクルのステートマシン図。Issue assigneeへのエージェント設定でenqueuedが発火し、各遷移でWebSocketイベントがブロードキャストされる。

エンキューフェーズ

タスクのライフサイクルはIssueのアサイニーにエージェントが設定された瞬間に開始される。MulticaのIssueスキーマは多態型アサイニー（assignee_type + assignee_id）を採用しており、assignee_type = 'agent' が設定されるとtask.goのEnqueueTask関数が呼び出される。

エンキューフェーズでは以下の処理が行われる。

  • タスクレコードがデータベースに作成され、ステータスが enqueued に設定される
  • 対応するIssueのステータスが「待機中」に更新される
  • task.enqueued WebSocketイベントが、同一ワークスペースに接続しているすべてのクライアントへブロードキャストされる
  • タスクはデーモンのポーリングキューに追加され、クレームを待機する

アサイニーが人間メンバーから別のエージェントに変更された場合、既存のタスクがキャンセルされ新しいタスクがエンキューされる。同一エージェントへの再割り当ての場合はタスクが再利用される。

クレームフェーズ

ローカルマシンで動作するエージェントデーモンはデフォルト3秒間隔でサーバーをポーリングし、自身に割り当てられた enqueued 状態のタスクを検索する。タスクを発見した際、デーモンはClaimTaskAPIを呼び出して排他的なクレームを取得する。

クレームは楽観的ロックで実装されており、複数のデーモンインスタンスが同時に同じタスクをクレームしようとした場合、最初に成功したインスタンスのみが処理を継続し、その他は無視する。クレーム成功後、task.claimed イベントがブロードキャストされる。

// internal/service/task.go（概略）
func (s *TaskService) ClaimTask(ctx context.Context, taskID, daemonID string) error {
    // 排他クレーム（CAS操作）
    rows, err := s.db.ClaimTask(ctx, db.ClaimTaskParams{
        ID:       taskID,
        DaemonID: daemonID,
        Status:   "claimed",
    })
    if rows == 0 {
        return ErrAlreadyClaimed
    }
    // WebSocketブロードキャスト
    s.hub.Broadcast(ws.EventTaskClaimed, taskID)
    return err
}

実行フェーズ

クレームに成功したデーモンは直ちに実行フェーズへ移行する。このフェーズはMulticaの中核機能であり、マルチワークスペースによる隔離された実行環境が提供される。

ワークスペース隔離

デーモンはタスクごとに一時的な隔離ワークスペースを作成する。リポジトリのクローン、依存パッケージのインストール、環境変数の注入がこのフェーズで行われる。ワークスペースは~/.multica/workspaces/{task-id}/に作成され、タスク完了後に自動クリーンアップされる。最大同時実行数（デフォルト20）を超えるタスクは、空きスロットが生じるまでclaimed状態のまま待機する。

エージェントCLI実行

ワークスペースの準備完了後、デーモンはIssueのタイトル・本文・コメントからプロンプトを構築し、対応するAI CLI（claude・codex等）をサブプロセスとして起動する。CLIの起動と同時にStartTaskが呼び出され、ステータスがstartedに遷移する。

  • task.started イベントブロードキャスト
  • Issueステータスが「進行中」に更新
  • エージェントのストリーム出力はIssueコメントとしてリアルタイム投稿される
  • エージェントが能動的にブロッカーを検出した場合、新しいIssueを自動作成することがある
図2: デーモン・タスクサービス・WebSocket Hubの連携図。デーモンの各アクションがAPIコール経由でtask.goを呼び出し、各遷移でWebSocket Hubがブロードキャストを行う。

完了・失敗ハンドリング

エージェントCLIプロセスが終了すると、デーモンはその終了コードと出力を収集し、サーバーへ報告する。

正常完了（complete）

CLIが終了コード0で正常終了した場合、デーモンはCompleteTaskを呼び出す。タスクサービスは以下の処理をトランザクション内で実行する。

  • タスクステータスを completed に更新
  • 対応するIssueのステータスを「完了」に更新
  • エージェントの出力（差分・PR URL等）をIssueにコメントとして投稿
  • task.completed WebSocketイベントをブロードキャスト
  • 隔離ワークスペースのクリーンアップをスケジュール

失敗ハンドリング（fail）

CLIが非ゼロの終了コード、タイムアウト（デフォルト30分）、またはパニックで終了した場合はFailTaskが呼び出される。失敗情報はIssueのコメントとして記録され、エラー内容がボード上に可視化される。再試行ポリシーは現時点でマニュアル（アサイニーの再設定）であり、自動リトライは実装されていない。

失敗のカテゴリとしては以下が存在する。

  • 実行エラー — エージェントCLIがエラーを検出して自発的に終了
  • タイムアウト — デーモンが規定時間内にCLI終了を確認できなかった
  • デーモン切断 — CLIプロセスの実行中にデーモンプロセス自体が終了した場合、サーバー側がハートビートの途絶を検出して強制失敗とする

WebSocketイベントフロー

MulticaのリアルタイムUI更新はWebSocketイベントによって駆動される。internal/hub/hub.goが実装するブロードキャストハブは、接続中のすべてのクライアントに対して同一のイベントを送信する。フロントエンド（Next.js）のZustandストアはこれらのイベントを受信して状態を更新し、ユーザーはページリロードなしにタスクの進捗をリアルタイムで確認できる。

ライフサイクル全体で発火するイベント一覧を以下に示す。

  • task.enqueued — Issue assigneeにエージェントが設定された時点
  • task.claimed — デーモンがタスクを排他取得した時点
  • task.started — エージェントCLIプロセスが起動した時点
  • task.output — エージェントが出力をストリーミングするたびに発火（高頻度）
  • task.completed — タスクが正常完了した時点
  • task.failed — タスクが失敗終了した時点
  • issue.updated — Issueステータス同期完了後に追加発火

イベントのペイロードはJSON形式で、task_id・issue_id・workspace_id・status・timestampフィールドを含む。フロントエンドはこれらを受信次第ZustandのissuesStoreおよびtasksStoreを更新し、UIに即時反映する。
