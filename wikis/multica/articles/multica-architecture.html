Multicaシステムアーキテクチャ詳解

最終更新: 2026-04-09
MulticaのフロントエンドからバックエンドDB、エージェントSDK、ビルドシステムまでの全技術層を解説する。Next.js 16 App Router・Go Chi・PostgreSQL 17 pgvector・TurborepoによるモノレポCI/CDの詳細なアーキテクチャドキュメント。

Multicaシステムアーキテクチャは、Multicaプラットフォームを構成するすべての技術層を体系的に記述したものである。フロントエンドのNext.js 16 App Routerから、GoベースのバックエンドAPI、PostgreSQL 17 + pgvectorデータベース層、AIエージェント実行のためのAgent SDK、そしてTurborepoモノレポによるビルド基盤に至るまで、各コンポーネントの設計原則と相互関係を詳説する。

アーキテクチャ概要

Multicaは5つの主要コンポーネントで構成されるモノレポ構成のフルスタックシステムである。フロントエンドWebアプリ（apps/web/）・Goバックエンドサーバー（server/）・PostgreSQLデータベース・エージェントデーモン・共有型定義パッケージが密接に連携し、タスク管理とAIエージェント実行基盤を提供する。

図1: Multicaシステム全体構成図。各技術層の役割と通信経路を示す。

フロントエンドアーキテクチャ

フロントエンドはapps/web/に配置されたNext.js 16（App Router）アプリケーションで、TypeScriptで実装される。インポートエイリアス@/がapps/web/ルートを指すよう設定されており、パスの管理を簡潔にしている。

Feature-Based Architecture

コードベースは3つの層に分割されたfeature-basedアーキテクチャを採用する。

  • app/ — Next.js App Routerのルーティング層。ページとレイアウトのみを定義し、ビジネスロジックを持たない
  • features/ — ドメインビジネスロジック。auth・workspace・issues・inbox・realtime・modals・skillsの各サブディレクトリが独立したドメインを担当する
  • shared/ — 横断的ユーティリティ。APIクライアント・型定義・ロガーなど複数featureが共有するコードを置く

状態管理（Zustand）

状態管理にはZustandを採用し、ドメインごとに独立したストアを持つ設計となっている。auth・workspace・issues・inboxの各featureが専用のZustandストアを持ち、グローバルな単一ストアによる過度な結合を回避している。リアルタイムWebSocket同期で受信したサーバーイベントはfeatures/realtime/が受け取り、対応するドメインストアを更新する仕組みである。

Next.js App Routerの活用

Next.js 16のApp RouterはReact Server Components（RSC）を基盤とし、初期ロードのパフォーマンスとSEOを向上させる。Multicaではダッシュボード・ボードビュー・設定画面などのルートがApp Router配下で管理される。shadcn/uiコンポーネントライブラリをUIプリミティブとして利用し、Tailwind CSSでスタイリングする。

図2: フロントエンドのfeature-based architecture。app/・features/・shared/ の3層構成を示す。

バックエンドアーキテクチャ

バックエンドはGoで実装され、server/ディレクトリに配置される。標準的なHTTPルーターとしてChiを採用し、シンプルで高パフォーマンスなAPIサーバーを構成する。

Chiルーターとハンドラー構成

APIハンドラーはinternal/handler/以下にドメインごとに1ファイルずつ配置される（ハンドラーファイル = 1ドメイン原則）。たとえばtask_handler.go・workspace_handler.go・agent_handler.goのように分割される。Chiのミドルウェアチェーンでロギング・リカバリ・CORS・認証検証を処理する。

JWT認証とヘッダー伝搬

認証はHS256アルゴリズムのJWTを使用する。JWTミドルウェアがトークンを検証し、後段のハンドラーへX-User-IDおよびX-User-Emailヘッダーとしてユーザー情報を伝搬する。ハンドラー内でこれらのヘッダーを参照することで、毎回DB問い合わせなしにユーザーを識別できる設計である。

タスクサービスとイベントバス

タスクライフサイクルの中核はinternal/service/task.goが担う。エンキュー→クレーム→開始→完了/失敗の各状態遷移を管理し、遷移ごとにinternal/events/の内部イベントバスへイベントを発行する。イベントバスのサブスクライバーはWebSocketブロードキャスターで、接続中クライアントへリアルタイムで状態変化を通知する。ロギングには標準ライブラリのslogを使用し、LOG_LEVEL環境変数で出力レベルを制御する。

データベース設計

データベースにはPostgreSQL 17をpgvector拡張と組み合わせて使用する。pgvectorはエージェントスキルのベクトル検索や類似タスク検索など、AI機能のための高次元ベクトルデータを効率的に格納・検索するために利用される。

sqlcコード生成

GoコードからのDBアクセスはsqlcが自動生成したタイプセーフなコードを経由する。pkg/db/queries/にSQLクエリファイルを配置し、sqlcがGoのインターフェースと実装コードを生成する。これにより、生SQLをGoコード内に直接記述することなく、コンパイル時の型チェックを享受できる。生成コードはpkg/db/以下に出力される。

マイグレーション管理

スキーマ変更はすべてmigrations/ディレクトリのSQLファイルとして管理される。適用はcmd/migrateエントリポイントで実行し、go run ./cmd/migrate upでマイグレーションを順次適用する。

マルチテナント設計

マルチワークスペース対応のため、すべてのDBクエリでworkspace_idによる行レベルフィルタリングを行う。テナント間のデータ混在を防ぐため、sqlcクエリには必ずWHERE workspace_id = $1条件が含まれる。この設計により単一のDBインスタンスで複数チームのデータを安全に分離できる。

Agent SDK

Agent SDKはpkg/agent/に実装されており、異なるAI CLIバックエンドを統一インターフェースで扱うための抽象化層を提供する。

Backendインターフェース

SDKのコアはBackendインターフェースで、Claude Code・Codex・その他AI CLIに対応した実装がこれを満たす。インターフェースは各CLIの起動・標準入出力のストリーミング・終了コードの取得を統一的に抽象化する。エージェントデーモンはPATHから利用可能なCLIを検出し、対応するBackend実装を選択して登録する。

Sessionチャネル

CLI実行はSession型で管理される。Session.MessagesチャネルはCLIの標準出力をリアルタイムでストリーミングし、Session.ResultチャネルはCLI終了時の実行結果（成功/失敗・終了コード）を送信する。Goのチャネル機構を活用することで、ノンブロッキングな非同期実行とリアルタイムログ転送を実現している。

ビルドシステム

MulticaのモノレポはTurborepoとpnpmワークスペースで管理される。フロントエンド（TypeScript/Node.js）とバックエンド（Go）を単一リポジトリで効率的にビルド・テスト・CI実行できる体制を整えている。

Turborepo

Turborepoはタスクの依存関係グラフを定義し、並列ビルドとキャッシュを最適化する。turbo.jsonでビルド・リント・テストタスクのパイプラインを宣言し、変更のないパッケージのビルド結果をキャッシュすることで、CI時間を大幅に短縮する。

pnpmワークスペース

フロントエンドの依存関係管理にはpnpmワークスペースを使用する。apps/web/と共有パッケージ間の依存解決を効率化し、node_modulesのホイスティングによる容量削減を実現する。

CI/CD

GitHub ActionsベースのCIパイプラインが設定されており、Node.js 22・Go 1.24の組み合わせでビルドとテストを実行する。3つのエントリポイントに対応したビルドジョブが並列実行される。

  • cmd/server — HTTPサーバー（メインAPI）
  • cmd/multica — CLI（ローカルデーモン実行）
  • cmd/migrate — DBマイグレーション実行
