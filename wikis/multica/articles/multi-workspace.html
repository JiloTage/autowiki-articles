マルチワークスペース管理

最終更新: 2026-04-09
Multicaのマルチワークスペース機能は、複数チームや組織が独立した作業環境を持てるようにする仕組みで、データベースレベルのアイソレーション・メンバーシップモデル・ヘッダーベースルーティングによって実現される。

マルチワークスペース管理は、Multicaプラットフォームが提供するマルチテナント機能であり、複数のチームや組織が同一インスタンス上で互いに干渉することなく独立した作業環境を運用できる仕組みである。各ワークスペースはエージェント・Issue・設定を個別に管理し、データベースレベルの厳格なアイソレーションとX-Workspace-IDヘッダーによるルーティングによってセキュリティを確保する。

概要

Multicaにおけるワークスペースは、プロジェクト管理・エージェント管理・スキル蓄積を行う最上位の組織単位である。単一のMulticaアーキテクチャのデプロイ上で複数ワークスペースを同時に稼働させることができ、それぞれが完全に独立した環境として機能する。

ワークスペース間のアイソレーションは主に以下の3層で実現される。

  • データ層 — 全テーブルにworkspace_idカラムを持ち、すべてのクエリでフィルタリングを実施する
  • API層 — X-Workspace-IDヘッダーでリクエストを正しいワークスペースにルーティングする
  • 認可層 — メンバーシップチェックによってワークスペースリソースへのアクセスを制御する
図1: マルチワークスペースのアイソレーション構造。各ワークスペースはDBレベルで分離され、APIレイヤーがX-Workspace-IDヘッダーでルーティングを行う。

データベースレベルのアイソレーション

Multicaのマルチワークスペースは、アーキテクチャの基盤となるPostgreSQLデータベースにおいて、すべてのテーブルにworkspace_idカラムを付与することでアイソレーションを実現している。agents・issues・tasks・skills・members など、ワークスペースに帰属するすべてのリソースはこのカラムを持ち、クエリ時に必ずフィルタリングが適用される。

sqlcによって自動生成されたGoコードでは、ワークスペースを横断するクエリが原則として存在しない設計になっており、誤ったデータ参照をコンパイル時の型検査レベルで防止している。

-- テーブル定義例（マイグレーション）
CREATE TABLE issues (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id UUID NOT NULL REFERENCES workspaces(id),
  title       TEXT NOT NULL,
  status      TEXT NOT NULL,
  -- ...
);

-- sqlc生成クエリ例
-- name: ListIssuesByWorkspace :many
SELECT * FROM issues
WHERE workspace_id = $1
ORDER BY created_at DESC;

このアプローチにより、アプリケーションコードの実装漏れによるデータリークリスクを最小化している。全クエリでworkspace_idフィルタリングを強制することは、Multicaのマルチテナント設計における最重要原則の一つである。

メンバーシップモデル

ワークスペースへのアクセス制御は、メンバーシップチェックによって実現される。ユーザーは複数のワークスペースに所属できるが、各ワークスペースのリソースにアクセスするには、そのワークスペースのメンバーシップが必要となる。

メンバーシップの確認はGoバックエンドのミドルウェア層で行われ、APIリクエストごとに現在認証されているユーザーが対象ワークスペースのメンバーであるかを検証する。メンバーでない場合は403レスポンスが返され、リソースの存在すら開示されない。

  • ワークスペースメンバー — 通常のアクセス権を持つメンバー
  • ワークスペース管理者 — メンバー管理・設定変更・エージェント登録などの管理操作が可能

multica CLIでログインすると、ユーザーが所属するすべてのワークスペースが自動的に検出され、デーモンのウォッチリストに追加される。これにより複数ワークスペースのタスクを単一のデーモンプロセスで並列処理できる。

# CLIによるログインとワークスペース自動検出
multica login
# 所属ワークスペースを自動検出してデーモンに登録
multica daemon start

X-Workspace-IDヘッダーによるルーティング

Multicaのフロントエンドとバックエンド間の通信では、X-Workspace-IDカスタムHTTPヘッダーを使ってリクエストを正しいワークスペースにルーティングする。すべてのAPIリクエストにこのヘッダーが含まれており、GoバックエンドのChi routerミドルウェアがこれを読み取って現在のワークスペースコンテキストを確立する。

フロントエンド（Next.js）はZustandのworkspaceストアで現在アクティブなワークスペースIDを管理し、APIクライアントのリクエストインターセプターが自動的にX-Workspace-IDヘッダーを付加する仕組みになっている。

// フロントエンド: APIクライアントのリクエストインターセプター例
api.interceptors.request.use((config) => {
  const workspaceId = useWorkspaceStore.getState().currentWorkspaceId;
  if (workspaceId) {
    config.headers['X-Workspace-ID'] = workspaceId;
  }
  return config;
});

バックエンドのミドルウェアはヘッダーからワークスペースIDを取得し、リクエストコンテキストに格納する。以降のハンドラーはこのコンテキストからワークスペースIDを参照してDBクエリを構築する。ヘッダーが欠落または不正な場合は400または401エラーを返す。

WebSocket接続においても同様に、接続確立時のハンドシェイクリクエストにX-Workspace-IDヘッダーが含まれ、タスクライフサイクルイベントのブロードキャストはワークスペースごとに分離されたチャンネルで行われる。

開発用ワークツリーサポート

Multicaの開発環境では、Gitのworktree機能を活用した並列開発をサポートしている。worktreeサポートもデータベースレベルのアイソレーションを活用しており、各worktreeに固有のDB名とポート番号が.env.worktreeファイルによって割り当てられる。

# .env.worktree の例（worktreeごとに自動生成）
DATABASE_URL=postgresql://user:pass@localhost:5433/multica_worktree_feature-branch
PORT=8081
NEXT_PUBLIC_API_URL=http://localhost:8081

この仕組みにより、複数のfeatureブランチを同一マシン上で独立して動作させることができ、ポートや接続先の競合を避けながら並列開発が可能になる。本番環境のマルチワークスペースアイソレーションと同じ設計原則をローカル開発環境にも適用している点が特徴的である。

worktree環境のセットアップはmultica CLIが管理し、make worktreeコマンドで必要な設定ファイルの生成・DBの初期化・マイグレーション実行を自動化している。
