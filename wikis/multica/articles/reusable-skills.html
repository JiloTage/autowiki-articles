再利用可能スキル

最終更新: 2026-04-09
再利用可能スキルはMulticaプラットフォームのコア機能で、エージェントが問題を解決するたびにその解法がチーム全体のスキルとして蓄積・再利用され、チームの能力を複利的に高める仕組みである。

再利用可能スキル（Reusable Skills）とは、Multicaプラットフォームにおいて、AIエージェントが解決した問題の解法を自動的にスキルとして保存し、将来のタスクで再利用できるようにする仕組みである。デプロイ・マイグレーション・コードレビューなどの反復的なタスクをスキルとして定義することで、チームの能力が時間とともに複利的に成長する。スキルの定義はskills-lock.jsonファイルで管理され、フロントエンドのfeatures/skills/モジュールから操作できる。

概要

ソフトウェア開発チームにおける多くの作業は反復的である。デプロイ手順、データベースマイグレーション、コードレビューのチェックリスト——こうした定型的な作業を毎回エージェントが一から学習するのは非効率であり、品質にもばらつきが生じる。Multicaの再利用可能スキルシステムはこの課題を解決するために設計されている。

核心的なアイデアは「問題解決の結果をナレッジとして永続化する」ことである。エージェントデーモンがタスクを完了するたびに、その解法がスキルとして登録される。次に同類のタスクが発生したときには、登録済みスキルを参照することで、エージェントはより速く・より正確に問題を解決できる。この積み重ねがチームの能力を「複利のように」増大させるという設計思想がMulticaのスキルシステムの根幹にある。

スキルの生成プロセス

タスクライフサイクルの完了フェーズにおいて、エージェントデーモンはタスクの解法を分析し、再利用可能なパターンを抽出してスキルとして登録する。スキル生成は以下の手順で行われる。

  1. タスク完了の検出 — エージェントデーモンがタスクを completed 状態に遷移させる際、解決内容をサーバーへ送信する
  2. パターン抽出 — バックエンドがタスクの種別・入力・出力・適用した手順を解析し、再利用可能なスキルの雛形を生成する
  3. スキル定義の保存 — 生成されたスキル定義が skills-lock.json に追記される
  4. インデックス更新 — フロントエンドのスキル一覧が自動的に更新され、チームメンバー全員から参照可能になる

スキルは一意のID・名称・適用条件・実行ステップ・作成日時などのメタデータを持つ。既存のスキルと類似した解法が発見された場合は、新規登録ではなく既存スキルの更新（バージョンアップ）として処理される。

図1: 再利用可能スキルの蓄積サイクル。タスク完了のたびにスキルが蓄積され、チームの能力が複利的に向上する。

skills-lock.json によるスキル管理

スキルの定義は skills-lock.json というロックファイルで一元管理される。このファイルは package-lock.json や Pipfile.lock と同様の役割を持ち、チームで共有するスキルの「確定版」を記録する。

skills-lock.json の主な構造は以下の通りである。

{
  "version": 1,
  "skills": [
    {
      "id": "deploy-production-v2",
      "name": "本番環境デプロイ",
      "version": "2.1.0",
      "type": "deployment",
      "trigger_patterns": ["deploy to production", "本番デプロイ"],
      "steps": [ ... ],
      "created_at": "2026-03-15T10:23:00Z",
      "updated_at": "2026-04-01T08:00:00Z",
      "usage_count": 42
    }
  ]
}

ロックファイルはGitリポジトリにコミットされ、チームメンバー全員が同一のスキル定義を参照する。スキルが更新されるとバージョン番号がインクリメントされ、変更履歴をGit上で追跡できる。フロントエンドの features/skills/ モジュールはこのファイルを読み込み、スキル一覧の表示・検索・手動実行のUIを提供する。

スキルの衝突（同一タスクに複数のスキルが適合する場合）は、スコアリングアルゴリズムによって最適なスキルが自動選択される。スコアは使用回数・成功率・最終使用日時などの指標に基づいて計算される。

エージェントによるスキルの適用

エージェントデーモンはタスクライフサイクルの開始フェーズで skills-lock.json を参照し、現在のタスクに適合するスキルを検索する。スキルが見つかった場合、エージェントはスキルに定義された手順を実行の出発点として利用する。

スキルの適用フローは以下の通りである。

  • スキルマッチング — タスクの説明文・種別・コンテキストを trigger_patterns と照合する
  • スキルの読み込み — 適合するスキルのステップ定義をエージェントに渡す
  • 実行と監視 — エージェントがスキルの手順に従って作業を進め、サーバーへ進捗をリアルタイム送信する
  • フィードバックループ — 実行結果（成功/失敗・実行時間・変更点）がスキルのメタデータに反映され、次回の精度向上に活用される

スキルが存在しない新規タスクの場合、エージェントはゼロから問題を解決した後、その解法を新しいスキルとして登録する。これによりスキルライブラリは継続的に拡張される。

スキル適用の透明性も重視されており、フロントエンドのタスク詳細画面では「このタスクにはスキル deploy-production-v2 が適用されました」と表示される。チームメンバーはどのスキルがいつ・どのタスクで使われたかをトレースできる。

スキルの複利効果

Multicaのビジョンである「あなたの次の10人の採用はヒトではない」を支える最も重要なコンセプトが、スキルの複利効果（Compound Effect）である。

複利（Compound Interest）の比喩が示す通り、初期のスキル数が少なくても、時間が経つにつれて蓄積されたスキルが新たなタスクを効率化し、その効率化がさらに多くのスキルを生む好循環が生まれる。具体的には以下のような相乗効果が発生する。

  • 実行時間の短縮 — スキルが蓄積されるにつれて、デプロイや設定変更などのタスクが数分から数十秒に短縮される
  • エラー率の低下 — 実績あるスキルを再利用することで、同じミスが繰り返されにくくなる
  • オンボーディングの加速 — 新しいエージェントが既存スキルライブラリを活用することで、即戦力として機能できる
  • 知識の外部化 — 特定のエージェントや人間メンバーのみが知っていた手順が、チームの共有知識として永続化される

Multicaのアーキテクチャ全体がスキルの複利効果を最大化するよう設計されており、PostgreSQLのpgvector拡張を活用したベクトル類似度検索によってスキルマッチングの精度を高め、スキルの再利用機会を最大化している。

スキルの種類と具体例

現在Multicaで定義されているスキルは大きく以下のカテゴリに分類される。

デプロイメント系スキル

最も一般的かつ高頻度で利用されるスキル群である。本番環境・ステージング環境・プレビュー環境へのデプロイ手順を定義し、環境変数の確認・ヘルスチェック・ロールバック手順を含む。

  • deploy-production — 本番環境への安全なデプロイ（ゼロダウンタイム）
  • deploy-staging — ステージング環境へのデプロイと動作確認
  • deploy-preview — PRごとのプレビュー環境の作成と削除
  • rollback-production — 直前バージョンへの緊急ロールバック

マイグレーション系スキル

データベーススキーマの変更を安全に行うためのスキル群である。マイグレーションファイルの作成・実行・ロールバックの標準手順を定義する。

  • db-migration-apply — ペンディングマイグレーションの適用と検証
  • db-migration-rollback — 直前マイグレーションの安全なロールバック
  • db-migration-create — 新規マイグレーションファイルのテンプレート生成

コードレビュー系スキル

Pull Requestのレビューを標準化するスキル群である。セキュリティチェック・パフォーマンス分析・コーディング規約への準拠確認などを自動化する。

  • review-security — 依存パッケージの脆弱性・SQLインジェクション・XSSリスクの検出
  • review-performance — N+1クエリ・不要なレンダリング・メモリリークの検出
  • review-style — プロジェクト固有のコーディング規約・命名規則への準拠確認

テスト・品質保証系スキル

テストの実行・カバレッジ計測・品質レポート生成を自動化するスキル群である。

  • run-e2e-tests — Playwrightによるエンドツーエンドテストの実行
  • coverage-report — テストカバレッジのレポート生成と閾値チェック
