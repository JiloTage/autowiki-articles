リアルタイムWebSocket同期

最終更新: 2026-04-09
MulticaのリアルタイムWebSocket同期は、バックエンドのHubとフロントエンドのWSClient/WSProviderを組み合わせ、タスク状態遷移をブラウザへ即座に反映するプッシュ通知基盤である。

リアルタイムWebSocket同期は、Multicaプラットフォームがタスク状態の変化をブラウザへリアルタイムに配信するための通信基盤である。Goバックエンドのinternal/realtime/パッケージに実装されたHubが全クライアント接続を管理し、タスクライフサイクルの各遷移（エンキュー・クレーム・開始・完了・失敗）でイベントをブロードキャストする。フロントエンドはTypeScriptで実装されたWSClient・WSProvider・useRealtimeSyncフックの三層構造でこれを受け取り、UIを即座に更新する。ポーリングを排除したプッシュ型アーキテクチャにより、エージェントの進捗が人間メンバーのダッシュボードに遅延なく反映される。

概要

Multicaが管理するAIエージェントは非同期にタスクを実行するため、タスク状態の変化をフロントエンドへ効率的に伝える仕組みが不可欠である。WebSocketによるプッシュ型通知は、RESTポーリングと比較して以下の利点を持つ。

  • 低レイテンシ — タスク状態の変化がほぼ即座（数十ms以内）にブラウザへ届く
  • サーバー負荷の軽減 — ポーリングによる無駄なHTTPリクエストが不要になる
  • 双方向通信の基盤 — 現在は受信専用（サーバー→クライアント）だが、将来的なインバウンドメッセージルーティングに対応できる設計となっている
  • ワークスペース分離 — 接続確立時にワークスペースIDで認証・フィルタリングを行い、テナント間のイベント混在を防ぐ

WebSocket接続はHTTP経由のアップグレードハンドシェイクで確立され、バックエンドのエンドポイント（/ws）に接続した各ブラウザタブがクライアントとして登録される。システムアーキテクチャ上では、Next.jsフロントエンドとGoバックエンドの間の常時接続チャネルを形成する。

バックエンドHub（gorilla/websocket）

バックエンドのリアルタイム通信はinternal/realtime/パッケージが担当する。中核となるのはHub構造体で、全WebSocketクライアントのライフサイクルを管理する。

Hub構造体

HubはGoのgoroutineとchannelを組み合わせた並行処理モデルで設計されている。主要フィールドは以下の通りである。

  • clients map[*Client]bool — 接続中の全クライアントセット
  • broadcast chan []byte — ブロードキャスト対象のメッセージキュー
  • register chan *Client — 新規クライアントの登録チャネル
  • unregister chan *Client — 切断クライアントの登録解除チャネル

HubはRun()メソッドをgoroutineで実行し続け、select文で各チャネルのイベントを処理する。クライアントの追加・削除・ブロードキャストはすべてこのgoroutineで直列処理されるため、マップへの競合アクセスが発生しない。

gorilla/websocket

WebSocketプロトコルの実装にはGoの定番ライブラリであるgorilla/websocketを使用する。HTTPハンドラ内でUpgraderを呼び出すことでHTTP接続をWebSocketにアップグレードし、接続ごとにgoroutineを起動してメッセージの読み書きを処理する。

var upgrader = websocket.Upgrader{
    CheckOrigin: func(r *http.Request) bool {
        return true // 開発環境では全オリジンを許可
    },
}

func (h *Hub) ServeWS(w http.ResponseWriter, r *http.Request) {
    conn, err := upgrader.Upgrade(w, r, nil)
    if err != nil {
        return
    }
    client := &Client{hub: h, conn: conn, send: make(chan []byte, 256)}
    h.register <- client
    go client.writePump()
    go client.readPump()
}

Broadcast メカニズム

サービス層（TaskService・各種ハンドラ）からイベントを受け取ったHubは、Broadcast()メソッドを通じて全クライアントへメッセージを送信する。メッセージはJSON形式でシリアライズされ、broadcastチャネルに投入される。Hubのメインループはこれを受け取り、登録済みの全クライアントのsendチャネルに配信する。送信バッファが満杯のクライアントは遅延クライアントとみなされ、切断処理が行われる。

なお、クライアントから Hub へのインバウンドメッセージルーティング（例：クライアントがコマンドを送信する機能）は現時点でTODOとして残されており、将来の拡張対象である。

フロントエンドクライアント

フロントエンドはThreeLayerで構成される。shared/api/のWSClientが低レベルのWebSocket管理を担い、features/realtime/のWSProviderがReactコンテキストとして接続を提供し、useRealtimeSyncフックが各コンポーネントへイベントを分配する。

WSClient（shared/api/）

WSClientはブラウザネイティブのWebSocket APIをラップしたTypeScriptクラスである。主な責務は以下の通りである。

  • WebSocket接続の確立・維持・切断
  • 接続失敗時の指数バックオフ再接続（デフォルト最大5回）
  • 受信メッセージのJSONパースと型安全なイベントオブジェクトへの変換
  • イベントリスナー管理（addEventListener / removeEventListener パターン）
  • 認証トークンをクエリパラメータまたはサブプロトコルで送付

WSProvider（features/realtime/）

WSProviderはWSClientのシングルトンインスタンスをReact Contextで提供するコンポーネントである。アプリケーションのルートレイアウトでマウントされ、以下を管理する。

  • 認証状態の変化（ログイン・ログアウト・ワークスペース切替）に応じたWSClient接続の再初期化
  • 接続状態（connected / reconnecting / disconnected）のReact状態への反映
  • コンポーネントアンマウント時のリソース解放

useRealtimeSync フック

useRealtimeSyncはWSProviderのコンテキストを消費し、受信したWebSocketイベントをZustandストアへ書き込むカスタムフックである。タスク関連のイベントを受け取るとIssue/Inboxストアを直接更新し、Reactの再レンダリングをトリガーする。

// features/realtime/hooks/useRealtimeSync.ts（概略）
export function useRealtimeSync() {
  const { client } = useWSContext();
  const { updateTask } = useIssueStore();

  useEffect(() => {
    if (!client) return;
    const handler = (event: TaskEvent) => {
      updateTask(event.payload);
    };
    client.addEventListener('task', handler);
    return () => client.removeEventListener('task', handler);
  }, [client, updateTask]);
}
図2: WSClientが受信したWebSocketメッセージはWSProvider経由でuseRealtimeSyncフックに届き、Zustandストアを更新してUIを再レンダリングする。

イベント種別とタスク状態遷移

WebSocketを通じてブロードキャストされるイベントは、タスクライフサイクルの各遷移フェーズに対応している。各イベントはJSON形式で送信され、typeフィールドでイベント種別を識別する。

タスクイベント一覧

イベント type タスク遷移 発火元 主なペイロード
task.enqueued pending → queued EnqueueHandler task_id, issue_id, workspace_id
task.claimed queued → claimed TaskService.Claim() task_id, agent_id, claimed_at
task.started claimed → in_progress TaskService.Start() task_id, started_at
task.completed in_progress → done TaskService.Complete() task_id, result_summary, completed_at
task.failed in_progress → failed TaskService.Fail() task_id, error_message, failed_at

イベントペイロードの基本構造は以下のJSONスキーマに従う。

{
  "type": "task.started",
  "workspace_id": "ws_abc123",
  "payload": {
    "task_id": "task_xyz",
    "issue_id": "issue_001",
    "agent_id": "agent_claude",
    "started_at": "2026-04-09T12:34:56Z"
  }
}

workspace_idフィールドはフロントエンド側でフィルタリングに使用され、自分のワークスペース以外のイベントは無視される。これによりマルチテナント環境での情報漏洩を防ぐ。

接続ライフサイクル

WebSocket接続はブラウザセッション開始から終了までの全過程を通じて管理される。以下に接続確立から切断・再接続までのフローを示す。

接続確立（ハンドシェイク）

  1. ユーザーがログインし、認証トークンを取得する
  2. WSProviderがWSClientを初期化し、wss://api.multica.ai/ws?token=JWT&workspace_id=WS_IDへ接続を試みる
  3. バックエンドがJWTを検証し、workspace_idをクライアントメタデータとして保存する
  4. アップグレードハンドシェイク完了後、HubがClientをregisterチャネルで受け付ける
  5. WSClientの接続状態がconnectedに更新され、UIに接続インジケータが表示される

再接続戦略

ネットワーク障害やサーバー再起動などで接続が切断された場合、WSClientは指数バックオフ（初回1秒、最大30秒）で自動再接続を試みる。再接続中はUIに警告バナーが表示され、再接続成功後は最新のタスク状態をREST APIで再取得してUIを同期する。この「再接続後のRESTフォールバック」により、WebSocket切断中のイベント欠損が補完される。

切断処理

ブラウザタブのクローズやナビゲーションによって接続が閉じられると、WSClientはWebSocketのcloseイベントを受け取り、HubのunregisterチャネルへClientを送信する。HubはClientをclientsマップから削除し、sendチャネルをcloseすることでgoroutineをクリーンアップする。

今後の課題

現在のWebSocket実装はサーバーからクライアントへの一方向ブロードキャストに特化している。将来的な拡張として以下が検討されている。

  • インバウンドメッセージルーティング — クライアントからサーバーへのコマンド送信（現在TODO）
  • ルームベース配信 — ワークスペースIDに加え、Issueレベルのルームへの絞り込みブロードキャスト
  • 接続数スケーリング — 複数Goインスタンス間でのHubメッセージ共有（Redis Pub/Subなど）
