セルフホスティング

最終更新: 2026-04-09
MulticaをDockerおよびローカル環境で自己ホストするための手順・前提条件・環境変数設定を解説する。

セルフホスティング（self-hosting）とは、Multicaプラットフォームをクラウドマネージドサービスに頼らず、自社または個人のインフラ上で運用する方式である。MulticaはオープンソースでGitHubに公開されており、DockerおよびDocker Composeを利用した手軽な構成から、worktree機能を活用した複数チェックアウトの並行動作まで柔軟に対応している。セルフホスティングによって、データを自組織の管理下に置き、ネットワーク環境やセキュリティポリシーに合わせたカスタマイズが可能となる。

概要

Multicaのセルフホスティング構成は大きく2通りある。一つはDockerを用いた最小構成で、数コマンドで本番に近い環境を立ち上げられる。もう一つは手動セットアップ（Manual Setup）で、Node.js・Go・PostgreSQLを個別に準備しフロントエンド・バックエンドを直接起動する開発者向けの方式である。いずれの方式でもアーキテクチャ上の構成要素（Next.jsフロントエンド、Goバックエンド、PostgreSQL + pgvector）は共通である。

図1: Multicaセルフホスティング構成図。Dockerホスト上でPostgreSQL・アプリが動作し、ローカルマシンからエージェントデーモンとCLIで接続する。

前提条件

セルフホスティングを行うには以下のソフトウェアが必要である。Dockerを使用する場合でも、初回セットアップやマイグレーションのためにNode.jsとGoが必要になる。

  • Node.js v20以上 — フロントエンドのビルドおよびpnpmの動作に必要
  • pnpm v10.28以上 — モノレポのパッケージ管理に使用。npm install -g pnpmまたはcorepack enableでインストール
  • Go v1.26以上 — バックエンドのビルドおよびマイグレーション実行に必要
  • Docker（Docker Compose含む） — PostgreSQLコンテナの起動に必要。Docker Desktop v4以降推奨
  • Make — make start・make setupコマンドの実行に必要
  • Git — リポジトリのクローンに必要

macOSおよびLinuxが主要な動作環境である。WindowsではWSL2（Windows Subsystem for Linux 2）上での動作が推奨される。

Dockerを使ったセットアップ

最短でMulticaを起動する手順を以下に示す。これがもっとも推奨される導入方式であり、PostgreSQLをDockerコンテナで管理することで環境差異を最小限に抑えられる。

リポジトリのクローン

git clone https://github.com/multica-ai/multica.git
cd multica

環境変数の設定

リポジトリには.env.exampleが同梱されており、これをコピーして編集する。最低限JWT_SECRETを設定しなければならない。

cp .env.example .env
# .env を開き JWT_SECRET を編集する

PostgreSQLの起動

Docker Composeを使ってPostgreSQLコンテナを起動する。イメージにはpgvector/pg17が使用される。

docker compose up -d

コンテナが正常に起動したことを確認するにはdocker compose psを実行する。

マイグレーションの実行

データベーススキーマを初期化するため、Goのマイグレーションコマンドを実行する。

cd server && go run ./cmd/migrate up && cd ..

アプリケーションの起動

make startでフロントエンドとバックエンドを同時に起動する。フロントエンドはポート3000、バックエンドはポート8080でリッスンする。

make start

起動後、ブラウザでhttp://localhost:3000にアクセスするとMulticaのUIが表示される。multica CLIのmultica loginコマンドでログインし、エージェントデーモンを接続できる。

手動セットアップ（開発者向け）

Dockerを使わずに全コンポーネントをローカルで管理したい開発者向けの手順である。

依存パッケージのインストール

pnpm install

Turborepoがapps/web・packages/配下の全パッケージを一括インストールする。

初回セットアップコマンド

make setupは初回に1度だけ実行するコマンドで、以下の処理を自動的に行う。

  • 共有PostgreSQLコンテナが起動していることを確認
  • アプリ用データベースを新規作成
  • マイグレーションを実行してスキーマを初期化
make setup

2回目以降はmake startのみで起動できる。

アプリの起動

make start

内部ではpnpm dev（フロントエンド）とgo run ./cmd/server（バックエンド）が並行して起動する。

環境変数の設定

Multicaの動作は.envファイルで制御される。以下に主要な環境変数を示す。

変数名 必須 説明
JWT_SECRET 必須 JWTトークン署名用シークレット。32文字以上の乱数文字列を推奨
DATABASE_URL 必須 PostgreSQL接続文字列（例: postgres://user:pass@localhost:5432/multica）
NEXT_PUBLIC_API_URL 任意 フロントエンドからバックエンドへのAPIエンドポイント。デフォルト: http://localhost:8080
PORT 任意 バックエンドのリッスンポート。デフォルト: 8080
FRONTEND_PORT 任意 フロントエンドのリッスンポート。デフォルト: 3000

本番環境ではJWT_SECRETにopenssl rand -base64 48などで生成した十分な長さの乱数を使用することが強く推奨される。

Worktreeサポート

Multicaはgit worktreeを活用した複数チェックアウトの並行動作をサポートしている。これにより、1台のマシン上で異なるブランチを複数の独立したMulticaインスタンスとして同時に起動できる。

Worktreeとは

通常、git cloneは単一の作業ディレクトリを作成する。git worktree addを使うと、同一リポジトリの別ブランチを別ディレクトリに展開でき、それぞれが独立した.envを持てる。Multicaではこの仕組みを利用して、1つのPostgreSQLコンテナを共有しながら各worktreeが異なるデータベース名とポートを使用する。

Worktree用環境変数ファイル

各worktreeのルートに.env.worktreeを作成し、そのチェックアウト固有の設定を記述する。

# .env.worktree の例（feature/new-ui ブランチ用）
DATABASE_NAME=multica_feature_new_ui
PORT=8082
FRONTEND_PORT=3002

.env.worktreeの値は.envの同名キーを上書きするため、共通設定（JWT_SECRETなど）は.envに、worktree固有設定は.env.worktreeに分離して管理できる。

共有PostgreSQLコンテナ

postgresコンテナは最初のworktreeでdocker compose up -dを実行した際に起動し、以降のworktreeはすでに起動中のコンテナを再利用する。各worktreeはDATABASE_NAMEで指定した別々のデータベースを使用するため、データが混在することなく独立して動作する。
